# Feature Specification: Flutter Desktop Control Panel for Local Operator Workspaces

**Feature Branch**: `012-flutter-control-panel`
**Created**: 2026-05-23
**Status**: Draft
**Input**: User description: "FEAT-012: Flutter Desktop Control Panel for Local Operator Workspaces — first packaged local Flutter desktop control panel for AgentTower over the FEAT-011 `app.*` backend contract; organized around three operator workspaces (Agent Operations, Project and Specs, Testing and Demo) plus shared notifications and settings; adopt-existing-panes only; managed pane creation deferred to FEAT-013; local-only, no remote SaaS control plane, no scraping of human CLI output."

## Summary

FEAT-012 ships the first end-user product surface of AgentTower: a **local Flutter desktop application** for Windows, macOS, and Linux that turns `agenttowerd` into a usable operator product instead of a CLI-only daemon. The app talks **only** to the local daemon, through the versioned `app.*` namespace introduced by FEAT-011 — never by parsing human CLI output and never via any hosted/remote backend.

The product is organized into three **operator workspaces** plus shared surfaces:

1. **Agent Operations** — the live control-tower view: containers, panes, agents, events, queue, routes, health, and the operator attention queue. Adopt-existing-pane workflows only in this release.
2. **Project and Specs** — the project-definition view: project cards, current work, specs/changes navigation, drift signals, "which master is driving which feature", and the generated handoff prompt flow. The product assumes operators spend a substantial share of their time here, not only in live agent operations.
3. **Testing and Demo** — the validation-control view: available validation entrypoints, run controls, run history, demo readiness summary. This workspace controls validation; it does not author tests.

Shared surfaces include a notifications panel, notification history, a global project switcher, and settings.

The first release is deliberately scoped to **adopt-existing-sessions** workflows. Managed tmux pane/session creation, lifecycle, and auto-launch of agent CLIs are deferred entirely to FEAT-013. Mobile, remote multi-host control, advanced raw-log inspection, and any hosted helper-policy subscription service are out of scope.

## Clarifications

### Session 2026-05-23

- Q: What accessibility level does the first release target? → A: WCAG 2.1 AA-equivalent for keyboard navigation, focus order, semantic labels, and 4.5:1 contrast minimum; no certified screen-reader pass committed in MVP.
- Q: What localization scope does the first release target? → A: English-only in MVP, with all user-facing strings routed through a single localization layer so additional locales are a translation drop-in.
- Q: How is the app distributed and updated? → A: Manual download of a signed installer per OS, plus an in-app "update available" indicator that links to the release page; no background auto-update.
- Q: What state persists between app launches? → A: UX state (window geometry, last project, last workspace + sub-view, list sort/filter, settings) in a local user-scoped config store; session token and domain data are NEVER persisted.
- Q: What does "compatible app launches" mean (for Workspace Selection durability)? → A: Same app major version AND same `app_contract_version` major as the previous run; on mismatch, persisted UX selection is dropped and the operator lands on onboarding/Dashboard.
- Q: How are PRD/architecture/roadmap/spec/change documents opened? → A: Markdown rendered inside the app with an "Open externally" affordance; non-markdown or unknown formats open in the system default.
- Q: How does an adopted agent qualify as a "master"? → A: An agent is a master iff its `role` is `master` (operator-set at adoption time) AND its capability matches a master-class capability registered with the daemon; UI surfaces master-specific views only for agents satisfying both.
- Q: What is the sub-agent tree depth limit in the UI? → A: Maximum 2 levels (master → direct sub-agents); deeper relationships are flattened to the nearest displayed parent with a "+N descendants" affordance.
- Q: How are handoff failure modes handled? → A: Submission failure → handoff stays `drafted` with the error attached; delivery failure (queue accepted but prompt rejected downstream) → handoff moves to `submitted` and surfaces a delivery-failure indicator with a "Retry delivery" action; target master offline between draft and submit → submission allowed and the handoff is held in `submitted` (no `accepted` transition) until reconnection.
- Q: How does the app behave when multiple OS users share a workstation? → A: Per-OS-user isolation; each OS user runs their own app instance against their own daemon socket, with app state stored in the OS user's config directory; no cross-user state sharing.
- Q: What app-internal diagnostics & logging are required? → A: Local rotating log file in the OS user's app data directory capturing app errors, contract-version events, and action latencies; exposed from Settings via "Open log folder" and "Copy diagnostics bundle"; no telemetry uploaded anywhere by default.
- Q: What theme and density options are required? → A: Light + Dark + System (follows OS) for theme; Comfortable + Compact for density.
- Q: What is the keyboard-navigation commitment? → A: Every primary action reachable from a documented keyboard shortcut; full Tab / Shift+Tab / Arrow navigation across all interactive elements; a global command palette (Ctrl+K on Linux/Windows, Cmd+K on macOS) covering project switching, workspace switching, and the most-used actions.
- Q: What is the concrete value of the interaction-stability window (FR-053 / SC-008a)? → A: 2 seconds since the operator's last hover/click/keypress on the attention queue.
- Q: What happens on first launch when the previously persisted project no longer resolves? → A: Restore the last-active project if it still resolves; otherwise land on the Projects view with no project selected and show a banner explaining that the previous project is no longer available.
- Q: How are projects removed from the app? → A: Operator can "Remove project" from the Projects view; removal asks for confirmation and clears project-scoped UI persistence (last sub-view, sort/filter) but does not delete any daemon-side data; the project reappears if it is later re-inferred from an adopted agent's `project_path`, with persisted UI state reset.
- Q: What query/filter scope is required for handoff records? → A: Structured filters by project, master, feature/change, and assignment state (per FR-045) PLUS a date-range filter on `created_at`; no free-text search in MVP.
- Q: How is list-view sort/filter persistence scoped? → A: Persist per-view sort/filter both within the current session and across sessions; scope per-project for project-scoped views (Drift, Available Validation, Runs) and globally for non-project views.
- Q: What is the concrete notification-grouping rule (FR-057)? → A: Collapse N ≥ 3 consecutive notifications that share `event_class` AND `agent_id` AND severity ≤ `warning` within a rolling 60-second window into a single grouped row showing count and most recent timestamp; `high` and `critical` are never grouped; operator can expand any grouped row and disable grouping globally in Settings.
- Q: What is the default global keyboard shortcut for the project switcher? → A: Ctrl+P on Linux/Windows and Cmd+P on macOS (distinct from the command palette Ctrl+K / Cmd+K).
- Q: How does the app behave when bootstrap detects a contract-version-incompatible state? → A: Global banner on every workspace naming the mismatch and upgrade path; views whose minimum contract version is unmet render the documented "contract-version-incompatible" state from FR-004 with the missing version called out; mutations on those views are disabled with an inline explanation; read views still render.
- Q: How are large Events/Queue/Runs lists rendered? → A: Virtualized infinite-scroll list with on-demand fetching of the next page as the operator scrolls, backed by the daemon's pagination cursors; event-style streams always show an explicit "Jump to most recent" affordance.
- Q: What happens to the prior master's in-flight queue rows when the operator chooses "supersede" on a double-driving conflict? → A: Supersede only updates the handoff record (prior handoff → `superseded`, `superseded_by_handoff_id` set on the new one); existing queue rows already created from the prior handoff are left to terminate naturally; the app does not auto-cancel them, and the operator may cancel them manually if desired.
- Q: How does onboarding handle skip/resume? → A: Onboarding is skippable from any step; skipped steps reappear as nudges in the Dashboard until completed; the eight onboarding milestones (daemon reachable → first route created) remain individually trackable so SC-011 can be measured on a per-step completion basis.
- Q: How does the app behave on close/quit while daemon work is in flight? → A: Close immediately without warning — the app is a thin client over the daemon and closing it does not stop daemon-side work (validations keep running, submitted handoffs keep moving, queue stays); reopen resumes the view.

### Session 2026-05-23 (round 2)

Scope: resolve the two Tier-1 findings from the post-checklist review that required operator decision before the spec-quality-pass change runs.

- Q: Where do baked helper-agent policy defaults live? → A: A daemon-side resource exposed via `app.*` (e.g. `app.helper_policies.list` / `app.helper_policies.resolve`); the app never reads policy files directly. The daemon may source defaults from any internal mechanism, but the app integrates only through `app.*` per FR-001 and FR-005.
- Q: What minimum fields make up a helper-agent policy? → A: `policy_id` (stable string), `allowed_helper_capabilities` (set of capability tokens), `default_helper_capability` (single token), and `policy_source` (`baked_default` | `operator_override` | `repo_override`). Quotas and per-capability tool whitelists are out of scope for the first release.
- Q: What is the scope of an operator override (FR-037)? → A: Per-handoff only — the override applies to the current submission and then resets to defaults for the next handoff. No per-master, per-project, or global operator overrides in this release.
- Q: Can a single project define a repo-level override of the baked defaults? → A: Yes — a repo-level override at a conventional path (e.g. `agenttower/helper-policy.yaml`) is allowed; the daemon discovers and surfaces it via `app.*`. Resolved policies that came from a repo override MUST record `policy_source = repo_override` in the handoff snapshot.
- Q: Is `deferred` a separate stage, a separate status value, a boolean flag, or not modeled at all? → A: A separate stage in FR-028; `deferred` is added to the stage enum.
- Q: Is the `deferred` stage terminal? → A: Non-terminal — a feature/change in `deferred` MAY transition back to `definition` or `spec_ready` via an explicit un-defer action; no other transitions from `deferred` are allowed. The feature id is preserved across un-defer.
- Q: How does feature-range resolution (FR-039) treat `deferred` items? → A: Excluded from the resolved list with an explicit annotation (e.g. `FEAT-N (excluded: deferred)`); the master receives only the resolved list. Symmetric with how `merged` items are already handled.

Scope: close the highest-leverage open items across the 19 FAIL checklists from `/speckit-checklist` Round 2 by resolving 21 underlying operator decisions. Decisions baked into research.md "Round 3 decisions" section (R-22..R-42).

- Q: What is the concrete WCAG 2.1 AA-equivalent baseline (FR-066)? → A: Enumerated criteria 1.3.1, 1.4.3, 2.1.1, 2.4.3, 2.4.7, 4.1.2. Scope: every interactive control, status indicator, error message, modal. Accessible-name patterns required for: every badge, every icon-only quick action, every severity color. Certified screen-reader pass remains out of MVP.
- Q: How far does the i18n layer (FR-067) stretch? → A: Ship ICU MessageFormat + `intl` date/number formatting + key-fallback for missing strings. RTL layout mirroring + locale-sensitive list sorting deferred to a future locale-add task. Pluralization required.
- Q: How do theme + density changes apply? → A: Live (no restart). Transient surfaces (toasts, tooltips, OS notifications) follow current theme. Compact density tokens guarantee ≥ 44 px touch targets to stay WCAG-compatible. High-contrast variant deferred.
- Q: How is the Settings surface organized + how do settings apply? → A: 5 groups (Display | Notifications | Connection | Privacy | Diagnostics). Theme/density/grouping toggles apply live. Socket-path change triggers immediate re-bootstrap. OS-notification first-enable invokes platform permission prompt. Reset-to-defaults is one global button. No live-preview for theme — the change IS the preview.
- Q: What's the concrete logging format + redaction policy (FR-074)? → A: JSON-lines format. Levels error+warn+info at production builds; debug toggleable from Settings. Redaction denylist: `app_session_token`, prompt-body content, operator-notes content. Timestamps: ISO-8601 wall-clock + monotonic-ns suffix. Diagnostics bundle: `.zip` archive saved via system file picker; clipboard option for bundles ≤ 1 MiB.
- Q: How is the per-surface minimum required `app_contract_version` (FR-002) declared? → A: Code-derived at build time from each feature module's `app.*` call list. No spec-level or plan-level enumeration. Settings → Doctor surfaces the resolved per-surface table for operator inspection.
- Q: What is the mutation safety / idempotency / read-only-mode policy? → A: Auto-generate `idempotency_key` (uuid v4) on every mutation call, retained for retry. No mutation supports dry-run except handoff preview (FR-040). Read-only mode (per FR-002 degradation) RENDERS mutation buttons but disables them with inline explanation tooltip — never hidden.
- Q: What's the live-update delivery model in the absence of a FEAT-011 push surface? → A: Per-surface polling at 1 s when surface is foreground-visible; 5 s when surface is in another workspace; paused when window is minimized. Reconnect mid-stream re-fetches from head with cursor invalidation handling. When FEAT-011 v1.x adds push, the polling Provider swaps cleanly.
- Q: How is the FR-061 trust model enforced on Windows + macOS (which lack `SO_PEERCRED`)? → A: Per-OS primitives — Linux `SO_PEERCRED`; macOS `LOCAL_PEERCRED` (or `getpeereid`); Windows AF_UNIX file ACL permitting current user only (no peer-credentials API). UID/owner mismatch on connect = immediate disconnect + log error + Dashboard banner. Session token lifetime = process lifetime only (no idle-timeout / refresh).
- Q: What does a diagnostics bundle (FR-074) contain + how is it previewed? → A: Bundle contents: (1) rotating log files post-redaction, (2) app version + contract version + socket path + OS user, (3) doctor report verbatim, (4) session-start + bundle-generation timestamps. Preview window BEFORE save/copy listing file inventory + first/last 20 lines of each log. Size cap 50 MiB with "trim to most recent N files" picker when exceeded.
- Q: What markdown subset does FR-079 render in-app? → A: CommonMark + GFM extensions (tables, strikethrough, task lists, fenced code, autolinks). HTML disabled entirely. `javascript:` + `data:` URLs blocked at the link-tap handler with inline warning. Cross-doc `.md` links resolve to in-app rendering; non-`.md` links open via `url_launcher`. Disk change while open → "stale" indicator + "Reload" button (no auto-reload). Missing path → inline error placeholder.
- Q: What are the notifications panel + attention queue edge-case behaviors? → A: Empty attention queue shows "All clear — no actionable items" placeholder. Default attention sort = severity then age; filterable by class. When a `high`/`critical` notification arrives in an `event_class` with an active grouped row, the grouped row stays grouped and the high notification appears as a separate ungrouped row above. OS-native dispatch suppressed if the same `event_class` + `agent_id` was dispatched within 60 s. OS-permission-denied → inline Settings warning + toggle stays on. Project-card unread count = unread notifications scoped to that project's agents.
- Q: What are the onboarding skip + Dashboard-nudge nuances (FR-010)? → A: "Skip onboarding" affordance present in the header of every step. Individual Dashboard nudges are dismissible via 1-week snooze; longer or "never" requires Settings. Re-entry from Settings starts at the first incomplete milestone. SC-011 cohort denominator = operators who attempted any milestone (anyone who opened onboarding and clicked at least one Next/Skip).
- Q: What are the per-OS installer specifics (FR-068)? → A: Sign with Opensoft's existing daemon code-signing CA. In-place upgrade only (no side-by-side). Installer refuses to launch on persisted-state schema-major downgrade. No autostart by default. Installer checks `agenttowerd` reachability; if absent shows "agenttowerd not detected — install/start it first" then proceeds with desktop install. Signing key rotation = annual + on-incident.
- Q: What are the pagination cursor semantics (FR-080)? → A: Daemon-owned opaque cursor (string token); app treats it as black box. TTL = 5 minutes (cursor invalidates if unused within window). On stale-cursor error, app re-fetches from head + shows "Stream resumed" indicator on event-style lists. No monotonicity guarantee on Events/Queue — operator may see duplicates on scroll-back during high-event-rate periods; the spec accepts this.
- Q: What is the project-removal (FR-077) confirmation + last-project-pointer behavior? → A: Removal-confirmation shows project name + repo path + "this clears local UI state only; daemon-side agents, handoffs, drift findings are NOT deleted". No in-session undo (operator can re-Add to recreate). When current project is removed, global last-project pointer falls back to most-recently-active OTHER project; if none, set null (Projects view with FR-076 banner).
- Q: What are the performance-budget environmental preconditions (FR-062..FR-065, SC-001..SC-013)? → A: Reference machine: 8-core x86-64 (≥ 3.0 GHz base), 16 GB RAM, NVMe SSD, OS at idle. Daemon fixture = FEAT-011 SC scale profile (≤ 10 containers, ≤ 200 agents, ≤ 1k events/day). No concurrent background apps. Budgets apply at p95 over 10-run repetitions.
- Q: What are the workspace shell UX defaults? → A: Workspace tabs use distinct icons + color accents (per the research R-15 severity palette mechanism). Sub-view ordering is FIXED at MVP — no operator-reorder; FR-011 / FR-023 / FR-046 orderings are authoritative. Zero-state per workspace: Agent Operations Dashboard renders without project selection (daemon-level info); Project and Specs + Testing and Demo show a project-picker placeholder. Deep-link from attention item is forward-only navigation; operator returns via standard back-button.
- Q: What goes on the Health view (FR-022) per subsystem? → A: Per-subsystem row shows name + state (healthy / degraded / down) + last-successful-event timestamp + (if degraded) human-readable reason from `app.readiness`. Daemon version displayed separately at top of Health view (distinct from `app_contract_version`). Project-card validation/drift badges roll up from per-project state, NOT from Health view directly. FR-068 update-available indicator is a Dashboard surface, not Health view.
- Q: How is handoff multi-driver state rendered + how deep is the supersede chain? → A: Project card shows up-to-2 master indicators + "+N more" overflow (per FR-025). Current Work view shows ALL drivers as a sortable list. Supersede chain renders at most 3 levels (oldest → ... → current); deeper truncated with "+N earlier supersessions" link to full chain in handoff history. Supersede confirmation copy: "This will mark H1 as superseded by H2. Existing queue rows from H1 will NOT be auto-cancelled; cancel them manually from the Queue view if needed."
- Q: Who owns the release feed (FR-068) and what's its contract? → A: Feed at `https://releases.opensoft.one/agenttower/control-panel/latest.json` owned by Opensoft Releases team. Signed JSON via TLS only. Never serves a version older than the previously-advertised version (no downgrades through feed). Failure to fetch is silent at MVP; Settings → Doctor surfaces the most recent fetch outcome.

## User Scenarios & Testing *(mandatory)*

### User Story 1 - Adopt and operate existing agent panes from the desktop (Priority: P1)

The operator already has an Opensoft bench container running with one or more live tmux panes that host AI agent CLIs (Claude, Codex, etc.). They launch the AgentTower desktop app, discover those panes through the app, adopt one into a registered agent with a label/role/capability, attach a log so AgentTower can observe events, see live events flow in, send a direct input, and manage at least one route — all without ever opening a terminal or invoking the `agenttower` CLI.

**Why this priority**: This is the absolute minimum value the app must deliver. Without it, the desktop product is a shell with no operational utility, and the FEAT-011 backend contract has no UI client. Every other workspace builds on the same daemon connection and the same adopted-agent identity model.

**Independent Test**: With `agenttowerd` running, a bench container present, and at least one live tmux pane in that container, an operator can launch the app on a clean machine and, within a single session, take the pane from "discovered but unmanaged" to "registered agent with attached log, observable events, a successful direct send, and at least one active route" — and confirm each state transition by reading the corresponding view in the app. This slice is shippable on its own as a strict-superset replacement for the equivalent CLI workflow.

**Acceptance Scenarios**:

1. **Given** `agenttowerd` is reachable on the configured socket and a bench container with at least one live tmux pane exists, **When** the operator opens the app for the first time, **Then** the Dashboard shows daemon Healthy, at least one container, at least one pane in "discovered and unmanaged" state, and offers an obvious next action ("Adopt a pane").
2. **Given** the operator is on the Panes view with one unmanaged pane visible, **When** they choose "Adopt", supply a label, role, capability, project path, and select "attach log now", **Then** the pane transitions to "discovered and registered", a new agent appears in the Agents view, a log attachment appears in active state, and the operator returns to a view that confirms all four state changes.
3. **Given** an adopted agent with a log attached, **When** the underlying pane emits classifiable output, **Then** events appear in the Events view in observed-at order and the agent's "last meaningful activity" timestamp on its summary updates within the live-update budget.
4. **Given** an active registered agent, **When** the operator triggers a direct send with a non-empty payload, **Then** the daemon accepts the send, the agent shows the send in its recent activity, and any resulting classified event appears in the Events view linked back to that send.
5. **Given** an adopted agent and a target peer agent, **When** the operator opens Route Management and adds a route from source agent + event class → target agent (default master rule), **Then** the route appears in the Routes view in `enabled, healthy` state and a subsequent matching event causes a queue row to be created against the target.
6. **Given** the daemon socket becomes unreachable mid-session, **When** the operator continues to interact with the app, **Then** the global health indicator switches to "Daemon unreachable", every view that depends on live data displays a clear "runtime-unreachable" empty state (not a generic spinner or generic error), and the Dashboard offers a "Retry connection" affordance.
7. **Given** the daemon's `app_contract_version` is below the app's minimum required version for the Routes view, **When** the operator opens the app, **Then** a global banner appears on every workspace naming the specific missing version and the upgrade path, the Routes view renders the documented "contract-version-incompatible" state, mutation actions on Routes are disabled with an inline explanation, and read-only Routes data still renders.

### User Story 2 - Re-orient to a project and see which master is driving which feature (Priority: P2)

The operator is juggling several AgentTower-managed projects. They return to the app after time away, see their projects as cards on the Projects view, glance at the cards to recall current state (active branch, active feature/change, current driving master, validation badge, drift badge, attention summary), pick a project, and on its Current Work view immediately see which master is driving which feature/change, what stage and status that work is in, and one-click links to the PRD, architecture, roadmap, current feature spec, and any active OpenSpec change.

**Why this priority**: The product reframing is explicit that the Project and Specs workspace is where the operator spends a large share of their time, especially when juggling several projects at once. Without this story, the app fails its "project-first" framing and operators cannot recover context after time away — the whole reason the workspace exists.

**Independent Test**: With two or more registered projects each carrying at least one active feature/change and a master agent assigned, the operator can, in a single session, identify the active driver per project, distinguish projects by phase/status/validation/drift state from card-level information alone, open at least one of {PRD, architecture, roadmap, feature spec, OpenSpec change} for the active feature in one click from Current Work, and answer "which master is driving FEAT-N" without leaving the app. This slice is shippable independently of US3 (handoff generation), US4 (drift workflow), and US5 (testing/demo).

**Acceptance Scenarios**:

1. **Given** at least two projects are tracked in the app and each has at least one active feature/change with an assigned master, **When** the operator opens Projects, **Then** each project appears as a card showing project name, repository path, repo state badge, active branch/worktree badge, active feature/change, current phase/status, current driving master, compact master strip, sub-agent count, last activity, validation badge + last run age, drift badge + source + age, attention summary, and unread notification count.
2. **Given** a project card with one active driving master, **When** the operator clicks "View current master" on the card, **Then** the app navigates to Current Work for that project and displays a master summary containing: master label, capability, role badge, active/inactive badge, current status, assigned project, primary assigned feature/change with overflow summary, human-readable workflow phase, optional underlying workflow token/detail, sub-agent rollup (count + state summary), attention severity + open actionable count, last meaningful activity, and a compact validation badge.
3. **Given** the operator is on Current Work for a project with an active FEAT-N being driven by master M, **When** they choose "Open current feature", **Then** the Specs view opens scoped to that project and feature, and the document list panel offers in-app navigation links to the PRD, architecture, roadmap, current feature spec, and any linked OpenSpec change paths recorded on the feature, and document open behavior follows FR-079.
4. **Given** the operator is anywhere in the app, **When** they use the global project switcher, **Then** the active project context updates everywhere within one navigation step and the same project remains selected when the operator returns to a workspace they previously left.
5. **Given** a project has multiple active features being driven by different masters, **When** the operator views the project card, **Then** the compact master strip shows up to two active masters with overflow summarized (e.g. "+2 more") and the card's "current phase/status" reflects the project's chosen primary feature/change rather than collapsing to "multiple".
6. **Given** the persisted last-active project no longer resolves (project removed or repo path moved), **When** the operator launches the app, **Then** the app lands on the Projects view with no project selected and a non-blocking banner names the project that could not be restored.

### User Story 3 - Generate, preview, and submit a master-driving handoff prompt (Priority: P3)

The operator wants to put a master agent to work on one or more features or OpenSpec changes. They choose a master, attach a project, select work items (one feature, several features, a range like `FEAT-N` through `FEAT-M`, or one or more OpenSpec changes), pick a mode (spec refinement / engineering execution / drift repair / validation-and-demo prep), let AgentTower auto-fill the context (repo identity, doc paths, current stage/status, known drift, validation state, helper-agent defaults), preview the generated prompt, optionally add operator notes, and submit. The handoff becomes a durable, reviewable object linked to the master, the project, and the selected features/changes.

**Why this priority**: Handoff generation is the primary product action that links the Project and Specs workspace to live agent execution. It is the operator's main lever for putting masters to work without rebuilding context every time. It depends on US1 (an adopted master must exist) and US2 (project + active feature/change context must be available), but it can be shipped after both without blocking either.

**Independent Test**: With a registered master agent and a project that has at least one feature spec, the operator can complete the full handoff flow (master → project → work-item selection → mode → preview → optional notes → submit) and afterwards see (a) the generated prompt arriving at the target master via the safe prompt queue and (b) a durable handoff record linked to the master, project, and selected features/changes, in `submitted` (or `accepted` once the master has picked it up) state. This slice is shippable without US4 (drift) and US5 (testing/demo).

**Acceptance Scenarios**:

1. **Given** at least one registered master agent and a project with two feature specs, **When** the operator opens the handoff flow, selects the master, selects the project, picks both features, picks the "Engineering execution" mode, and reaches the preview step, **Then** the preview shows a sectioned prompt with Assignment, Project Context, Workflow Instruction, Helper-Agent Policy, Success Criteria, and Stopping and Escalation Rules sections, and the Project Context section names the repository, PRD, architecture, roadmap, and both feature spec paths.
2. **Given** the operator selects a feature range `FEAT-N` through `FEAT-M` where some intermediate numbers are deferred or already merged, **When** the preview is generated, **Then** the resolved work-item list shown to the operator and embedded in the prompt is an explicit ordered list in ascending FEAT order, with deferred or merged items present as explicit excluded annotations rather than silently omitted.
3. **Given** the operator changes the mode from "Engineering execution" to "Drift repair" on the preview step, **When** they regenerate the preview, **Then** the prompt body shifts emphasis to as-built-vs-as-designed comparison, OpenSpec re-alignment, and corrective-proposal generation, without losing any operator notes the operator had already typed.
4. **Given** the operator submits the handoff, **When** submission completes, **Then** a durable handoff record exists with handoff id, timestamp, operator, target master, project, selected work items, resolved work items, primary work item, mode, helper-policy snapshot, generated prompt text, optional notes, and `assignment_state = submitted`; the prompt is delivered to the target master through the safe prompt queue with the handoff id retained as correlation context.
5. **Given** a previously submitted handoff is still active, **When** the operator opens the corresponding feature in Current Work, **Then** they see that this feature is "being driven by master M under handoff H" with a one-click link to the handoff record, and the master's summary surface reflects the assignment.
6. **Given** the operator tries to edit the system-owned prompt skeleton (Assignment / Project Context / Workflow Instruction / Helper-Agent Policy / Success Criteria / Stopping and Escalation Rules) directly, **When** they save, **Then** the system rejects edits to skeleton sections with an explanation and accepts edits only to the operator-notes layer and to the editable inputs (selected work items, mode, helper-policy override, priority, deadline).
7. **Given** the operator submits a handoff and the daemon rejects submission, **When** the rejection is received, **Then** the handoff stays in `drafted` state with the daemon error attached inline and the operator can amend and retry without losing input.

### User Story 4 - See and act on drift signals for a project (Priority: P3)

The operator opens Drift for a project and sees a list of current drift signals — each with status, source (static check, agent review, operator report, test result), severity, confidence, age, scope (project/feature/change/branch/worktree/assignment), summary, supporting evidence, recommended action, and any linked feature/change/branch. From a drift signal they can launch a drift-repair handoff (which pre-selects the affected feature(s) and the `drift_repair` mode), mark the drift as accepted-as-built, dismiss it, or move it through the review lifecycle.

**Why this priority**: Drift is named as a first-class product signal and as the trigger for the OpenSpec-then-Speckit repair workflow. Operators with multiple projects need a clear "where is reality diverging from intent" surface. P3 because the value depends on US1 (the operator must have working projects) and US2 (project framing), and because drift detection is an ongoing activity that benefits from but does not require US3.

**Independent Test**: With at least one project that has a static drift check producing at least one finding (e.g. active branch does not match intended feature/change, or a referenced spec artifact is missing), the operator can open Drift for that project, see the finding rendered with the documented fields, drill into details, launch a drift-repair handoff that pre-fills the affected feature(s) and `drift_repair` mode, and walk the finding through `new → review_needed → confirmed → repair_planned → resolved` lifecycle states with the project card's drift badge updating accordingly. **Acceptance Scenarios**:

3. **Given** a drift finding is in `new` state, **When** the operator transitions it to `review_needed → confirmed → repair_planned`, **Then** each transition is reflected in the finding's status and updated_at, and the project card's drift badge age reflects the latest update.

5. **Given** static checks and agent review both produce findings about the same scope, **When** the operator views the Drift list, **Then** the two findings remain distinct rows with their respective sources visible so the operator can compare deterministic and semantic signals side by side.

### User Story 6 - Operator attention queue, notifications, and notification history (Priority: P3)

The operator uses the attention queue, notifications panel, and notification history as the day-to-day signal-over-noise UX. The attention queue surfaces actionable operational items (blocked queue rows, degraded subsystems, drift findings needing review, validation failures, route skip explanations needing attention) each with class, severity, age, icon, and color. Processed items move to operator history grouped by agent (with sub-agents nested). Incoming surfaced changes/events flow into the notifications panel; once acknowledged, they move to notification history. The active queue stays stable while the operator is interacting with it.

**Why this priority**: This is the connective UX that makes the live workspace operationally usable rather than just informational. P3 because it amplifies but does not block US1–US5; the first slice of US1 can ship without a polished attention queue.

**Independent Test**: With a session that produces at least three distinct actionable items (e.g. a blocked queue row, a route-skip needing attention, a degraded-subsystem indicator) and at least three notifications (e.g. new event, new drift finding, validation run completed), the operator can distinguish the attention queue from the notifications panel from notification history by their respective contents and behaviors; can click an attention item and arrive at its resolution surface; can acknowledge a notification and see it move to notification history; and can observe that the attention queue does not reorder under the operator's pointer while they are interacting with it. Shippable independently.

**Acceptance Scenarios**:

1. **Given** the operator has at least three live actionable items, **When** they open the attention queue, **Then** each item shows its class (icon), severity (color), age, and a one-line summary; items are sorted by severity-then-age by default; and the sort is stable for at least N seconds (the documented interaction-stability window) after the operator's last interaction with the queue.
2. **Given** an attention item is "blocked queue row needs operator approval", **When** the operator clicks it, **Then** the app navigates directly to that queue row's resolution surface (the Queue view scoped to that row with approve/delay/cancel actions visible).
3. **Given** an attention item has been resolved (operator approved it, the system processed it, etc.), **When** the resolution event lands, **Then** the item moves out of the active queue into operator history, default-rolled-up by agent, with sub-agent items nested beneath their parent agent.
4. **Given** the notifications panel has accumulated several notifications, **When** the operator clicks one to process it, **Then** the notification moves to notification history and the unread notification count on the project card and global notification badge updates accordingly.
5. **Given** the operator has enabled OS-native notification integration in Settings, **When** a high-severity notification fires, **Then** an OS-native notification is delivered in addition to the in-app surface; with the setting disabled, only the in-app surface is used.
6. **Given** the notifications panel receives 5 notifications within 30 seconds that share `event_class = route_skipped`, `agent_id = agent-42`, and severity `warning`, **When** the panel updates, **Then** those 5 notifications appear as a single grouped row showing count and most recent timestamp; given the operator then expands the row, **Then** all 5 individual notifications are visible.

### Edge Cases

- **Daemon not running on launch**: app launches into the "runtime-unreachable" state with a clear explanation, a retry control, and a path to the doctor/preflight output; no other workspace renders live data until reachability returns.
- **Daemon present but mount/socket misconfigured**: bootstrap surfaces a specific, actionable diagnostic naming the failing check (socket path, permissions, mount, version skew) rather than a generic error.
- **App and daemon contract-version skew**: app reports the contract version mismatch on the Dashboard, falls back to read-only operation for surfaces whose required version is unmet, and blocks mutations whose required minor version is missing.
- **Bench container present but no panes**: containers view shows the container; panes view shows the "containers found, no panes" empty state with operator-actionable next steps.
- **Pane discovery degraded**: panes view distinguishes "discovery-degraded" from "no panes" so the operator does not misread a probe failure as an empty environment.
- **Pane disappears under an adopted agent**: the agent's pane state transitions to inactive/stale; the agent's summary reflects the change; existing log attachments transition to a non-active status per the underlying daemon state.
- **Operator selects a feature range with gaps**: handoff preview resolves and displays the explicit ordered list of in-scope features, naming deferred and merged items separately; the resolved list — not the shorthand — is what the master receives.
- **Two operators try to drive the same feature from two different masters**: the second handoff submission is allowed but the affected feature surfaces a conflict indicator on the project card and Current Work view; the operator chooses whether to supersede the prior handoff (which records `superseded_by_handoff_id` per FR-042 and follows FR-081 — supersede is record-only and does NOT auto-cancel prior queue rows) or cancel.
- **Spec or OpenSpec doc paths recorded on a feature have moved or been deleted**: the Specs view shows the missing-artifact state, and a static drift check raises a finding for "expected spec artifact missing".
- **Validation run takes longer than the entrypoint's estimated duration**: the Runs view continues to show `running`; demo readiness does not flip to `ready` until the run resolves with a `pass` result.
- **Notifications arrive at high volume**: AI-assisted noise reduction collapses runs of equivalent low-severity notifications into grouped rows so the panel remains scannable; the operator can expand grouped rows and can disable grouping in settings.
- **Attention queue reorders mid-interaction**: while the operator is pointing at or hovering over the queue, live updates queue up and apply at a stable point (e.g. after the interaction settles) so the operator does not click a different item than they intended.

## Requirements *(mandatory)*

### Functional Requirements

#### Daemon connection and contract

- **FR-001**: The app MUST connect to a single local `agenttowerd` instance through the FEAT-011 `app.*` namespace over the local Unix socket on the same host; it MUST NOT include any code path that opens network sockets, contacts hosted services, or scrapes human-readable CLI output to populate its views.
- **FR-002**: The app MUST perform a versioned bootstrap (`app.hello`-equivalent) on startup, present the resulting `app_contract_version` to the user on the Dashboard, and degrade to read-only for any surface whose minimum required contract version is unmet. On a contract-version-incompatible bootstrap (any minimum requirement unmet), the app MUST display a global banner on every workspace that names the specific mismatch and the upgrade path, MUST render the documented `contract-version-incompatible` state from FR-004 on each affected view (naming the missing version), and MUST disable mutation actions on those views with an inline explanation while still rendering read-only views.
- **FR-003**: The app MUST hold its session token in-memory only and re-bootstrap on socket reconnect; the app MUST NOT persist the session token to disk.
- **FR-004**: The app MUST distinguish at minimum the following runtime states in every surface that depends on live data: runtime-unreachable, contract-version-incompatible, runtime-healthy-empty (no containers/panes), runtime-healthy-populated, and runtime-degraded.
- **FR-005**: The app MUST not invent or mutate domain state locally; all writes MUST go through `app.*` mutation methods and the app MUST reflect the daemon's authoritative response.

#### Workspace and shared surfaces

- **FR-006**: Top-level navigation MUST present four entry points in this order: Agent Operations, Project and Specs, Testing and Demo, Settings; switching workspaces MUST NOT lose the operator's current project selection.
- **FR-007**: The app MUST provide a global project switcher reachable from any workspace via a documented keyboard shortcut (`Ctrl+P` on Linux/Windows, `Cmd+P` on macOS) AND a visible UI affordance; switching projects MUST update project-scoped views across all workspaces.
- **FR-008**: The app MUST provide a notifications panel and a separate notification history surface, both populated from daemon-side events that are classified as notifications (not arbitrary log lines).
- **FR-009**: The app MUST provide a Settings surface that includes at minimum: daemon socket path, contract version display, notifications grouping toggle, OS-native notification integration toggle, theme selection (Light / Dark / System), density selection (Comfortable / Compact), an "Open log folder" affordance (per FR-074), a "Copy diagnostics bundle" affordance (per FR-074), and a config doctor / preflight check action. The doctor action MUST run AT MINIMUM these checks and present a per-check pass/fail with a human-readable explanation on failure:
  1. Daemon socket reachable at the configured path.
  2. Daemon socket peer UID matches the current OS user (per FR-061).
  3. `app_contract_version` satisfies the app's minimum required versions for every primary workspace surface.
  4. Per-OS-user app-data directory is writable (per FR-061a, FR-074).
  5. Rotating log file is writable and not at its size cap.
  6. OS-native notification permission is granted IF the FR-058 toggle is enabled (skipped otherwise).
  Doctor output MUST be includable verbatim in the FR-074 diagnostics bundle and MUST also be reachable from the FR-075 command palette.
- **FR-010**: The app MUST package and run as a single-window desktop application on Windows, macOS, and Linux; first-launch onboarding MUST walk the operator through the eight milestones below, each with an automatically-detectable completion criterion the app can observe without requiring explicit operator confirmation:
  1. Daemon reachable — complete when the bootstrap (`app.hello`-equivalent) returns success.
  2. Bench container check — complete when the daemon reports at least one container in the discovery list.
  3. Pane discovery check — complete when the daemon reports at least one pane in any state other than `discovery-degraded`.
  4. First pane adoption — complete when the daemon confirms at least one pane transition to `discovered-and-registered`.
  5. First agent registration — complete when the daemon's adopted-agent list contains at least one entry.
  6. First log attachment — complete when at least one adopted agent's log-attachment status is `active`.
  7. First direct send — complete when the daemon acknowledges at least one Direct Send (FR-018) with a non-failure response.
  8. First route creation — complete when the daemon's routes list contains at least one route in `enabled` state, regardless of whether the route has yet matched.
  Onboarding MUST be skippable from any step via an explicit "Skip onboarding" affordance; skipped (incomplete) onboarding milestones MUST reappear as actionable nudges on the Dashboard until completed; per-milestone completion state MUST be persisted (and is included in the FR-069 persisted set) so a completed milestone is not re-prompted after completion.

Allowed transitions: `discovered-and-unmanaged ↔ discovered-and-registered` (adoption / de-adoption); any state may transition to `inactive/stale` on pane disappearance and may return to its prior state on rediscovery; any state may transition to `discovery-degraded` on probe failure and back on recovery. - **FR-036**: The handoff flow MUST require these inputs in order: master agent, project/repository, work-item selection (one feature, multiple features, feature range, one OpenSpec change, or multiple OpenSpec changes), and mode (`spec_refinement`, `engineering_execution`, `drift_repair`, `validation_demo_prep`).

- **FR-038**: The handoff flow MUST auto-fill at minimum: project/repository identity, active branch/worktree if relevant, PRD path, architecture doc path, roadmap path, selected feature spec paths, selected OpenSpec change paths, current feature/change stage/status/subphase, known drift state for the selected items, current validation state for the selected items, allowed helper-agent defaults, and repo-specific workflow rules.

A helper-agent policy MUST carry at minimum these fields: `policy_id` (stable string), `allowed_helper_capabilities` (set of capability tokens), `default_helper_capability` (single token), and `policy_source` (`baked_default` | `operator_override` | `repo_override`).   Repo-level overrides are permitted: a project repository MAY provide a conventional override file (e.g. `agenttower/helper-policy.yaml`) that the daemon discovers and surfaces via `app.*`. A resolved policy that originated from a repo override MUST be recorded in the handoff snapshot as `policy_source = repo_override`.

Excluded items (deferred or merged) MUST appear in the resolved list with an explicit exclusion annotation in the form `FEAT-N (excluded: deferred)` or `FEAT-N (excluded: merged)`; the master receives the resolved list and excluded items are present in it as annotated entries, not silently omitted. The resolved list — not the shorthand — MUST be what the master receives.
- **FR-040**: The generated prompt MUST be presented in a stable sectioned structure with sections in this order: Assignment, Project Context, Workflow Instruction, Helper-Agent Policy, Success Criteria, Stopping and Escalation Rules; switching mode MUST regenerate the prompt body while preserving operator notes already entered.
- **FR-041**: The operator MUST be able to edit operator notes, selected work items, mode, helper-policy override, priority, and deadline; the operator MUST NOT be able to edit the system-owned prompt skeleton (the six sections above) directly — attempted skeleton edits MUST be rejected with an explanation.
- **FR-042**: On submission the handoff MUST be persisted as a durable object containing at minimum: handoff id, timestamp (`created_at`, `updated_at`), operator, target master (agent id + label), project (repo + label), mode, priority, optional deadline, assignment state, selected work items, resolved work items, primary work item, linked feature ids, linked change ids, context bundle, helper policy id, helper policy snapshot, generated prompt text, optional operator notes, and lifecycle timestamps (`submitted_at`, `accepted_at`, `completed_at`, `cancelled_at`, optional `superseded_by_handoff_id`).
- **FR-043**: On submission the prompt MUST be delivered to the target master via the daemon's safe prompt queue with the handoff id retained as correlation context so the resulting queue row, audit entries, and downstream events are traceable back to the handoff.
- **FR-044**: The handoff record MUST progress through assignment states: `drafted`, `submitted`, `accepted`, `active`, `waiting`, `blocked`, `completed`, `cancelled`, `superseded`; this state model is independent of the feature/change lifecycle status model and both MUST be visible on the handoff detail surface. Allowed transitions: `drafted → submitted → accepted → active`; from `active` the state may transition to `waiting`, `blocked`, `completed`, or `cancelled`; `waiting` and `blocked` may return to `active`; `submitted` and `accepted` may transition directly to `cancelled` or `superseded`; `completed`, `cancelled`, and `superseded` are terminal. Operator-driven transitions and daemon-driven transitions are both permitted; the daemon is authoritative on conflicts.
- **FR-045**: Handoff records MUST be queryable by project, master, feature/change, and assignment state from within the app, AND by date range against `created_at`. Free-text search across prompt text and operator notes is NOT required in the first release.

- **FR-047**: The Available Validation view MUST present validation entrypoints grouped by scope, each showing label, entrypoint type (`unit_test`, `integration_test`, `contract_test`, `smoke`, `e2e`, `demo_flow`, `doctor`), scope, description, blocking level (`informational`, `recommended`, `required`), estimated duration, and an explicit enabled/disabled state.
- **FR-048**: The Runs view MUST present validation runs with state (`queued`, `running`, `completed`, `cancelled`, `failed_to_start`), result (`pass`, `fail`, `partial`, `error`, `cancelled`), started/completed timestamps, summary, triggered-by, and the entrypoint and target identities. Allowed transitions: `queued → running → completed`; from `queued` or `running` the run may transition to `cancelled`; from `queued` only it may transition to `failed_to_start`. The `result` field is only meaningful in terminal states (`completed`, `cancelled`, `failed_to_start`). `completed`, `cancelled`, and `failed_to_start` are terminal.
- **FR-049**: The operator MUST be able to trigger a run from an entrypoint card and cancel a `running` or `queued` run from the Runs view; the app MUST NOT execute runners locally — execution MUST be invoked through the daemon.
- **FR-050**: The Demo Readiness view MUST present an overall state (`unknown`, `not_ready`, `at_risk`, `ready`), a summary, an explicit `blocking_findings` list, a `recommended_next_runs` list, and references to the recent runs that informed the conclusion; the overall state MUST be at most `at_risk` if any required entrypoint has not run on the current branch.
- **FR-051**: The Testing and Demo workspace MUST NOT include test authoring or in-app test case creation; it is a control and visibility surface only.

#### Attention queue, notifications, and noise reduction

- **FR-052**: The Agent Operations workspace MUST include an operator attention queue containing actionable items only; each item MUST show class (icon), severity (color), age, and a one-line summary; default sort MUST be severity-then-age.
- **FR-053**: The attention queue MUST be stable while the operator is interacting with it: incoming updates MUST NOT reorder visible items under the operator's pointer within the interaction-stability window of 2 seconds since the operator's last hover, click, or keypress on the attention queue. Live updates received during the stability window MUST be queued and applied at the next stable point after the window elapses.
- **FR-054**: Clicking an attention item MUST navigate directly to its resolution surface (e.g. a blocked queue row's row in the Queue view, a degraded subsystem's Health detail, a drift finding's Drift detail).
- **FR-055**: Resolved attention items MUST move into an operator history surface, default-rolled-up by agent, with sub-agent items nested beneath their parent agent; operator history MUST be durable and reviewable across sessions.
- **FR-056**: The notifications panel MUST display incoming surfaced changes/events that the daemon classifies as notifications; processed notifications MUST move to notification history; unread notification counts MUST be visible at the project-card level and globally.
- **FR-057**: The app MUST apply rule-based noise reduction by default: it MUST collapse N ≥ 3 consecutive notifications that share `event_class` AND `agent_id` AND severity ≤ `warning` within a rolling 60-second window into a single grouped row showing the count and the most recent timestamp. The operator MUST be able to expand any grouped row and to disable grouping globally in Settings. Notifications at severity `high` or `critical` MUST NOT be grouped under any circumstances.
- **FR-058**: OS-native notification integration MUST be opt-in via Settings; when disabled, only the in-app surfaces are used.

#### Explainability

- **FR-059**: The app MUST provide explainability surfaces for at minimum: route match, route skip, blocked queue item, and degraded subsystem state; each MUST present what happened, why it happened, the source that caused it, and the route/rule/permission gate that applied — all using terminology a human operator (not a daemon developer) can understand.

#### Security and local-only posture

- **FR-060**: The app MUST refuse to operate against any non-local daemon target; configuration MUST NOT expose a remote host/port field in the first release.
- **FR-061**: The app MUST present a clear in-app statement of the local-only trust model (Unix socket + same-host UID) on first launch and within Settings.
- **FR-061a**: On a workstation with multiple OS users, the app MUST operate per-OS-user: each OS user runs an independent app instance, connects to its own daemon socket, and stores its UX state (settings, persisted last-project, persisted sort/filter, log files) in that OS user's standard config / data directories. The app MUST NOT read or write any other OS user's app state.

#### Accessibility, internationalization, theming, and keyboard navigation

- **FR-066**: The app MUST meet a WCAG 2.1 AA-equivalent accessibility baseline for keyboard operability, visible focus order, semantic labelling of interactive controls, and a contrast ratio of at least 4.5:1 for normal text and 3:1 for large text and meaningful non-text contrast. A certified screen-reader pass is NOT required in the first release, but interactive controls MUST carry semantic labels sufficient to make a future screen-reader pass additive rather than a rewrite.
- **FR-067**: The first release MUST ship English-only, but ALL user-facing strings MUST be routed through a single localization layer (e.g. `flutter_localizations` with ARB files or equivalent) such that adding a locale is a translation-and-config change with no source-code changes to feature surfaces.
- **FR-068**: The app MUST be distributed as a signed installer per supported OS (Windows, macOS, Linux); the app MUST display its installed version on the Dashboard and in Settings, MUST compare against the latest released version available from the configured release feed at most once per app launch, and MUST show an "update available" indicator with a link to the release page when a newer version exists. The app MUST NOT auto-download or auto-install updates in the first release.
- **FR-069**: The app MUST persist UX state across launches in a local user-scoped config store, covering at minimum: window geometry, theme + density, notifications grouping toggle, OS-native notification toggle, last active workspace, last active sub-view per workspace, last active project, per-view sort/filter (per FR-078), and Settings values. The app MUST NOT persist the session token (per FR-003) and MUST NOT persist any cached domain data (containers, panes, agents, events, queue, routes, drift findings, validation runs, demo readiness, handoffs).
- **FR-070**: Persisted UX selections (last project, last workspace, last sub-view, list sort/filter) MUST only be re-applied on launch when the run is a "compatible app launch" — defined as same app major version AND same `app_contract_version` major as the previous run. On any mismatch, persisted UX selections MUST be dropped and the operator MUST land on onboarding (if not previously completed) or the Dashboard.
- **FR-074**: The app MUST maintain a local rotating log file in the OS user's standard app-data directory capturing at minimum: app errors and uncaught exceptions, contract-version bootstrap events, daemon connection state transitions, and operator-action latency above a documented threshold. Settings MUST provide "Open log folder" and "Copy diagnostics bundle" actions that surface this log alongside the current app version, the current `app_contract_version`, the daemon socket path, and the active OS user. The app MUST NOT upload any diagnostics, telemetry, or logs to any remote service.
- **FR-075**: Every primary action surfaced in the app MUST be reachable from a documented keyboard shortcut; the app MUST support full Tab / Shift+Tab / Arrow navigation across all interactive elements; the app MUST provide a global command palette opened with `Ctrl+K` on Linux/Windows and `Cmd+K` on macOS, covering at minimum: project switching, workspace switching, jumping to a named sub-view, and triggering each documented primary action. Documented shortcuts MUST be discoverable from within Settings (or an equivalent reference surface).

#### App state, project lifecycle, and list-view persistence

- **FR-076**: On launch, the app MUST attempt to restore the last-active project from persisted UX state. If the persisted project still resolves (registered with the daemon or inferred from a current adopted agent's `project_path`), the app MUST restore it. If it no longer resolves, the app MUST land on the Projects view with no project selected and MUST display a non-blocking banner naming the project that could not be restored.
- **FR-077**: The operator MUST be able to remove a project from the app via a "Remove project" affordance on the Projects view. Removal MUST be confirmation-gated and MUST clear that project's UI-side persistence (last sub-view, sort/filter). Removal MUST NOT delete any daemon-side data (agents, handoffs, drift findings, validation runs remain owned by the daemon). A removed project MUST reappear in the Projects view if it is later re-inferred from an adopted agent's `project_path`, with its UI persistence reset to defaults.
- **FR-078**: For every list view enumerated in FR-063 (Containers, Panes, Agents, Events, Queue, Routes, Projects, Available Validation, Runs, Drift), the operator's sort and filter selections MUST persist across navigation within a session AND across app launches. Persistence MUST be scoped per-project for project-scoped views (Drift, Available Validation, Runs) and globally for non-project-scoped views. Disabling persistence per-view MUST NOT be required in the first release.

#### Document rendering and large-list rendering

- **FR-079**: When the operator clicks a one-click navigation link to a PRD, architecture doc, roadmap, feature spec, or OpenSpec change document, the app MUST render the document inside the app when the file is markdown (`.md`) and MUST also provide an "Open externally" affordance on every rendered document that hands off to the operator's system default for that file type. For non-markdown or unknown file types the app MUST open the file in the system default and MUST NOT attempt an in-app render.
- **FR-080**: List surfaces that stream live or paginated daemon data (at minimum: Events, Queue, Runs, Drift) MUST be rendered as virtualized lists with on-demand fetching of the next page driven by the daemon's pagination cursors. Event-style streams (at minimum: Events, Queue) MUST always surface an explicit "Jump to most recent" affordance that returns the operator to the head of the stream. Explicit page-number controls (Prev / Next / page N of M) are NOT required in the first release.

#### Handoff failure recovery and supersede

- **FR-072**: The handoff submission flow MUST handle the following failure modes explicitly: (a) on submission failure (daemon rejected `handoff.submit`-equivalent), the handoff MUST remain in `drafted` with the daemon error attached and surfaced inline so the operator can amend and retry without losing context; (b) on delivery failure (the daemon accepted submission but the safe prompt queue subsequently rejected delivery to the target master), the handoff MUST move to `submitted` and the handoff detail surface MUST show a delivery-failure indicator with a "Retry delivery" action; (c) when the target master is offline at submission time, submission MUST be allowed and the handoff MUST be held in `submitted` (no `accepted` transition) until the master reconnects, with the offline-master state visible on the handoff detail surface.
- **FR-081**: When the operator chooses "supersede" on a double-driving conflict (per the existing edge case), the prior handoff MUST transition to `superseded` and `superseded_by_handoff_id` MUST be recorded on the new handoff (per FR-042). Daemon-side queue rows already created from the prior handoff MUST NOT be auto-cancelled by the app; they MUST be left to terminate naturally and remain manually cancellable by the operator from the Queue view if desired. The supersede action MUST be documented in the handoff detail surface as a record-only intent change, not as a queue-state mutation.

#### Quit/close behavior

- **FR-082**: Closing the app window MUST close the app immediately without warning, regardless of in-flight daemon work (running validation runs, recently-submitted handoffs, unprocessed attention items). Closing the app MUST NOT cancel daemon-side work; the daemon retains authoritative state and the operator MUST be able to reopen the app and resume their previous view (subject to FR-070 compatibility). The app MUST NOT offer a "minimize to tray" mode in the first release.

#### Performance and responsiveness (user-facing budgets)

- **FR-062**: On a healthy daemon and a populated runtime, the Dashboard MUST become operationally readable (daemon health + container/pane/agent counts visible) within 2 seconds of app launch.
- **FR-063**: List views (Containers, Panes, Agents, Events, Queue, Routes, Projects, Available Validation, Runs, Drift) MUST render their first screenful of rows within 1 second of view entry on a runtime with up to the daemon-supported pagination page size.
- **FR-064**: Live-update surfaces (Events, Queue, attention queue, notifications panel, master summary "last activity") MUST reflect a new daemon event within 2 seconds of the event being observable on the daemon side.
- **FR-065**: Adopt-existing-pane completion (from "submit" to confirmed registered-agent state in the app) MUST complete within 5 seconds on a healthy daemon.

### Key Entities *(include if feature involves data)*

These are the user-facing entities the desktop app surfaces. They mirror or compose entities exposed by the FEAT-011 `app.*` contract; this list defines what the UI represents, not the underlying storage.

- **Project** — A tracked repository. Identity: daemon-issued project id, derived from the canonicalized repository absolute path; two registrations of the same repository path resolve to the same project id. Identity attributes: project label, repository path, repo state badge, active branch/worktree, active feature/change reference, current driving master reference, last activity, validation badge, drift badge, attention summary, unread notification count.
- **Master Summary** — The compact operational view of a master agent. Identity: the underlying Adopted Agent's id; Master Summary is a view, not a separate entity, and exists only for agents satisfying FR-071. An agent qualifies as a master per FR-071 (role `master` AND master-class capability). Attributes: master label, capability, role badge, active/inactive badge, current status, assigned project, primary assigned feature/change with overflow summary, workflow phase (human + optional token), sub-agent rollup (count + state summary), attention severity + open actionable count, last meaningful activity, compact validation badge.
- **Sub-agent** — A child agent under a master, rendered as a tree node beneath its parent in the operator-history rollup. The displayed tree is limited to two levels (parent → direct sub-agents); deeper descendants are flattened to the nearest displayed parent and surfaced as a "+N descendants" affordance per FR-015.
- **Pane** — A discovered tmux pane in a bench container. States: discovered-and-unmanaged, discovered-and-registered, inactive/stale, discovery-degraded.
- **Adopted Agent** — A registered agent created by adopting a pane. Identity: daemon-issued agent id; the operator-supplied label is mutable and not used for identity. Attributes: label, role, capability, project path, log attachment state, parent/child relationship, current goal/task.
- **Feature/Change Status** — A composite operator-facing status: stage + execution status + optional subphase; rendered as a human-readable combined label first, with the underlying token visible as supporting context. Stages include `deferred`; `deferred` is non-terminal and may transition only back to `definition` or `spec_ready` while preserving the feature/change id.
- **Handoff** — A durable, reviewable record of an operator handing work to a master. Identity: daemon-issued handoff id assigned at submission time; drafts before submission have a transient client-side draft id. Inputs: target master, project, work-item selection (with explicit resolved list), mode. Auto-filled: doc paths, current stage/status/subphase, drift state, validation state, helper-agent defaults, repo workflow rules. Generated: the structured sectioned prompt (system-owned). Editable: operator notes, work-item selection, mode, helper-policy override, priority, deadline. Lifecycle: drafted, submitted, accepted, active, waiting, blocked, completed, cancelled, superseded.
- **Drift Signal** — A first-class signal that implementation no longer matches intended specification. Identity: daemon-issued finding id, stable across lifecycle transitions; a recurrence of a previously-resolved finding receives a new id. Attributes: status, source, severity, confidence, age, scope, summary, evidence, recommended action, linked features/changes/branches/worktrees. Lifecycle: new → review_needed → confirmed → (repair_planned → resolved | accepted_as_built | dismissed).
- **Validation Entrypoint** — A runnable test/demo action the daemon exposes. Identity: daemon-issued entrypoint id, stable per project; an entrypoint disabled then re-enabled retains its id. Attributes: label, type, scope, description, recommended-when, estimated duration, blocking level, tags, enabled state.
- **Validation Run** — One execution of a Validation Entrypoint against a target. Identity: daemon-issued run id, unique per execution; cancellation followed by re-trigger produces a new id. Attributes: state, result, started/completed timestamps, summary, log reference, artifacts, triggered-by, linked features/changes.
- **Validation Target** — The thing the operator is validating: a project, a branch, a feature set, or a change set.
- **Demo Readiness Summary** — The operator-facing answer to "can I demo this now?". Attributes: branch, updated_at, overall state, summary, blocking findings, recommended next runs, recent runs, linked features.
- **Attention Item** — An actionable operational item. Attributes: class (icon), severity (color), age, summary, resolution target.
- **Notification** — A surfaced change/event classified as a notification by the daemon. Lifecycle: incoming → processed → in notification history.
- **Operator History Entry** — A durable resolved-attention or completed-workflow record, rolled up by agent with sub-agents nested.
- **Workspace Selection** — The operator's persisted UX state, restored on a "compatible app launch" (FR-070). Attributes: current workspace, current sub-view per workspace, current project, per-view sort/filter (per FR-078), theme + density (FR-009), notifications-grouping toggle (FR-057), OS-native-notification toggle (FR-058), window geometry, and Settings values (per FR-009). Identity: per-OS-user singleton in the OS user's standard config directory (per FR-061a). The Workspace Selection is the only app-owned persisted entity; all other Key Entities are daemon-owned and MUST NOT be persisted by the app (per FR-005, FR-069).

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: An operator on a clean workstation can go from launching the app for the first time to a fully adopted-and-operating agent (registered agent + attached log + first observed event + first successful direct send) in under 10 minutes, following the in-app onboarding only and without consulting external documentation.
- **SC-002**: An operator switching between projects can identify the active driving master and current feature/change phase for any registered project from card-level information alone (no drill-down required) in under 5 seconds per project.
- **SC-003**: Generating a handoff for a single feature with auto-filled context completes in under 30 seconds from "open handoff flow" to "submitted", and the generated prompt's Project Context section names all of: repository, PRD path, architecture path, roadmap path, and selected feature spec path with no operator typing of paths.
- **SC-004**: For a feature range with at least one deferred and at least one merged intermediate item, the resolved work-item list shown in the handoff preview and the list embedded in the submitted prompt match exactly and explicitly call out the excluded items.
- **SC-005**: Drift findings produced by static checks become visible on the affected project card within 60 seconds of the daemon emitting them, and the operator can reach the drift detail and launch a pre-filled drift-repair handoff in fewer than three navigation steps.
- **SC-006**: Validation runs triggered from the Available Validation view reach a visible `running` state in under 2 seconds and resolve to a terminal state with a result shown in Runs without the operator having to refresh the view.
- **SC-007**: The Demo Readiness summary updates within 5 seconds of a validation run resolving, and any required-entrypoint gap is reflected in the overall state and recommended-next-runs list immediately upon view entry — not on a manual refresh.
- **SC-008**: Among at least 5 distinct daemon-side event classes that produce attention-queue items (blocked queue row, route skip, degraded subsystem, drift confirmed, validation failed), an operator can correctly classify and navigate to the resolution surface for each class within 10 seconds using only the queue's icon and color treatment (no text reading required for classification).
- **SC-008a**: While the operator hovers over the attention queue, no item under the pointer changes position for at least the documented interaction-stability window, measured by automated UI interaction tests across 100 simulated live-update bursts.
- **SC-009**: The app never opens a non-local network socket and never invokes any subprocess that parses human CLI output to populate its views; this is verifiable from a network/process trace during full-workspace exercise without any required-functionality regression.
- **SC-010**: On daemon unreachability, every live-data surface transitions to its documented unavailable state within 2 seconds and reverts to live state within 5 seconds of daemon return; no surface shows stale data labelled as live during the outage.
- **SC-011**: Onboarding's step completion rate (steps presented → steps completed) is at least 90% across the eight onboarding milestones (daemon reachable → first route created) for a user who completes the flow at least once, measured across an internal cohort of operators.
- **SC-012**: At least 90% of new operators report on a post-onboarding survey that they could identify, from card-level information alone, which agent is driving which feature for their primary project.
- **SC-013**: After at least one prior session, an operator returning to the app and selecting a project from the Projects view can name, within 30 seconds of opening Current Work, (a) the active feature/change, (b) the driving master, (c) the current phase/status, and (d) the path to the relevant PRD or feature spec — without typing or external lookup.

## Assumptions

- **Project = repository (one-to-one)**: A project is exactly one git repository in this release. Multi-repo projects are out of scope. Worktrees and branches are subordinate to the parent project.
- **Adopt-only**: The first release surfaces only adopt-existing-pane workflows. Managed pane/session creation, auto-launch of agent CLIs (Claude, Codex), auto-attach on pane creation, and pane lifecycle operations beyond detach are deferred to FEAT-013.
- **Single-window app**: One main window per app instance; multi-window/multi-monitor split is deferred.
- **Single local daemon target**: The app talks to one local `agenttowerd` over a single Unix socket. Multi-daemon, multi-host, or remote control are out of scope.
- **Handoff delivery via the safe prompt queue (FEAT-009)**: Submitted handoffs deliver their generated prompts to the target master through the safe prompt queue so they benefit from arbitration, audit, and the documented queue-state vocabulary; bypassing the queue (e.g. raw direct-send delivery) is not offered in this release.
- **Specs/docs viewing first, in-app editing deferred**: The Specs and Changes views are read-only navigation surfaces in the first release. Direct in-app editing of spec/PRD/architecture/roadmap files is deferred; refinement happens by handing a `spec_refinement` mode handoff to a master.
- **AI-assisted noise reduction is rule-based first**: The first release applies deterministic grouping rules (e.g. collapse N consecutive equivalent low-severity notifications). Model-based noise reduction is a deferred enhancement that may be toggled in Settings when introduced.
- **OS-native notifications are opt-in**: Default off. Enabled only via Settings.
- **Trust model**: Local-only same-host Unix-socket trust (Unix socket file permissions + `SO_PEERCRED` UID match) as established by FEAT-011. No additional authentication, no per-user identity model, no remote auth.
- **Onboarding cohort and survey metrics (SC-011, SC-012)**: "Operator" in success criteria means a member of the internal Opensoft operator cohort working on tracked AgentTower-managed repositories; survey-based criteria are measured against this cohort, not the open user base.
- **Document path discovery**: PRD, architecture, and roadmap paths are resolved per-project either from operator-supplied repo config or from conventional locations (`docs/product-requirements.md`, `docs/architecture.md`, `docs/mvp-feature-sequence.md`) when present; missing files surface as drift findings rather than crashes.
- **Interaction-stability window (FR-053, SC-008a)**: The concrete value is 2 seconds since the operator's last hover, click, or keypress on the attention queue. Live updates received during the stability window are queued and applied at the next stable point after the window elapses.
- **Project registration model**: Projects enter the app either (a) explicitly via an "Add Project" action that supplies a repository path or (b) automatically by inference from adopted agents' `project_path`. Both paths are visible in this release; the app does not silently auto-register projects without operator confirmation.

## Dependencies

- **FEAT-011 Local App Backend Contract** — Hard dependency. The app uses only the `app.*` namespace exposed by FEAT-011; any operation not yet covered by an `app.*` method MUST NOT be invoked from the app and the corresponding UI MUST be hidden or marked unavailable.
- **FEAT-001..FEAT-010** — Indirect dependencies (containers, panes, agents, log attachment, events, queue, routes, arbitration) reach the app only through FEAT-011's `app.*` surface.
- **FEAT-013 Managed Session Creation and Lifecycle** — Reverse dependency. FEAT-013 introduces managed pane creation; FEAT-012 surfaces MUST be designed so adopted and managed agents can coexist without UI restructuring when FEAT-013 lands.

## Out of Scope

- Managed tmux pane/session creation, lifecycle, auto-launch of agent CLIs (deferred to FEAT-013)
- Hosted web app, SaaS backend, or any non-local control surface
- Remote multi-host control
- Advanced raw log inspector / "raw mode" surface as a first-class workspace
- External helper-policy subscription service (helper-agent capability mapping and prompt policy ship as defaults baked into this release)
- Mobile or web companion app
- Antigravity support
- In-app authoring of test cases or validation entrypoints
- In-app direct editing of PRD / architecture / roadmap / feature spec files (refinement is performed by handing work to a master)
- Speculative or future-relationship planning UX in the relationship graph
- Multi-daemon or multi-host concurrent operation
