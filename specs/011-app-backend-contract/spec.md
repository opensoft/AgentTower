# Feature Specification: Local App Backend Contract for Desktop Control Panel

**Feature Branch**: `011-app-backend-contract`
**Created**: 2026-05-18
**Status**: Draft
**Input**: User description: "FEAT-011: Local App Backend Contract for Desktop Control Panel — make `agenttowerd` a stable local backend for a packaged desktop control panel so a future Flutter app can operate AgentTower without scraping human CLI output. Local-only, no remote SaaS control plane, adopt-existing-panes workflow first, managed session creation deferred to FEAT-013."

## Summary

FEAT-011 turns `agenttowerd` into a stable **local-only application backend** for a future packaged desktop control panel (first target: Flutter desktop on Windows, macOS, Linux). It introduces a deliberate, versioned, app-facing namespace (`app.*`) over the **existing** newline-delimited JSON Unix socket so a UI process can render every primary AgentTower operational surface — daemon health, containers, panes, agents, log attachments, events, queue, routes — and perform every adopt-mode mutation **without parsing human-readable CLI output**.

FEAT-011 is post-MVP (depends on FEAT-001..FEAT-010). It is **backend contract only**: no Flutter UI, no managed tmux session/pane creation (deferred to FEAT-013), no network listener, no hosted/SaaS control plane, no multi-user remote collaboration.

Three core design decisions are fixed in this spec:

1. **A new `app.*` method namespace is added alongside the existing socket methods.** Existing CLI-facing methods (`ping`, `status`, `list_agents`, `register_self`, `route add`, etc.) remain unchanged and continue to be supported as-is for FEAT-002..FEAT-010 callers. The app namespace is a **façade** built on the same daemon-internal validation and persistence paths — never a parallel write path.
2. **The app identifies itself via a per-process opaque session token** issued by `app.hello` and presented on every subsequent `app.*` call. The token is local-only, has no user identity, and rides on top of the existing same-machine trust model (Unix socket file permissions + `SO_PEERCRED` UID match to the host user, FEAT-002 §23).
3. **The contract is versioned via `app_contract_version` (major.minor)** returned from `app.hello`. Additive minor changes do not break clients; major bumps require client opt-in.

## Clarifications

### Session 2026-05-19 (round 4 — checklist walkthrough)

This round closes the genuine open gaps surfaced by the 23 checklists. Where the original starred-default recommendation was overridden, the override is called out explicitly with a `(override)` tag. All other answers below match the recommended default.

**Wire framing (Block A)**

- Q: Line-ending byte for NDJSON request lines? → A: `\n` only (strict). A request line containing any `\r` byte MUST be rejected with `validation_failed` and `details = {field: "request_line", reason: "stray CR"}`. The daemon emits `\n` only on response lines.
- Q: UTF-8 and embedded-NUL handling on request lines? → A: Request lines MUST be valid UTF-8. Any embedded NUL byte (`\x00`) MUST be rejected with `validation_failed` and `details = {field: "request_line", reason: "embedded NUL"}`.
- Q: Two JSON objects on one request line (framing violation)? → A (override): The daemon MUST reject the **entire** line with the new closed-set code `malformed_request` (Block B Q5 below). The daemon MUST NOT attempt to parse the first object and discard the remainder.

**Error codes (Block B)**

- Q: Code for a malformed JSON request line (parse failure before dispatch)? → A (override): Add a **27th** closed-set code `malformed_request` to FR-034. `details = {reason: string}` describing the parse failure class (e.g., `"json decode error"`, `"trailing content"`, `"empty line"`). This code MUST also be emitted for the Block A Q3 "two JSON objects on one line" framing violation.
- Q: Code for `app.send_input` when `target_agent_id` is not a registered agent? → A: `agent_not_found` with `details.agent_id`.
- Q: Code for the FEAT-009 permission gate refusing `app.send_input` (distinct from the global routing kill switch)? → A (override): `permission_denied`. `routing_disabled` is reserved exclusively for the global FEAT-009 kill switch being off; per-message permission-gate refusals MUST return `permission_denied` with `details = {reason: "feat009_permission_gate"}`.
- Q: Code for "session token format invalid" (e.g., not a 36-char hex UUID)? → A: Treat as `app_session_expired`. No new code is added; the semantic is "this token cannot be valid for any current session."
- Q: Code for "daemon shutting down" during graceful shutdown? → A: Return `internal_error` with `details = {reason: "shutting_down"}`. No new closed-set code.
- Q: Code for "scan in progress" when `wait=false` and a prior same-kind scan is still running? → A: Return success with the in-flight `scan_id` (coalesce per Block D Q24). No error code; the caller polls `app.scan.status`.
- Q: Code for a malformed `app_id` or detail-id parameter (wrong type passed to `app.<entity>.detail`)? → A: `validation_failed` with `details.field == "<param_name>"`. `not_found`-family codes are reserved for "well-formed id, no matching row."

**Mutation semantics (Block C)**

- Q: Adopt with partial pane identity match (`pane_id` resolves but `session_name` / `window_index` differs)? → A (override): Require **all 6** pane-identity fields (`container_id`, `tmux_socket`, `session_name`, `window_index`, `pane_index`, `pane_id`) to match the currently-discovered row byte-for-byte. Any mismatch MUST return `pane_not_found` with `details = {pane_id, mismatch_field: "<first offending field>"}`. This prevents stale-identity adopts after a tmux reshuffle.
- Q: Adopt with `attach_log: true` and the container is inactive at adopt time? → A (override): The whole adopt MUST fail with `container_inactive` and `details.container_id`. The daemon MUST NOT create the `agents` row in this case; the operator must retry adopt without `attach_log` (or wait for the container to recover). Adopt with `attach_log: false` against an inactive container remains acceptable per FR-026 if FEAT-006's existing rules allow it.
- Q: Adopt with `parent_agent_id` referring to a non-existent agent? → A (override): Return `agent_not_found` with `details.agent_id = <parent_agent_id>`. (The default `validation_failed.details.field == "parent_agent_id"` would conflate a typo with a missing-row case; using `agent_not_found` aligns with how the rest of the contract surfaces "no such row.")
- Q: Adopt with `container_id` from a scan whose container has since become inactive (pane was discovered but container is now down)? → A: `container_inactive` with `details.container_id`.
- Q: `label` normalization on adopt and `app.agent.update`? → A: Trim leading/trailing whitespace; reject embedded newlines (`\n`, `\r`); free-form otherwise. Length cap **256 chars after trim**. Out-of-bounds or newline-containing input → `validation_failed.details.field == "label"`.
- Q: `capability` validation? → A: Free-form non-empty string within FEAT-006's existing length bound (reuse). No new closed-set vocabulary at v1.0.
- Q: `app.agent.update` with multiple invalid fields in one call? → A: All-or-nothing. Validate every input before any SQLite mutation. On any failure, return one `validation_failed` whose `details.field` names the **first offending field** in request-key iteration order; no partial mutation MUST occur.
- Q: `app.agent.update` setting `role=master` from a host-driven caller? → A: Allowed iff FEAT-006's existing rules already permit master promotion from a host-driven caller. FEAT-011 introduces no carve-out and no additional restriction.
- Q: `app.log.attach` input shape beyond `agent_id`? → A: `agent_id` only.
- Q: `app.queue.approve` / `delay` / `cancel` input shape? → A: `approve`: `{message_id}` only. `delay`: `{message_id, delay_ms}`. `cancel`: `{message_id, reason?}` (`reason` is optional, ≤256 chars after trim, recorded in the audit row if present).
- Q: `app.route.remove` of a route that does not exist? → A: `route_not_found` with `details.route_id`.
- Q: `app.hello` called twice on the same connection? → A: Idempotent. Return the same `app_session_token` and `app_session_id` issued by the first call. No new session row, no audit row.

**Concurrency & reliability (Block D)**

- Q: Two concurrent `app.scan.<kind>` calls (or same-kind scans) in flight? → A: Coalesce. The second caller MUST receive the in-flight `scan_id` and observe the same terminal state on `app.scan.status`. Independent scans of the same kind MUST NOT run in parallel.
- Q: Global cap on concurrent in-flight scans across all sessions? → A: Cap at **4** in-flight scans (room for 2 containers + 2 panes when staggered). A 5th scan request MUST be rejected with `validation_failed` and `details = {field: "scan_kind", reason: "too_many_scans_in_flight"}`.
- Q: `app.scan.status` polling SLA? → A: SHOULD answer within **100 ms** wall-clock. Informational, not a Success Criterion in v1.0.
- Q: Two `app.send_input` calls with the same `(app_session_id, idempotency_key)` where the first is still in flight? → A: The second request MUST block until the first commits; both calls MUST return the same response payload; the second carries `deduplicated: true`. The dedupe-in-flight wait MUST share the same 30 s upper bound as scans (Block C Q12 parallel — applies to send_input only).
- Q: Mutation succeeded server-side but daemon died before the response was sent? → A: No FEAT-011 mechanism. Clients use `idempotency_key` on `app.send_input`; other mutations rely on natural idempotency (`update` by value, `route.add` by identity, terminal-state guards). A reconcile-on-next-session mechanism is explicitly deferred.
- Q: Max concurrent app sessions per daemon? → A: Cap at **8** concurrent app sessions. The 9th `app.hello` MUST be rejected with `validation_failed` and `details = {field: "app.hello", reason: "too_many_sessions"}`. Recommended max is in line with plan.md's "≤ 3 concurrent" upper bound with headroom for retries and tooling.

**Sessions & versioning (Block E)**

- Q: Daemon restart mid-session — distinct code from `app_session_expired`? → A: Use `app_session_expired`. Clients reconnect and call `app.hello` regardless of cause (their disconnect vs. daemon restart).
- Q: `app.hello` with empty or absent `client_id` / `client_version`? → A: Accept; store as empty string. Both fields are informational only.
- Q: Peer UID re-checked after `app.hello` succeeded? → A: Accept-time only (FR-041). Per-request re-check is not introduced in v1.0.
- Q: Connection drops between mutation commit and audit row write? → A: Audit row is written **after** SQLite commit and **before** the response envelope is sent (Block G Q47). On connection drop after the SQLite commit but before envelope send, the row is durable; client retries with `idempotency_key` (for `app.send_input`) or natural idempotency (everything else).
- Q: Daemon's `supported_minor_range.max` advertises a minor it doesn't actually implement (erroneously high)? → A (override): Add a contract test that asserts, for every method declared at the daemon's `supported_minor_range.max`, the method is wired into the dispatcher and returns a non-`unknown_method` response on a smoke call. This guards against advertised-but-unwired methods. (See SC-029.)
- Q: Malformed `client_app_contract_major` (string, negative, missing)? → A: Missing → default to `1` (per closed-sets.md). String, negative, or non-integer → `validation_failed.details.field == "client_app_contract_major"`. The daemon MUST NOT silently coerce.
- Q: Should `app.preflight` expose the daemon's contract major (so clients can detect major mismatch without calling `app.hello`)? → A: No. `app.preflight` stays minimal; clients call `app.hello` to learn `app_contract_version`.

**Readiness & dashboard (Block F)**

- Q: Which readiness subsystems are mandatory for top-level `state == "ready"`? → A: All 6 subsystems (`docker`, `tmux_discovery`, `sqlite`, `jsonl`, `routing_worker`, `log_attachment_workers`) MUST be `ok` for `state == "ready"`. Any subsystem `degraded` or `unavailable` lowers the top-level state per the closed-sets.md aggregation table.
- Q: Subsystem status flapping (e.g., Docker briefly unreachable)? → A: Reflect instantaneous state on every call. No debouncing in v1.0.
- Q: `subsystem.reason` length cap? → A: ≤ **512 chars**, no embedded newlines. Over-cap or newline-containing reasons MUST be truncated to 509 chars + `"..."` by the daemon before serialization (an internal invariant, not a client-visible error).
- Q: Upper bound on `app.readiness` wall-clock? → A: SHOULD complete within 100 ms (plan.md target, informational).
- Q: Platform-specific subsystems on Windows (Docker not installed)? → A: The `docker` row is always present; `status == "unavailable"`, `reason == "docker not detected on platform"`. Subsystems MUST NOT be omitted on any platform — clients tolerate "missing row" only as forward-compat (FR-037).
- Q: `events` count buckets on `app.dashboard`? → A: Single `events.total` count (count of events in the FEAT-008 rolling window; today's behavior). No per-event-type breakdown at v1.0.
- Q: `recent_limit = 0`? → A: `validation_failed.details = {field: "recent_limit", reason: "out of bounds [1,50]"}`. Already implied by FR-017; called out here for completeness.

**Observability & audit (Block G)**

- Q: Per-method audit event types — which audit event name does each mutation emit? → A: Reuse the upstream FEAT audit event names byte-for-byte. `app.queue.approve` emits `queue_approved`, `app.route.add` emits `route_created`, `app.agent.register_from_pane` emits `agent_registered`, etc. The only FEAT-011-specific marker is the additive `origin == "app"` field plus `app_session_id`.
- Q: Failed mutations emit audit rows? → A: No. Only successful commits emit audit rows. Failures live in daemon stderr/logs only.
- Q: JSONL audit unwritable mid-mutation (disk full, permission lost)? → A (override): Mutation **proceeds** (SQLite commit completes), the audit row is dropped, the daemon emits a single stderr warning per JSONL outage window, and the response envelope is returned normally. Audit is best-effort and MUST NOT block functional mutations. (Operators monitor JSONL writability through readiness; the `jsonl` subsystem flips to `degraded`/`unavailable` and the dashboard surfaces a hint.)
- Q: Audit-row ordering vs SQLite commit? → A: Audit row is written **after** the SQLite commit and **before** the response envelope is sent. Sync flush. If the JSONL write fails per Q46 (override), the response envelope is still sent normally.
- Q: Concurrent app sessions emitting audit rows simultaneously — serialization? → A (override): The daemon MUST hold a process-wide mutex around the JSONL audit writer. POSIX `O_APPEND` atomicity is not relied upon (some hosts have weak guarantees for large lines under contention); the mutex is the durable serialization point.
- Q: `app.preflight` and `app.hello` audit logging? → A: Not audited. Connection accept/close events are already covered by FEAT-002.
- Q: `client_id` and `client_version` in audit rows? → A: Not included. `app_session_id` is the only attribution token in the audit row.
- Q: JSONL schema version bump for adding `origin == "app"`? → A: No bump. `origin` is an open string field; existing readers tolerate new values.

**Performance bounds (Block H)**

- Q: SC-002 / SC-004 latency budget percentile? → A: Worst-case across 5 trials (deterministic CI test). The wording is "≤ X ms in 5/5 trials on the documented test fixture."
- Q: SC-004 (2 s adopt round-trip) — sum across 4 calls or each individually? → A: Sum of all 4 wall-clocks (`app.scan.panes` + `app.pane.list` + `app.agent.register_from_pane` + `app.agent.detail`). The 2 s budget is on the aggregate.
- Q: Scale assumption for SC tests? → A: Use exactly the plan.md fixture sizes (≤ 10 containers, ≤ 200 agents, ≤ 1 k events/day, ≤ 100 routes). Higher-scale tests are a separate suite, not part of FEAT-011 SCs.
- Q: Latency budget exceeded behavior at runtime? → A: Nothing. Budgets are for testing, not runtime enforcement. The daemon always waits for completion.
- Q: `scan.status` polling SLA at runtime? → A: 100 ms informational (matches Block D Q26 / Block F Q40).
- Q: `app.events.list` with deep history? → A: Default ordering `event_id DESC` + pagination cap of 200 rows per page (FR-020a) keeps any single response bounded. Deep history requires cursor iteration; the contract makes no statement about end-to-end cursor traversal latency.
- Q: Dashboard perf in degraded states? → A: SC-002 (≤ 500 ms) holds in degraded states because readiness probes are cached; degraded state adds no extra I/O on the dashboard path.

**Security boundary (Block I)**

- Q: macOS `SO_PEERCRED` variant? → A: Reuse FEAT-002's existing peer-cred implementation byte-for-byte. FEAT-011 does not change the peer-cred path.
- Q: Privilege drop on daemon startup? → A: Out of scope. Daemon is assumed to run as the host user; documented in Assumptions.
- Q: Socket file permissions changed mid-session (external `chmod`)? → A: Daemon does not monitor; existing sessions continue unaffected. Documented as operator responsibility.
- Q: Symlink attacks on the socket path? → A: Out of scope; FEAT-002's socket-path policy is unchanged.
- Q: Connection exhaustion (DoS) via many `app.hello` calls from the same UID? → A: Mitigated by the 8-session cap (Block D Q29). No per-UID rate limit at v1.0.
- Q: Secret redaction in `error.message` / `error.details`? → A: `error.message` is operator-facing prose; the daemon MUST truncate any quoted `payload` reference inside `error.message` to the **first 64 chars** before serialization. `error.details.reason` strings on validation_failed-class codes carry only field names and short rule violations, never raw values.
- Q: Peer-cred check timing? → A: Accept-time only (FR-041). No per-request re-check.
- Q: Root daemon + non-root client? → A: Reject with `permission_denied`. Daemon's effective UID MUST equal the peer UID.
- Q: Static-analysis test ensuring no `AF_INET` import in FEAT-011 code? → A: Add a contract test that greps `src/agenttower/app_contract/**` for `socket.AF_INET` and fails if found. Complements the runtime packet-capture check in SC-006.

**Closed-set bookkeeping (Block J)**

- Q: `capability_flags` size cap at `app.hello` response? → A: ≤ **64 keys**. The cap is a daemon-side invariant; v1.0 ships with `{}` (0 keys), so the cap is forward-looking.

Other app mutations rely on their existing natural idempotency (`update` by value, `route.add` by identity) or closed-set guards (`pane_already_registered`, `stale_object`); no global idempotency mechanism is introduced.

The field MUST still be present in every `app.hello` response so clients can rely on it. Future optional features (e.g., event stream subscribe, managed pane create) add named flags additively in later minors.
- Q: `app.scan.*` `wait=true` timeout cap and client-cancel behavior? → A: Hard cap `wait=true` at **30 s**. On timeout, the response MUST be a structured failure with `code == "scan_timeout"` and `details.scan_id` populated so the app can poll `app.scan.status`. Client socket disconnect MUST NOT cancel an in-flight scan; the scan completes server-side and remains reachable via the same `scan_id` on a later `app.scan.status` call.
- Q: `app.scan.status` method shape and retention? → A: Add `app.scan.status(scan_id)` returning `{state, scan_kind, started_at, completed_at, result}` where `state ∈ {running, completed, failed, expired}` and `result` mirrors the normal `wait=true` scan summary once terminal. Retention is in-memory only, last 100 scans per daemon process, lost on daemon restart. Unknown or expired `scan_id` returns `scan_not_found` (new closed-set code).
- Q: `hints[]` array contract — where it appears, schema, closed-set codes? → A: `hints[]` is always present (even when empty `[]`) on both `app.dashboard` and `app.readiness`. Per-hint schema: `{code, severity, message, target?}`. `severity` is a closed set `{info, warning, action_required}`. `code` comes from a closed v1.0 registry: `start_bench_container`, `check_container_filter`, `register_first_agent`, `attach_logs`, `enable_first_route`, `docker_unavailable_hint`. Macro-state `hints[]` is separate from the per-subsystem `hint` string on readiness rows (FR-012), which remains subsystem-scoped guidance.
- Q: `error.details` schema per closed-set code? → A: Add a formal per-code registry. `error.details` MUST always be present and always be an object, even when empty (`{}`). Required keys per code are MUST; extra keys are MAY and additive across minors. Codes not explicitly listed in the registry MUST carry `details == {}`. Initial v1.0 registry: `validation_failed → {field, reason}`, `app_contract_major_unsupported → {daemon_app_contract_version, client_app_contract_major}`, `pane_already_registered → {agent_id}`, `pane_not_found → {pane_id}`, `agent_not_found → {agent_id}`, `route_not_found → {route_id}`, `queue_message_not_found → {message_id}`, `scan_timeout → {scan_id}`, `scan_not_found → {scan_id}`, `container_inactive → {container_id}`, `log_attach_blocked → {agent_id, reason}`.
- Q: `state_priority` and `role_priority` orderings backing FR-021 defaults? → A: Operational-first, normative. `state_priority` (queue rows): `pending(1) < in_flight(2) < blocked(3) < expired(4) < cancelled(5) < delivered(6)`. `role_priority` (agent rows): `master(1) < slave(2) < swarm(3) < test-runner(4) < shell(5) < unknown(6)`. Default-ordering tests MUST assert these mappings byte-for-byte.
- Q: Bench-container caller policy for `app.*`? → A: The entire `app.*` namespace is host-only. Any bench-container caller MUST receive the closed-set code `host_only` from every `app.*` method (including `app.preflight` and `app.hello`). The legacy FEAT-002..FEAT-010 socket methods remain fully available to bench-container thin clients unchanged. The existing host-vs-container peer distinction (already required for the FEAT-009 routing-toggle host-only rule) MUST be reused. This is normative, not merely an informational assumption.
- Q: `order_by` direction syntax for `app.<entity>.list`? → A: Direction is expressed as a suffix on the field name: `field:asc` or `field:desc`. A bare field name (no suffix) uses the per-surface default direction declared by FR-021. Unknown direction suffix → `validation_failed` with `details.field == "order_by"`.
- Q: `app.agent.update` semantics for omitted vs empty-string fields? → A: Absent field means "do not change". Empty-string value clears the field, but only for the explicitly clearable fields `project_path` and `label`. Empty-string on a non-clearable field (`role`, `capability`) → `validation_failed` with `details.field` set to that field name. `role` and `capability` may only be changed by supplying a valid non-empty value from their respective closed sets.
- Q: `app.log.detach` for an agent whose log was never attached? → A: Success-idempotent. Response is a normal success envelope with the post-state `AgentViewModel` carrying `log_attached: false`. No closed-set error code is emitted; this is intentional UX so the app does not need to special-case "detach a non-attached log" as a non-error non-state.
- Q: Filter operator vocabulary at v1.0? → A: Exact match only. `app.event.list`, `app.queue.list`, and `app.route.list` accept exact-match filter fields from their per-surface closed set (FR-024) plus paired `since` / `until` time-range params. No relational, prefix, or regex operators ship in v1.0; adding them is an additive minor per FR-035.
- Q: `scan_state.expired` value with no defined v1.0 trigger? → A: Remove `expired` from v1.0. The `scan_state` closed set at v1.0 is `{running, completed, failed}` only. Reintroducing `expired` (e.g., for a wall-clock TTL on retained scan records) MUST be an additive minor with an explicit trigger defined.
- Q: Maximum request and response payload sizes? → A: Request hard cap **1 MiB** (1,048,576 bytes) per NDJSON request line. Response hard cap **8 MiB** (8,388,608 bytes) per NDJSON response line. Exceeding the request cap returns the closed-set code `payload_too_large` with `details.size_limit_bytes` and `details.actual_size_bytes`. The response cap is a daemon-side invariant — existing pagination caps (FR-020a: 200) and recent-limit cap (FR-017: 50) bound responses well below 8 MiB at v1.0 scale; the cap exists to guard against handler bugs and is enforced before serialization.

### Session 2026-05-18

The following decisions are recorded as design defaults. They may be revisited by `/speckit.clarify` if the operator wants them surfaced as explicit questions; otherwise they bind the rest of the FEAT-011 design.

- **Q: Should the desktop app consume the existing socket API directly, or a new app namespace over the same socket?**
  → A: **New `app.*` namespace on the same socket.** Existing methods remain supported unchanged for the CLI; the app namespace adds aggregate/summary reads, app-suitable mutation envelopes, and bootstrap/readiness calls tuned for a UI consumer. Internally, both surfaces dispatch into the same daemon services so validation and persistence behavior are identical.
- **Q: How does the app identify itself?**
  → A: **Local app session marker** issued by `app.hello`. The daemon assigns an opaque `app_session_token` (uuid v4, kept in memory only, not persisted) and an `app_session_id` (short numeric, useful for audit). The client echoes the token on every subsequent `app.*` call. Sessions are best-effort scoped to the calling process — when the socket connection that called `app.hello` closes, the session is invalidated. There is no per-user identity; the underlying trust assumption remains "same-machine, host UID" as enforced by Unix socket permissions and SO_PEERCRED (already required by FEAT-002).
- **Q: What is the bootstrap / readiness contract?**
  → A: **Two calls.** `app.hello` is the cheap handshake — proves the socket is reachable, returns daemon identity, schema/runtime versions, app contract version, supported capability flags, and a fresh session token. `app.readiness` returns a structured "is the local runtime healthy enough to render the control panel?" assessment with a top-level `state ∈ {ready, degraded, unavailable}` and per-subsystem rows (docker, tmux discovery, sqlite, jsonl, routing worker, log attachment workers).
- **Q: Which views need aggregate responses rather than forcing the app to compose many small calls?**
  → A: **One `app.dashboard` call** returns the home-view payload (counts, top-line health, the few "most-recent" event/queue/route rows). Individual list pages (containers, panes, agents, queue, routes, events) have their own `app.<entity>.list` calls. Detail pages call `app.<entity>.detail`. No deep cross-joins required client-side for the dashboard or the entity list pages.
- **Q: How is version compatibility surfaced?**
  → A: **`app_contract_version` is `MAJOR.MINOR`** (e.g., `1.0`). `app.hello` returns the version the daemon implements plus a `supported_minor_range` for the current major (e.g., `1.0–1.0` initially). A newer-app/older-daemon mismatch on major returns `app_contract_major_unsupported` from `app.hello` with the daemon's actual version, and the client MUST refuse to call any other `app.*` method. Newer-daemon/older-app is always allowed within the same major (additive evolution rule, see FR-035).
- **Q: What distinguishes "discovered but unmanaged" panes from "registered agents" in the contract?**
  → A: **Two distinct read types and an explicit adopt mutation.** `app.pane.list` / `app.pane.detail` return discovered tmux panes (FEAT-004 `panes` rows) including a derived `registered: bool` and, if registered, the `agent_id` they map to. `app.agent.list` / `app.agent.detail` return registered agents (FEAT-006 `agents` rows). The adopt-mode mutation `app.agent.register_from_pane` is the only path the app uses to promote a discovered pane to a registered agent; it is **explicitly scoped to host-driven registration** (no `register-self` semantics, no agent-side execution) and is rejected for already-registered panes with `pane_already_registered`.
- **Q: Which operations are synchronous vs asynchronous from the app's perspective?**
  → A: **All FEAT-011 mutations are synchronous from the app's perspective**, returning a final-state envelope. Scans (`app.scan.containers`, `app.scan.panes`) accept an optional `wait: bool` (default `true`): when `true`, the call blocks until the scan cycle completes and returns the post-scan state; when `false`, the call returns immediately with a `scan_id` the app can poll via `app.scan.status`. Long-running event/queue follow streams are **not** introduced in FEAT-011 (event stream subscription is deferred to a follow-up feature); the app polls `app.events.list` / `app.queue.list` with cursor pagination instead.

### User Story 4 — App correctly surfaces degraded and unavailable states (Priority: P2)

The control panel must show meaningful UX when something on the host is wrong: the daemon is not running, the socket file is missing, the schema/runtime version is incompatible, Docker is unavailable, no bench containers are discovered, containers exist but no panes, panes exist but none are registered. **Why this priority**: A control panel that crashes or silently shows an empty view when the daemon is degraded is worse than the CLI. The brief explicitly demands "fail fast with actionable diagnostics." P2 because the happy path (Stories 1–3) is the first thing to prove; degraded UX hardens it.

**Independent Test**: A test harness manually induces each failure class and verifies the contract behavior:

1. Daemon process not running → preflight returns `daemon_unavailable`.
2. Socket file missing → preflight returns `socket_missing`.
3. Daemon running but a major version newer than the client expects → `app.hello` returns `app_contract_major_unsupported` with the daemon's actual version.

3. **Given** Docker is unavailable, **When** the harness calls `app.readiness`, **Then** the `docker` subsystem row has `status == "unavailable"`, `reason != ""`, and the top-level `state` is at least `degraded`.
4. **Given** zero bench containers are discovered, **When** the harness calls `app.dashboard`, **Then** the response is successful, `containers.active == 0`, and a structured `hints[]` array suggests at least one actionable next step (e.g., `start_bench_container`, `check_container_filter`).

### User Story 5 — App and daemon negotiate contract version cleanly across upgrades (Priority: P3)

A newer app talks to an older daemon, or an older app talks to a newer daemon. The contract MUST detect major-version mismatch in `app.hello` and refuse. Within a major version, additive minor changes (new optional fields, new optional methods) MUST be transparent — older clients ignore unknown fields and never call new methods; newer clients use new optional methods only when `capability_flags` advertises them.

**Why this priority**: Required for the upgrade story to work without breaking already-deployed app builds. P3 because the very first packaged app and daemon will ship together; cross-version drift becomes a problem after.

**Independent Test**: A test harness with one daemon at `app_contract_version=1.0` and one synthetic client emitting `client_app_contract_major=2`:

1. The handshake fails with `app_contract_major_unsupported`, no session is issued, and no other `app.*` call is accepted.

A second harness simulating a newer client (`client_app_contract_major=1`) but using a method only available at minor `1.1` against a `1.0` daemon:

2. `app.hello` succeeds; the client inspects `capability_flags` and chooses not to call the `1.1`-only method, OR if it does, receives a structured `unknown_method` failure.

**Acceptance Scenarios**:

1. **Given** a major-version mismatch, **When** `app.hello` is called, **Then** the response is `app_contract_major_unsupported`, no session token is issued, and subsequent `app.*` calls fail with `app_session_required`.
2. **Given** a within-major minor difference, **When** the client calls `app.hello`, **Then** the response includes `capability_flags` so the client can decide whether to use optional newer methods.
3. **Given** the client calls an `app.*` method not implemented at the daemon's minor, **When** the daemon dispatches, **Then** the response is `unknown_method` (closed-set code) and the daemon's running state is unchanged.

### Edge Cases

- **Socket file present but daemon crashed mid-init**: the preflight or `app.hello` must surface `daemon_unavailable` rather than hang. The daemon socket file existing without an alive process is the FEAT-002 stale-pid scenario; the app contract MUST inherit that semantics with a closed-set code.
- **Concurrent app sessions**: more than one packaged app instance MAY connect at once; each receives its own `app_session_token`. The daemon does not gate concurrency between sessions; all mutations remain ordered by the same daemon-side locks as the CLI.
- **App calls `app.*` without first calling `app.hello`**: returns `app_session_required` from every `app.*` method except `app.hello` itself and the lightweight `app.preflight`.
- **Session token replay across reconnects**: closing the underlying socket connection invalidates the session. Reconnecting and presenting the old token returns `app_session_expired`; the app MUST call `app.hello` again.
- **Permission boundary**: any caller that successfully passes the SO_PEERCRED + socket-permission gate is treated as fully authorized at the daemon level — the app contract does not introduce a finer-grained per-method authorization layer in FEAT-011. The trust assumption is "same host UID" and is documented explicitly so the rest of the security boundary is auditable.
- **Adopt-mode race**: between `app.pane.list` and `app.agent.register_from_pane`, another caller (CLI inside the container, or another app session) may register the pane first. `app.agent.register_from_pane` MUST return `pane_already_registered` with the existing `agent_id` rather than overwriting silently.
- **Dashboard atomicity**: `app.dashboard` is read-only and best-effort consistent — it does not take a single global snapshot. Counts within one response are read sequentially and may slightly disagree under heavy concurrent mutation; this is acceptable for a UI dashboard and is documented in the contract.
- **Schema version vs contract version drift**: the underlying SQLite/JSONL `schema_version` is independent from `app_contract_version`. A daemon that supports the same `app_contract_version` MUST behave identically from the app's perspective even if its internal schema migrated. `app.hello` returns both for transparency.
- **No network listener invariant**: FEAT-011 MUST NOT introduce any non-Unix-socket listener, even behind a flag. Any future remote/multi-host access is explicitly a different feature and out of scope.

## Requirements *(mandatory)*

### Functional Requirements

#### Contract surface and namespace

- **FR-001**: The daemon MUST expose a new socket method namespace prefixed `app.*` over the existing Unix socket. Newline-delimited JSON request/response framing per FEAT-002 §19 is reused unchanged.
- **FR-002**: All existing socket methods (FEAT-002..FEAT-010, e.g., `ping`, `status`, `list_agents`, `register_self`, `set_role`, `attach_log`, `send_input`, `queue *`, `route *`, `routing enable|disable`, `events`) MUST continue to be supported with their current request/response shapes for FEAT-011. The `app.*` namespace is additive, not a replacement.
- **FR-003**: The daemon MUST NOT introduce any non-Unix-socket listener (TCP, HTTP, WebSocket, named pipe over a non-Unix mechanism) as part of FEAT-011. Local-only is invariant.
- **FR-003a**: A single NDJSON request line on the socket MUST NOT exceed **1 MiB (1,048,576 bytes)**. A request exceeding this cap MUST be rejected with `payload_too_large` and `details = {size_limit_bytes: 1048576, actual_size_bytes: <observed>}`; the daemon MUST NOT begin handler execution. A single NDJSON response line MUST NOT exceed **8 MiB (8,388,608 bytes)**. The response cap is a daemon-side invariant — at v1.0 the FR-020a `limit ≤ 200` and FR-017 `recent_limit ≤ 50` caps bound responses well below 8 MiB; if a handler attempts to build a response that would exceed the cap, the daemon MUST emit `internal_error` rather than serialize a truncated or invalid envelope.
- **FR-003b**: NDJSON request lines on the `app.*` socket MUST be valid UTF-8 with `\n` as the sole line terminator. Lines containing any `\r` byte, any embedded `\x00` byte, or trailing content after the first JSON object MUST be rejected with the closed-set code `malformed_request` and `details = {reason: "<short class>"}` (e.g., `"stray CR"`, `"embedded NUL"`, `"trailing content"`, `"json decode error"`, `"empty line"`). The daemon MUST emit `\n` only on response lines. (Round-4 Block A.)
- **FR-004**: Internally, every `app.*` method MUST dispatch into the same daemon-internal service layer used by the existing CLI-facing methods so that validation, persistence, audit JSONL emission, and FEAT-009/FEAT-010 worker integration are identical regardless of caller surface.

#### Identity, sessions, and authorization

- **FR-005**: The daemon MUST treat any caller reaching the socket as authorized at the host-user level, gated by the existing FEAT-002 socket file permissions and SO_PEERCRED UID check. FEAT-011 introduces no additional per-method ACL.
- **FR-006**: The app MUST initiate every session with `app.hello`. The daemon MUST issue an opaque `app_session_token` and a numeric `app_session_id`, neither of which is persisted across daemon restarts.
- **FR-007**: Every `app.*` method other than `app.hello` and `app.preflight` MUST require a valid `app_session_token` and return `app_session_required` when missing or `app_session_expired` when invalid/stale.
- **FR-008**: The daemon MUST invalidate the session when the underlying socket connection that issued it closes. The same client MAY reconnect and call `app.hello` again to obtain a new session.
- **FR-008a**: `app.hello` called twice on the same connection MUST be idempotent. The second call MUST return the same `app_session_token` and `app_session_id` issued by the first call; no new session row is created and no audit row is emitted. (Round-4 Block C Q23.)
- **FR-008b**: The daemon MUST enforce a hard cap of **8 concurrent app sessions** process-wide. The 9th `app.hello` on a fresh connection MUST be rejected with `validation_failed` and `details = {field: "app.hello", reason: "too_many_sessions"}`. Sessions count against the cap until their underlying connection closes (FR-008). (Round-4 Block D Q29.)
- **FR-009**: The daemon SHOULD record `app_session_id` in audit JSONL rows produced by app-driven mutations so an operator can attribute changes to a UI session vs CLI calls. Recording MUST NOT include the opaque `app_session_token` itself.

#### Bootstrap and readiness

- **FR-010**: `app.hello` MUST return at minimum: `app_session_token`, `app_session_id`, `daemon_version`, `schema_version`, `app_contract_version` (string `MAJOR.MINOR`), `supported_minor_range` (object with `min` and `max` strings), `host_user_id`, `capability_flags` (object with named boolean flags), and `state == "ok"`. The set of fields is **additive**: future minor versions may add fields but MUST NOT remove or rename existing ones.
- **FR-011**: `app.preflight` MUST be callable without `app_session_token` and MUST be safe before `app.hello`. It returns a small envelope reporting `socket_reachable`, `daemon_reachable`, and a closed-set `code` ∈ {`ok`, `daemon_unavailable`, `socket_missing`, `socket_permission_denied`} so the app can fail fast on the lowest-level errors.
- **FR-012**: `app.readiness` MUST return a top-level `state ∈ {ready, degraded, unavailable}`, a `subsystems` array, and a top-level `hints` array (always present, possibly empty per FR-014a). Each subsystem row MUST include: `name`, `status ∈ {ok, degraded, unavailable}`, `reason` (string, empty when `status == "ok"`), and an optional `hint` field (free-form, subsystem-scoped guidance — distinct from the structured top-level `hints[]` macro-state nudges defined in FR-014a).
- **FR-013**: The `subsystems` array MUST cover at minimum: `docker`, `tmux_discovery`, `sqlite`, `jsonl`, `routing_worker`, `log_attachment_workers`. The contract MAY add subsystems in additive minor releases.
- **FR-014**: `app.readiness` MUST distinguish "no bench containers discovered" (a readiness hint) from "Docker unavailable" (a degraded subsystem). The former is `state == ready` (or `degraded` only if another subsystem fails) with a hint; the latter is `degraded`.
- **FR-014a**: Both `app.readiness` and `app.dashboard` MUST return a top-level `hints` array. The field MUST always be present, with `[]` when no hints apply. Each hint MUST conform to the schema `{code, severity, message, target?}` where `code` is from the closed v1.0 registry `{start_bench_container, check_container_filter, register_first_agent, attach_logs, enable_first_route, docker_unavailable_hint}`, `severity ∈ {info, warning, action_required}` (closed set), `message` is a short human-readable string (operator-facing, not for log scraping by the app), and `target` (optional) is a structured pointer `{kind, id}` linking the hint to a specific entity (e.g., `{kind: "container", id: "..."}`) when applicable. New hint codes and new severities are additive minor changes per FR-035. The structured top-level `hints[]` is distinct from the per-subsystem `hint` string in readiness rows (FR-012), which remains subsystem-scoped free-form guidance.

#### Aggregate dashboard

- **FR-015**: `app.dashboard` MUST return a single response payload containing structured counts and recent-activity rows for: containers, panes, agents, log_attachments, events, queue, routes, and a top-level `hints` array as defined by FR-014a.
- **FR-016**: Container counts MUST include at minimum `active`, `inactive`, and `degraded_scan`. Pane counts MUST include `total`, `registered`, `unregistered`. Agent counts MUST include totals plus per-role breakdown for the FEAT-006 closed set (`master`, `slave`, `swarm`, `test-runner`, `shell`, `unknown`). Queue counts MUST include `pending`, `in_flight`, `delivered`, `cancelled`, `expired` (using the FEAT-009 closed set). Route counts MUST include `enabled` and `disabled`.
- **FR-016a**: A container MUST count as `degraded_scan` (per FR-016) iff its most recent FEAT-003 container-discovery scan completed successfully but its FEAT-004 pane-discovery scan inside that container failed or returned partial data (e.g., `docker exec` inside the container failed even though the container itself is up). `degraded_scan` is distinct from `inactive` (container is down) and from `active` (container is up AND pane discovery succeeded). A container in `degraded_scan` MUST appear in `app.container.list` with `state == "degraded_scan"` and contributes to `app.dashboard.counts.containers.degraded_scan`, not to `active` or `inactive`.
- **FR-017**: The `recent` sub-payloads MUST include compact rows (id, timestamp, type, key labels, derived summary) sufficient to render a "Recent activity" UI block without a follow-up call. Default size is `10` per surface; the app MAY request a different size via an optional `recent_limit` parameter bounded by `[1, 50]`.
- **FR-018**: `app.dashboard` MUST be read-only, MUST NOT take any global lock, and MUST tolerate slight inter-surface inconsistency under concurrent mutation. This is explicitly called out in the contract documentation.

#### Read surfaces (per entity)

- **FR-019**: For each of the following entities the daemon MUST expose `app.<entity>.list` and `app.<entity>.detail`: `container`, `pane`, `agent`, `log_attachment`, `event`, `queue`, `route`.
- **FR-020**: Every `app.<entity>.list` response MUST include: a `rows[]` array, a `total` (or `total_estimate` if pagination cursors are used), an explicit ordering rule documented per surface, and a stable cursor for pagination (`cursor_next`).
- **FR-020a**: Every `app.<entity>.list` MUST accept an optional `limit` parameter with default `50` and hard cap `200`. A request with `limit > 200` MUST be rejected with `validation_failed` and `details.field == "limit"`. A request with `limit < 1` or a non-integer `limit` MUST be rejected with the same code and field.
- **FR-020b**: `cursor_next` MUST be an opaque string with a hard cap of **512 characters**. The daemon MAY format it as it chooses (base64-encoded JSON, signed token, etc.); clients MUST treat it as opaque and pass it back verbatim on the subsequent list request. A `cursor_next` value that exceeds 512 characters, is not valid UTF-8, or cannot be decoded by the daemon MUST be rejected with `validation_failed` and `details.field == "cursor_next"`. A `cursor_next` produced under one `order_by` / filter combination MUST be rejected with the same code if the next request uses a different `order_by` or filter set.
- **FR-021**: Default ordering MUST be: `containers` by `name` ASC, `panes` by `(container_name, session_name, window_index, pane_index)` ASC, `agents` by `(role_priority, registered_at)` ASC, `log_attachments` by `last_output_at` DESC, `events` by `(event_id)` DESC (newest first), `queue` by `(state_priority, created_at)` ASC, `routes` by `(created_at, route_id)` ASC. Each list method MAY accept an `order_by` override from a closed set defined per surface.
- **FR-021a**: The `state_priority` and `role_priority` integer mappings used by FR-021 are normative. `state_priority` (queue rows, lower integer = higher priority, applied to FEAT-009 row states): `pending = 1`, `in_flight = 2`, `blocked = 3`, `expired = 4`, `cancelled = 5`, `delivered = 6`. `role_priority` (agent rows, applied to the FEAT-006 closed role set): `master = 1`, `slave = 2`, `swarm = 3`, `test-runner = 4`, `shell = 5`, `unknown = 6`. Default-ordering contract tests MUST assert these mappings byte-for-byte. Adding a new state or role in a future minor MUST extend these mappings by appending integers strictly greater than the current maximum; renumbering existing mappings MUST require a major bump per FR-035.
- **FR-021b**: Each `app.<entity>.list` `order_by` parameter MUST accept a direction suffix: `field:asc` or `field:desc`. A bare field name (no suffix) MUST use the per-surface default direction declared by FR-021. An unknown direction suffix or any value not matching `^<field>(:asc|:desc)?$` MUST be rejected with `validation_failed` and `details = {field: "order_by", reason: "<short rule violation>"}`. The set of accepted `field` values per surface remains the closed set defined in `contracts/closed-sets.md` §"Order-By Closed Sets".
- **FR-022**: `app.pane.list` rows MUST include a derived `registered: bool` and a nullable `agent_id` so the app can render unmanaged vs managed panes without joining `agents` client-side.
- **FR-023**: `app.agent.list` rows MUST include the agent's container/pane identity plus a derived `log_attached: bool` and a derived `pane_active: bool`.
- **FR-024**: `app.event.list`, `app.queue.list`, and `app.route.list` MUST support filter parameters consistent with FEAT-008/FEAT-009/FEAT-010 closed-set fields (e.g., `event_type`, `origin`, `state`, `role`, `capability`, `route_id`, time-range `since`/`until`).
- **FR-024a**: All filter fields at v1.0 MUST use **exact match** semantics only. The daemon MUST NOT introduce relational (`<`, `>`, `<=`, `>=`), prefix, substring, regex, or set-membership operators on filter fields at v1.0. Time-range constraints use the paired `since` / `until` unix-ms integer parameters defined per surface and are not a generalized operator vocabulary. Adding new operators is an additive minor per FR-035; any v1.0 filter value containing operator-like syntax MUST be rejected with `validation_failed` and `details.field` set to the offending field name.

#### Adopt-mode mutation

- **FR-025**: `app.agent.register_from_pane` MUST accept a structured pane identity (`container_id`, `tmux_socket`, `session_name`, `window_index`, `pane_index`, `pane_id`) plus `role`, `capability`, `label`, optional `project_path`, optional `parent_agent_id`, optional `attach_log: bool`.
- **FR-026**: `app.agent.register_from_pane` MUST reuse the same daemon-side validation and persistence path as FEAT-006 `register-self`. It MUST NOT bypass any validation rule, including the "no silent promotion to master" rule (the operator may set `role=master` only if the existing FEAT-006 rules already allow it from a host-driven caller).
- **FR-027**: If the named pane is already registered, the response MUST be `pane_already_registered` with the existing `agent_id` in `details` and no second `agents` row created.
- **FR-028**: If the pane identity does not match a currently-discovered pane, the response MUST be `pane_not_found` and the app MUST be instructed to call `app.scan.panes` first.
- **FR-028a**: Pane-identity matching on `app.agent.register_from_pane` MUST require **all 6 identity fields** (`container_id`, `tmux_socket`, `session_name`, `window_index`, `pane_index`, `pane_id`) to match the currently-discovered pane row byte-for-byte. Any single-field mismatch MUST return `pane_not_found` with `details = {pane_id, mismatch_field: "<first offending field>"}`. The daemon MUST NOT accept a partial-identity match even when `pane_id` alone resolves uniquely. (Round-4 Block C Q12 override.)
- **FR-028b**: When `app.agent.register_from_pane` is called with `attach_log: true` and the target container is inactive at adopt time (post-discovery), the entire adopt MUST fail with `container_inactive` and `details.container_id`. The daemon MUST NOT create the `agents` row in this case. Adopt with `attach_log: false` or `attach_log` omitted against an inactive container MAY proceed iff FEAT-006's existing rules permit it. (Round-4 Block C Q13 override.)
- **FR-028c**: When `app.agent.register_from_pane` is called with a `parent_agent_id` that does not match any registered agent row, the response MUST be `agent_not_found` with `details.agent_id = <parent_agent_id>`. This is distinct from "malformed `parent_agent_id`" (which is `validation_failed`). (Round-4 Block C Q14 override.)
- **FR-028d**: `label` on `app.agent.register_from_pane` and `app.agent.update` MUST be trimmed (leading/trailing whitespace stripped) before validation. Trimmed values containing any `\n` or `\r` byte MUST be rejected with `validation_failed.details.field == "label"`. The trimmed value MUST be ≤ **256 chars**. Empty trimmed values on `label` clear the field per FR-029a (clearable). (Round-4 Block C Q16.)

#### Operator mutation surfaces

- **FR-029**: The contract MUST expose the following mutation methods, each returning a structured success envelope containing the post-mutation state of the affected entity: `app.scan.containers`, `app.scan.panes`, `app.agent.update` (role/capability/label/project), `app.log.attach`, `app.log.detach`, `app.send_input`, `app.queue.approve`, `app.queue.delay`, `app.queue.cancel`, `app.route.add`, `app.route.remove`, `app.route.update` (enable/disable only — per FEAT-010 immutability).
- **FR-029a**: `app.agent.update` field-update semantics: an absent field in the request MUST mean "do not change". An empty-string value MUST clear the field, but only for the explicitly clearable fields `project_path` and `label`. An empty-string value on `role` or `capability` MUST be rejected with `validation_failed` and `details = {field: "<role|capability>", reason: "field is not clearable; provide a valid non-empty value"}`. `role` MUST be a value from the FEAT-006 role closed set; `capability` MUST be a non-empty string within the FEAT-006 length bound.
- **FR-029b**: `app.log.detach` MUST be success-idempotent. If the target agent has no active log attachment, the response MUST be a success envelope carrying the agent's `AgentViewModel` with `log_attached: false`. No closed-set error code is emitted for the "already detached" case. `agent_not_found` is still returned when the `agent_id` does not match any row.
- **FR-030**: Every mutation MUST be synchronous from the caller's perspective and return the final post-mutation state in the response. `app.scan.containers` and `app.scan.panes` MAY accept `wait: bool` (default `true`); when `wait == false` the response returns immediately with a `scan_id` and the app polls `app.scan.status`.
- **FR-030a**: `app.agent.update` and `app.route.update` MUST be last-write-wins. FEAT-011 MUST NOT introduce an `expected_version` / `etag` / `If-Match` field on entity-update mutations. The mutation response MUST contain the full post-update row so the app can refresh its local view immediately. The closed-set code `stale_object` MUST NOT be returned by entity-update mutations; it remains scoped to queue lifecycle / terminal-state guards (FEAT-009).
- **FR-030b**: `app.scan.containers` and `app.scan.panes` with `wait == true` MUST enforce a hard timeout of **30 s**. On timeout, the response MUST be a structured failure with `code == "scan_timeout"` and `details.scan_id` populated so the app can poll `app.scan.status` to retrieve the eventual result. A client socket disconnect during an in-flight scan MUST NOT cancel the scan; the scan MUST complete server-side and remain reachable via the same `scan_id` on a later `app.scan.status` call.
- **FR-030c**: The daemon MUST expose `app.scan.status(scan_id)` returning a success envelope with `{state, scan_kind, started_at, completed_at, result}` where `state ∈ {running, completed, failed}` (closed set) and `scan_kind ∈ {containers, panes}` (closed set). When `state ∈ {completed, failed}`, `completed_at` MUST be populated and `result` MUST mirror the post-scan summary that `wait=true` would have returned (counts and per-container/per-tmux-server breakdown for panes; container roster for containers). When `state == running`, `completed_at` MUST be `null` and `result` MUST be `null`. Scan records MUST be retained in memory only, capped at the last **100** scans per daemon process, and MUST be lost on daemon restart. An unknown or evicted `scan_id` MUST return the closed-set code `scan_not_found`. A future minor MAY introduce an additional state (e.g., `expired`) tied to an explicit wall-clock trigger; v1.0 does not define such a trigger, so the `expired` state is intentionally absent from the v1.0 closed set.
- **FR-030d**: Two or more concurrent `app.scan.<kind>` calls for the same `scan_kind` MUST coalesce on the existing in-flight scan. The second caller MUST receive the in-flight `scan_id` (with `state == "running"` when `wait=false`, or block on the same scan's terminal state when `wait=true`). Independent scans of the same kind MUST NOT run in parallel. (Round-4 Block D Q24.)
- **FR-030e**: The daemon MUST enforce a hard cap of **4 concurrent in-flight scans** across all sessions and scan kinds. The 5th scan request MUST be rejected with `validation_failed` and `details = {field: "scan_kind", reason: "too_many_scans_in_flight"}`. The cap protects against runaway concurrent docker/tmux fanout. (Round-4 Block D Q25.)
- **FR-031**: `app.send_input` MUST go through the FEAT-009 message_queue and respect the FEAT-009 permission gate and global routing kill switch. The response MUST include the resulting `message_id` and queue row state. When the FEAT-009 **per-message permission gate** refuses the call, the response MUST be `permission_denied` with `details = {reason: "feat009_permission_gate"}`. When the FEAT-009 **global routing kill switch** is off, the response MUST be `routing_disabled` (the two codes are now distinct — Round-4 Block B Q7 override). When `target_agent_id` does not match any registered agent, the response MUST be `agent_not_found` with `details.agent_id`.
- **FR-031a**: `app.send_input` MUST accept an optional `idempotency_key` (caller-supplied string). When present, the daemon MUST deduplicate against a store keyed by `(app_session_id, idempotency_key)`. A duplicate retry within the dedupe window MUST return the original `message_id` and include `deduplicated: true` on the success envelope; no second queue row is created and no duplicate audit JSONL entry is emitted. The dedupe store is per-session and in-memory only (lost across daemon restart or session close, which is acceptable because the same `app_session_id` cannot survive either). No other `app.*` mutation MUST accept `idempotency_key` in FEAT-011.
- **FR-032**: `app.route.add` / `app.route.remove` / `app.route.update` MUST go through the FEAT-010 route catalog and emit the same `route_created` / `route_updated` / `route_deleted` audit JSONL entries.

#### Error envelopes and closed-set codes

- **FR-033**: Every `app.*` response MUST use one of two envelope shapes: `{ok: true, app_contract_version, result}` for success or `{ok: false, app_contract_version, error: {code, message, details}}` for failure. The `app_contract_version` MUST always be present on the response so the app can detect drift. `error.details` MUST always be present and MUST always be a JSON object, even when empty (`{}`); it MUST NOT be `null`, an array, or a primitive.
- **FR-034**: `error.code` MUST come from a documented closed set. Initial closed set at v1.0 contains **27 entries** (additive in future minors): `app_session_required`, `app_session_expired`, `app_contract_major_unsupported`, `unknown_method`, `malformed_request`, `validation_failed`, `not_found`, `stale_object`, `pane_already_registered`, `pane_not_found`, `agent_not_found`, `route_not_found`, `queue_message_not_found`, `scan_timeout`, `scan_not_found`, `daemon_unavailable`, `socket_missing`, `socket_permission_denied`, `docker_unavailable`, `tmux_unavailable`, `container_inactive`, `log_attach_blocked`, `routing_disabled`, `permission_denied`, `host_only`, `payload_too_large`, `internal_error`. Free-form prose belongs in `error.message` and `error.details`, never in `error.code`. `malformed_request` is emitted at the wire-framing layer **before** dispatch when a request line is not valid JSON, contains trailing content after one JSON object, contains a stray `\r` byte (Round-4 Block A Q1), or contains an embedded `\x00` byte (Round-4 Block A Q2). `routing_disabled` is reserved exclusively for the FEAT-009 global kill switch being off; per-message permission-gate refusals on `app.send_input` MUST use `permission_denied` (Round-4 Block B Q7).
- **FR-034a**: For each closed-set code in FR-034, `error.details` MUST carry the required keys listed in the v1.0 registry below. Additional keys MAY be present and are treated as additive minor evolutions. Codes not listed MUST carry `error.details == {}`. The v1.0 registry: `malformed_request → {reason: string}`; `validation_failed → {field: string, reason: string}`; `app_contract_major_unsupported → {daemon_app_contract_version: string, client_app_contract_major: int}`; `pane_already_registered → {agent_id: string}`; `pane_not_found → {pane_id: string}`; `agent_not_found → {agent_id: string}`; `route_not_found → {route_id: string}`; `queue_message_not_found → {message_id: string}`; `scan_timeout → {scan_id: string}`; `scan_not_found → {scan_id: string}`; `container_inactive → {container_id: string}`; `log_attach_blocked → {agent_id: string, reason: string}`; `payload_too_large → {size_limit_bytes: int, actual_size_bytes: int}`. All other codes in FR-034 carry `{}`. Removing a required key from this registry MUST require a major bump per FR-035; adding a required key MUST also require a major bump (only optional additional keys are additive).
- **FR-034b**: `unknown_method` MUST be returned for any method name in the `app.*` namespace that the daemon does not implement at its current minor, regardless of cause (client typo, method from a future minor not yet supported by this daemon, or wholly nonexistent name). The daemon MUST NOT distinguish these cases via separate codes or via `error.details` content; clients differentiate by checking `capability_flags` (FR-039) before invoking optional methods. `unknown_method` MUST NOT mutate any daemon state and MUST be returnable without a valid session (i.e., it is checked at dispatcher time on the method name, before the session gate of FR-007 if the method is `app.preflight` or `app.hello`, otherwise after the session gate). Method names outside the `app.*` namespace MUST be handled by the existing legacy dispatcher (FEAT-002), which has its own semantics and is unaffected by FR-034b.

#### Versioning and evolution

- **FR-035**: `app_contract_version` MUST follow `MAJOR.MINOR` semantics. Within a major: only additive changes (new optional fields, new optional methods, new closed-set codes, new readiness subsystems, new capability flags). Removing or renaming any of the above MUST increment major.
- **FR-036**: A client whose declared `client_app_contract_major` does not match the daemon's major MUST receive `app_contract_major_unsupported` from `app.hello`, no session token, and `app_session_required` from every other `app.*` call.
- **FR-037**: Clients MUST treat unknown response fields as ignorable so newer daemons remain compatible.
- **FR-038**: Daemons MUST treat unknown request fields as ignorable so newer clients remain compatible within a major.
- **FR-039**: `capability_flags` returned by `app.hello` MUST declare optional methods/features available at the daemon's minor. Clients MUST check the flag before invoking an optional method. At v1.0 specifically, `capability_flags` MUST be the empty object `{}` because every FEAT-011 method is required; the field MUST still be present so clients can rely on its shape. Optional methods introduced in later minors (e.g., event stream subscribe, managed pane create) MUST add named boolean flags here additively.

#### Local-only security boundary

- **FR-040**: FEAT-011 MUST NOT alter the FEAT-002 socket-permission model for the legacy CLI-facing methods. The socket file remains user-owned, mode-restricted, and accessed by clients on the same host — including bench-container thin clients that reach it via mount, which continue to call the legacy FEAT-002..FEAT-010 methods unchanged. The host-only restriction defined in FR-042 applies to the `app.*` namespace specifically.
- **FR-041**: The daemon MUST reject any `app.*` call whose peer UID does not match the host user (same rule that already applies to existing methods via FEAT-002).
- **FR-042**: The entire `app.*` namespace is host-only. The daemon MUST reject every `app.*` call (including `app.preflight` and `app.hello`) from a bench-container peer with the closed-set code `host_only`. The host-vs-container peer distinction MUST reuse the same mechanism already required by the FEAT-009 routing-toggle host-only rule. The legacy FEAT-002..FEAT-010 namespace remains fully available to bench-container thin clients (the existing FEAT-009 routing-toggle host-only check still applies there as before).
- **FR-043**: The contract MUST NOT introduce any new persisted secret, token, key, or remote authentication primitive.

#### Observability

- **FR-044**: The daemon SHOULD emit JSONL audit entries for app-initiated mutations consistent with existing FEAT-006..FEAT-010 audit semantics. An app-initiated mutation row SHOULD carry an `origin == "app"` marker plus `app_session_id` so operators can distinguish app-driven changes from CLI-driven ones. App-initiated mutations MUST reuse the **upstream FEAT audit event names byte-for-byte** (e.g., `app.queue.approve` → `queue_approved`, `app.route.add` → `route_created`, `app.agent.register_from_pane` → `agent_registered`); FEAT-011 does NOT introduce an `app.*`-namespaced audit event-type vocabulary. Only successful commits emit audit rows; failures live in daemon stderr/logs only. `app.preflight` and `app.hello` are NOT audited (Round-4 Block G Q49). `client_id` and `client_version` are NOT included in audit rows (Round-4 Block G Q50).
- **FR-044a**: The daemon MUST serialize concurrent JSONL audit writes with a **process-wide mutex** around the audit writer. POSIX `O_APPEND` atomicity MUST NOT be relied upon. (Round-4 Block G Q48 override.)
- **FR-044b**: When the JSONL audit writer is unwritable mid-mutation (disk full, permission lost, fs error), the daemon MUST: (1) complete the SQLite commit and return the normal success envelope; (2) drop the audit row; (3) emit one stderr warning per JSONL-outage window. The daemon MUST NOT roll back the SQLite mutation. The `jsonl` readiness subsystem (FR-013) MUST reflect the outage by flipping to `degraded` or `unavailable` so operators can detect lost audit rows. (Round-4 Block G Q46 override.)
- **FR-044c**: The JSONL audit row MUST be written **after** the SQLite commit and **before** the response envelope is sent. The write is synchronously flushed. (Round-4 Block G Q47.) Combined with FR-044b, this means: SQLite commit → JSONL write attempt → (success: row durable | failure: warning logged) → response envelope sent.
- **FR-045**: `app.readiness` and `app.dashboard` MUST be cheap and side-effect-free. They MUST NOT trigger a discovery scan (the app uses `app.scan.*` explicitly when it wants one).

### Key Entities

- **App Session**: Per-connection identity held in memory by the daemon. Attributes: `app_session_token` (opaque), `app_session_id` (numeric, audit-friendly), `client_id` (informational), `client_version` (informational), `app_contract_major` (the major the client claims to speak), `connection_started_at`. Sessions are not durable.
- **App Contract Version**: A `MAJOR.MINOR` value advertised by `app.hello`. Additive evolution within a major; major bumps require client opt-in.
- **Readiness Subsystem Row**: `name`, `status ∈ {ok, degraded, unavailable}`, `reason`, optional `hint`. Aggregated to a top-level `state`.
- **Dashboard Snapshot**: A single read-only payload containing structured counts (per surface) and a small set of recent rows per surface; not durable, no global lock.
- **App-Facing Container/Pane/Agent/Log Attachment/Event/Queue Message/Route view models**: Each derived from the same underlying SQLite/JSONL state as the CLI's read surfaces, exposed with explicit ordering and derived summary fields suitable for direct UI rendering.
- **Adopt Mutation Input**: Structured pane identity (`container_id`, `tmux_socket`, `session_name`, `window_index`, `pane_index`, `pane_id`) + agent metadata (`role`, `capability`, `label`, optional `project_path`, optional `parent_agent_id`, optional `attach_log`).
- **App-Originated Mutation Audit Row**: A JSONL entry produced by an app-driven mutation, carrying `origin == "app"` and `app_session_id` for operator-side attribution.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: A control panel can render the dashboard, panes, agents, log-attachments, queue, routes, and events surfaces using `app.*` responses alone, with **zero** lines parsing human CLI text. Verified by an integration harness that asserts no `agenttower` subprocess output is consumed for any UI-rendering code path.
- **SC-002**: From process start to first rendered dashboard payload (cold start, daemon already running), the app reaches `app.dashboard` success in **≤ 500 ms** on a workstation with at least one running bench container and at least one registered agent. Measured as wall-clock from `app.hello` request send to `app.dashboard` response receive in a no-cache test.
- **SC-003**: 100% of FEAT-011 mutation methods return a structured success envelope or a closed-set error code; no method returns a free-form prose error in `error.code`. Verified by a contract test that asserts `error.code` is in the documented closed set for **every** error path covered in the contract test suite, with **zero** exceptions.
- **SC-004**: Adopt-mode round-trip — `app.scan.panes` → `app.pane.list` → `app.agent.register_from_pane` → `app.agent.detail` — completes in **≤ 2 s** wall-clock and produces an `agents` row indistinguishable from a CLI `register-self` row for the same pane, validated by a side-by-side SQLite fixture comparison.
- **SC-005**: Major-version drift is enforced: a synthetic client with mismatched major receives `app_contract_major_unsupported` from `app.hello` and `app_session_required` from every other `app.*` method, with **zero** stray fields and **zero** internal state mutation on the daemon side. Verified by a contract test using a synthetic client identity.
- **SC-006**: Local-only invariant: a packet capture during the FEAT-011 contract test suite shows **zero** non-Unix-socket I/O from the daemon attributable to FEAT-011, and the daemon process MUST NOT bind any TCP or non-Unix-domain socket during the entire test run.
- **SC-007**: Degraded-state coverage: each of the readiness failure modes called out in the Edge Cases (daemon unavailable, socket missing, schema/version incompatible, Docker unavailable, no bench containers, no panes, no registered agents) produces a structured, app-renderable state where the app can decide UI behavior **without** parsing any prose. Verified by an integration harness covering each failure mode.
- **SC-008**: Audit attribution: every app-driven mutation in the contract test suite produces at least one JSONL audit row with `origin == "app"` and the issuing `app_session_id`. The opaque `app_session_token` MUST NOT appear in any JSONL row.
- **SC-009**: Within-major additive evolution does not break older clients: a synthetic minor-N client running against a minor-(N+1) daemon completes the dashboard + adopt + queue + route flows with **zero** failures attributable to unknown response fields and **zero** failures attributable to unknown closed-set codes (clients are expected to surface unknown codes as `internal_error`-class display states without crashing).
- **SC-010**: Operator parity: at least the FEAT-006 (set role / set capability / set label), FEAT-007 (attach log / detach log), FEAT-009 (queue approve / delay / cancel), FEAT-010 (route add / remove / enable / disable), and the FEAT-006 register-self equivalent (`app.agent.register_from_pane`) are all reachable through `app.*` and each produces SQLite/JSONL state byte-for-byte identical (modulo `origin`/`app_session_id` metadata) to the equivalent CLI invocation. Verified by a fixture-comparison test.
- **SC-011**: Pagination contract: every `app.<entity>.list` honors `limit` default `50`, accepts values in `[1, 200]`, and rejects out-of-bounds or non-integer values with `validation_failed` and `details.field == "limit"`. `cursor_next` round-trips correctly across multi-page fetches, is opaque to clients, and is rejected with `validation_failed.details.field == "cursor_next"` when exceeding 512 chars, when malformed, or when the `order_by`/filter combination changes mid-pagination. Verified by a contract test exercising default, cap, under-cap, over-cap, malformed-cursor, and order-change-mid-cursor paths against every list method (FR-020, FR-020a, FR-020b).
- **SC-012**: Scan timeout + status: `app.scan.containers` / `app.scan.panes` with `wait=true` against a synthetic slow scanner returns `scan_timeout` with `details.scan_id` within `30 s ± 500 ms`; the scan completes server-side; a subsequent `app.scan.status(scan_id)` returns terminal state `{completed | failed}` with populated `completed_at` and `result`. An unknown or evicted `scan_id` returns `scan_not_found` with `details.scan_id`. The v1.0 `scan_state` closed set is exactly `{running, completed, failed}` — `expired` is absent. Verified by a contract test using a synthetic blocking scanner and a fixture that drives `scan_id` retention past the 100-record FIFO cap (FR-030b, FR-030c).
- **SC-013**: Idempotency: two `app.send_input` requests with identical `(app_session_id, idempotency_key)` produce exactly one `message_queue` row and exactly one JSONL audit row; the second response carries `deduplicated: true` and the original `message_id`. A third request with a different `idempotency_key` in the same session produces a new `message_id` and `deduplicated: false`. The dedupe store is invalidated when the session connection closes. Verified by a contract test asserting SQLite row count, JSONL row count, and response payloads (FR-031a).
- **SC-014**: Capability flags contract: at v1.0, every `app.hello` success response carries `capability_flags == {}` (exact-match against the empty object). A synthetic forward-version client receiving an unknown `capability_flags` key on a hypothetical minor-N+1 daemon MUST tolerate it (forward-compat regression test). Verified by a contract test asserting equality at v1.0 and a synthetic-future-daemon fixture for forward compatibility (FR-039).
- **SC-015**: Hints contract: every `app.dashboard` and `app.readiness` response includes a top-level `hints` array (possibly `[]`). Every emitted hint conforms to `{code, severity, message, target?}`; `code` is in the v1.0 registry; `severity` is in `{info, warning, action_required}`; `target` when present is `{kind, id}`. A fixture inducing each registered hint code (`start_bench_container`, `check_container_filter`, `register_first_agent`, `attach_logs`, `enable_first_route`, `docker_unavailable_hint`) produces the documented hint. Verified by a contract test with one fixture per hint code (FR-014a).
- **SC-016**: Default-ordering normative mapping: with a fixture of one queue row per FEAT-009 state and one agent row per FEAT-006 role, `app.queue.list` and `app.agent.list` (no `order_by` override) return rows in exactly the FR-021a integer-priority order. Verified by a fixture-comparison test that asserts byte-for-byte ordering against the published `state_priority` and `role_priority` mappings (FR-021, FR-021a).
- **SC-017**: `order_by` direction syntax: every list method accepts `<field>`, `<field>:asc`, `<field>:desc`; an invalid direction suffix returns `validation_failed.details.field == "order_by"`. Bare field uses the per-surface default direction in FR-021. Verified by a contract test exercising both directions, the bare-field default, and at least one invalid-suffix path per list method (FR-021b).
- **SC-018**: Filter exact-match only: every filter field on `app.event.list` / `app.queue.list` / `app.route.list` accepts exact-match values from its closed set or domain. Any filter value containing operator-like syntax (`<`, `>`, `~`, `*`, `LIKE`, regex metacharacters in fields not declared regex) returns `validation_failed.details.field` set to the offending field. Verified by a contract test sending operator-laden filter values per surface and asserting rejection (FR-024a).
- **SC-019**: `app.agent.update` clearable-fields semantics: an absent field leaves the row unchanged; an empty-string value on `project_path` or `label` clears that field to `null` / `""`; an empty-string value on `role` or `capability` returns `validation_failed` with `details.field == "<role|capability>"`. A non-empty `role` outside the FEAT-006 closed set returns `validation_failed.details.field == "role"`. Verified by a contract test exercising all four cases per mutable field (FR-029a).
- **SC-020**: `app.log.detach` idempotency: detaching an agent's never-attached log returns success with the post-state `AgentViewModel` carrying `log_attached: false`; detaching with an unknown `agent_id` returns `agent_not_found`. No closed-set error code is emitted for "already detached". Verified by a contract test exercising both paths and asserting SQLite `log_attachments` row count is unchanged on the idempotent path (FR-029b).
- **SC-021**: Error envelope + per-code `details` registry: every failure path covered by the contract test suite emits `{ok: false, app_contract_version, error: {code, message, details}}` where `code` matches `^[a-z][a-z0-9_]*$`, is a member of the 26-entry closed set (FR-034), and `details` is always a JSON object. Codes listed in the FR-034a registry carry their required keys with the documented types; codes not listed carry `details == {}`. Verified by a registry-driven contract test that runs every covered failure path against an enforcement table (FR-033, FR-034, FR-034a).
- **SC-022**: Host-only `app.*`: a synthetic bench-container peer calling any `app.*` method (including `app.preflight` and `app.hello`) receives `host_only` with `details == {}`. Verified by a contract test using two synthetic peers (host and container) against the full `app.*` surface plus a representative legacy method (FR-042).
- **SC-023**: Payload caps: a request line exceeding 1 MiB is rejected with `payload_too_large` and `details = {size_limit_bytes: 1048576, actual_size_bytes: <observed>}` before any handler executes; SQLite/JSONL state is unmodified. - **SC-024**: Last-write-wins on entity updates: two concurrent `app.agent.update` requests on the same `agent_id` both succeed; the final SQLite row matches the request that committed last (determined by SQLite commit order); neither response carries `stale_object`. ## Assumptions

- FEAT-001 through FEAT-010 are shipped and stable. FEAT-011 builds on existing services (FEAT-002 socket, FEAT-003 container discovery, FEAT-004 pane discovery, FEAT-005 thin client, FEAT-006 agent registration, FEAT-007 log attachment, FEAT-008 event pipeline, FEAT-009 message queue, FEAT-010 routing layer) rather than reimplementing them.
- The packaged desktop app runs on the same host as the daemon (Windows/macOS/Linux). The `app.*` namespace is host-only and bench-container callers are rejected normatively per FR-042 (the legacy FEAT-002..FEAT-010 namespace remains the channel for bench-container thin clients).
- Operator UID identifies the trust boundary. The contract assumes the host user owns the daemon socket and the running app process. Multi-user desktops are out of scope (one user, one daemon, one or more concurrent app sessions).
- Same daemon, multiple app sessions, no cross-session isolation beyond per-connection lifetime. Two concurrent apps see the same mutations land in the same SQLite/JSONL.
- The Flutter target is informational. The contract is language-agnostic and could be consumed by a Rust, Swift, or Electron client as long as it speaks newline-delimited JSON over the Unix socket.
- The CLI continues to be the primary scriptable interface; `app.*` is the structured-UI interface. The two MUST never diverge in validation/persistence behavior.
- Event subscription/push (a long-running daemon→app event stream) is **deferred**. The app polls list endpoints with cursor pagination in FEAT-011. Adding a subscription is an additive minor change later.
- Managed tmux session/pane creation is **deferred to FEAT-013**. The adopt-existing-panes workflow is the only path to a registered agent for FEAT-011.
- No Antigravity, no TUI, no mobile, no remote multi-host, no hosted SaaS — all explicitly out of scope per the FEAT-011 brief.
- The opaque `app_session_token` is **not** a security boundary against malicious local processes; the security boundary remains the Unix socket file permissions and SO_PEERCRED UID check inherited from FEAT-002. The token's job is connection-scoped identity and audit attribution, nothing more.
- FEAT-008's `event_id` is assumed to be **monotonically non-decreasing within a given daemon process**. FR-021's default `events by event_id DESC` ordering and SC-016's normative ordering tests depend on this. If FEAT-008 ever loses monotonicity (e.g., during a downgrade or crash recovery that resets the counter), FEAT-011 readers will still order what they receive but may surface out-of-time-order rows; this is documented as an upstream invariant, not a FEAT-011 contract guarantee.

## Out of Scope

- Flutter UI implementation (deferred to FEAT-012).
- Managed tmux session/pane creation, automatic agent launch (deferred to FEAT-013).
- A hosted website, SaaS backend, mobile app, remote multi-host control, TUI, Antigravity support — all explicitly excluded.
- Network listeners of any kind (TCP, HTTP, WebSocket). The contract is Unix-socket-only.
- Per-user authentication, RBAC, or sub-user permission models — the trust boundary remains "same host UID."
- A long-running daemon→app push/subscribe event stream — left for a follow-up additive minor release.
- Schema migration of existing SQLite tables purely for app-rendering convenience — FEAT-011 adds derived fields in the response envelopes, not new persisted columns.
- Cross-host federation or cluster mode — the daemon remains a single-host process.
