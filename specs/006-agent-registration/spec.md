# Feature Specification: Agent Registration and Role Metadata

**Feature Branch**: `006-agent-registration`
**Created**: 2026-05-07
**Status**: Draft
**Input**: User description: "FEAT-006 Agent Registration and Role Metadata: let humans and agents turn discovered tmux panes inside bench containers into registered AgentTower agents. Provides `agenttower register-self`, `agenttower list-agents`, `agenttower set-role`, `agenttower set-label`, `agenttower set-capability`. Defines durable agent identity bound to a FEAT-004 pane composite key, idempotent re-registration, mutable vs immutable fields, swarm parent linkage, human-controlled master promotion (no silent privilege escalation), and effective permission derivation from role. Out of scope: prompt delivery, route configuration, log attachment, event ingestion, multi-master arbitration, automatic swarm inference, TUI/web UI, in-container daemon."

## Clarifications

### Session 2026-05-07 (continued)

- Q: How are case differences handled for closed-set tokens (role, capability) and identifiers (agent_id, parent_agent_id, container_id)? → A: Strict case-sensitive matching everywhere; no normalization or case-folding. Locked behavior: (1) closed-set tokens (`role`, `capability`) are lowercase and case-sensitive; mixed-case inputs (`Slave`, `MASTER`, `Codex`) are rejected with `value_out_of_set`, NOT rewritten; (2) `agent_id`, `parent_agent_id`, and `container_id` are lowercase hex and case-sensitive; mixed-case inputs (`AGT_abc...`, `ABC123def456`) are rejected, NOT normalized; (3) every validator, filter, lookup, and comparison site MUST treat case differences as distinct values; (4) closed-set error messages list canonical lowercase tokens only.
- Q: How is the race between `set-role --role master --confirm` validation (read of `agent.active` and `container.active`) and a concurrent FEAT-004 reconciliation that could flip either flag to `0` resolved? → A: One SQLite write transaction with an atomic re-check inside. Locked behavior: (1) master promotion MUST run inside a single `BEGIN IMMEDIATE` SQLite write transaction; (2) immediately after `BEGIN IMMEDIATE`, the daemon MUST re-read the target agent's `active` column AND the bound container's `active` column from `containers` in the same transaction; (3) if either is `0` (inactive), the daemon MUST `ROLLBACK` and return the closed-set error code `agent_inactive` — no agent row is mutated, no `effective_permissions` is recomputed, no JSONL audit row is appended; (4) only when both are `1` (active) MAY the daemon write the role change, recompute `effective_permissions`, append the audit row, and `COMMIT`. SQLite's `BEGIN IMMEDIATE` serializes the master-promotion transaction against the FEAT-004 reconciliation transaction at the SQLite level, closing the validate-then-write window without requiring cross-subsystem mutex coordination.
- Q: How does the FEAT-006 per-(container, pane-key) registration mutex interact with FEAT-004 pane reconciliation that also writes to `agents` rows (FR-009, FR-009a)? → A: SQLite-level serialization is the contract; the FEAT-006 mutex covers `register_agent` vs `register_agent` only. Locked behavior: (1) the per-(container_id, pane_composite_key) advisory mutex (FR-038) MUST cover `register_agent` requests against other `register_agent` requests addressing the same composite key, and nothing else; (2) FEAT-004 pane reconciliation MUST NOT acquire the FEAT-006 per-key registration mutex; the two subsystems remain decoupled; (3) cross-subsystem ordering between a `register_agent` transaction and a FEAT-004 reconciliation transaction touching the same `agents` row is provided exclusively by SQLite's `BEGIN IMMEDIATE` semantics; (4) both transactions remain atomic — neither can observe a partial write from the other; (5) the transaction that commits last wins for any overlapping mutable column (e.g., if `register_agent` sets `active=1` and a concurrent reconciliation sets `active=0` for the same row, whichever commits last is the post-state visible to subsequent reads); (6) the daemon MUST surface `SQLITE_BUSY` as `internal_error` (FR-035) without retry — the operator can re-invoke the CLI if needed.
- Q: What value is recorded in the audit row's `confirm_provided` field when `set-role` is called with `--confirm` but `--confirm` was not required (e.g., demotion from master, or set-role to a non-master role)? → A: The literal request value, never rewritten. Locked behavior: (1) `confirm_provided` MUST record the literal boolean the operator passed in the `confirm` request parameter — `true` if `--confirm` was passed, `false` if it was not; (2) the value MUST NOT be rewritten based on whether `--confirm` was *required* by the transition; (3) demotion from master with a redundant `--confirm` MUST log `confirm_provided: true`; (4) `set-role` to any non-master role with redundant `--confirm` MUST also log `confirm_provided: true`; (5) `register-self` creation (where `--confirm` is not a valid flag at all) MUST log `confirm_provided: false`; (6) the audit log captures operator intent verbatim — consumers that need "was confirm required" derive it from `prior_role` + `new_role`.

### Session 2026-05-07

- Q: When `register-self` is re-run on an existing agent without `--role` (or other mutable flags), should the CLI default values overwrite the stored row, or should omitted flags leave the stored value unchanged? → A: CLI sends only flags the user explicitly passed; daemon treats absent fields as "leave unchanged." Argparse-style defaults are not transmitted, so idempotent re-runs preserve role, capability, label, project_path, and parent unless the user explicitly changes them.
- Q: What event updates `last_seen_at`? → A: FEAT-004 pane reconciliation. Every FEAT-004 scan that observes the agent's bound pane as active updates `last_seen_at` to the scan time, in the same SQLite transaction as the pane reconciliation. `last_registered_at` (explicit registration intent) and `last_seen_at` (passive pane observation) are independent.
- Q: What happens when `register-self` is re-run on an existing agent and explicitly passes `--parent` with a value that differs from the stored immutable `parent_agent_id`? → A: Reject with closed-set error code `parent_immutable`. Locked behavior: (1) re-registration that explicitly re-passes the same `--parent` value as the stored `parent_agent_id` is a no-op success (returns the existing `agent_id` and updates `last_registered_at` per FR-007); (2) re-registration that explicitly passes a different `--parent` value MUST be rejected with closed-set code `parent_immutable`; (3) on that rejection, no mutable fields (role, capability, label, project_path) MUST be updated, even if the same call also supplied changes to other fields — the rejection is atomic, all-or-nothing.
- Q: Does the initial role assignment on first `register-self` append a JSONL audit record? → A: Yes — every first successful `register-self` for a pane appends exactly one audit row with `prior_role: null`. Locked behavior: (1) the first successful `register-self` for a pane MUST write one JSONL audit row regardless of which role was assigned; (2) the audit row's `prior_role` MUST be the JSON literal `null`; (3) the default-role case (no `--role` flag, role assigned as `unknown`) MUST also write the row; (4) idempotent re-registration where the stored `role` does not change MUST NOT write a new audit row (per FR-027 no-op writes are not logged); only writes that actually transition `agents.role` from one value to another (including `null → X` on creation) are logged.
- Q: What fields does the default (non-JSON) `list-agents` form display? → A: Tab-separated table with header row and a locked, fixed column set in this exact order: `AGENT_ID`, `LABEL`, `ROLE`, `CAPABILITY`, `CONTAINER`, `PANE`, `PROJECT`, `PARENT`, `ACTIVE`. Locked behavior: (1) output MUST be tab-separated values (TSV); (2) the header row is required (it is the first line of output, before any agent rows); (3) `CONTAINER` MUST render as the bare 12-character short container id (no prefix); (4) `PARENT` MUST render as the full `agt_<12-character-lowercase-hex>` form (16 chars total) when `parent_agent_id` is non-null and the literal `-` (single ASCII hyphen) when null; (5) `PANE` MUST render as `<session>:<window>.<pane>` (FEAT-004 short form); (6) `AGENT_ID` MUST render as the full `agt_<12-character-lowercase-hex>` form so it is consistent with `PARENT`; (7) only `CONTAINER` uses the bare 12-char short form — every agent-id-bearing column uses the full `agt_<12-hex>` form so operator pipelines (grep, awk by full id) compose uniformly across columns; (8) any future fields added to the agent record MUST be exposed via `--json` or a separately-introduced `--wide` flag — they MUST NOT be added to the default form, so the column set above is a stable contract for operator scripts and snapshot tests.

## User Scenarios & Testing *(mandatory)*

### User Story 1 - Register an agent from inside a bench-container tmux pane (Priority: P1)

A developer working inside a tmux pane in a running bench container runs `agenttower register-self --role slave --capability codex --label codex-01 --project /workspace/acme`, the host daemon binds that command to the current pane (resolved from FEAT-005 container identity and `$TMUX` / `$TMUX_PANE`, cross-checked against the FEAT-004 pane registry), persists a durable agent record with a stable `agent_id`, and the same `agenttower list-agents` invocation surfaces the new agent with its container, pane, role, label, capability, project path, and effective permission summary.

**Why this priority**: Registration is the smallest end-to-end value slice for FEAT-006. Without it, FEAT-007 log attachment, FEAT-008 events, FEAT-009 input delivery, and FEAT-010 routing have nothing to address. It is also the slice that exercises every other dependency in the stack at once: the FEAT-002 daemon socket, the FEAT-003 container registry, the FEAT-004 pane registry, and the FEAT-005 thin-client identity detection.

**Independent Test**: Can be fully tested by spawning the host daemon under the existing FEAT-002 test harness, seeding the FEAT-003 `containers` and FEAT-004 `panes` tables with one active bench container that hosts one active tmux pane, then running `agenttower register-self --role slave --capability codex --label codex-01 --project /workspace/acme` from a second subprocess whose environment simulates "inside that bench container, in that tmux pane" (the FEAT-005 detection signals are overridden to resolve to the seeded container, `$TMUX` / `$TMUX_PANE` resolve to the seeded pane). The CLI MUST exit `0`, return a stable `agent_id`, and a follow-up `agenttower list-agents` MUST show the agent with every required field populated.

**Acceptance Scenarios**:

1. **Given** the daemon is running, FEAT-003 has persisted one active bench container, FEAT-004 has persisted one active pane in that container, and the caller's environment resolves to that container and pane, **When** the user runs `agenttower register-self --role slave --capability codex --label codex-01 --project /workspace/acme`, **Then** the CLI exits `0`, prints the assigned `agent_id`, the role, the label, the capability, and the project path; the daemon persists exactly one new agent row bound to the pane's FEAT-004 composite key; and `agenttower list-agents` shows that row with `active=true`.
2. **Given** an agent has already been registered for the current pane, **When** the user re-runs `agenttower register-self` from the same pane with the same arguments, **Then** the CLI exits `0` with the same `agent_id`, no new agent row is created, and `last_registered_at` is updated.
3. **Given** an agent has already been registered for the current pane, **When** the user re-runs `agenttower register-self` from the same pane with a different `--label`, `--capability`, or `--project`, **Then** the CLI exits `0` with the same `agent_id`, the changed mutable fields are persisted, and `list-agents` reflects the new values.
4. **Given** `agenttower list-agents --json` is invoked while two agents exist (one in container A, one in container B), **When** the command runs, **Then** the JSON output contains exactly two objects, each with `agent_id`, `label`, `role`, `capability`, `parent_agent_id` (nullable), `container_id`, `container_name`, `container_user`, `tmux_socket_path`, `tmux_session_name`, `tmux_window_index`, `tmux_pane_index`, `tmux_pane_id`, `pane_pid`, `cwd`, `project_path`, `effective_permissions`, `created_at`, `last_registered_at`, `last_seen_at`, and `active`.

### User Story 2 - Human-controlled role, label, and capability changes (with master safety) (Priority: P2)

A developer who registered a pane as a `slave` decides it should serve as a `master` for the project's spec-kit workflow. They run `agenttower set-role --target <agent-id> --role master --confirm`, the daemon promotes the agent, and `list-agents` shows the new role plus the updated effective permissions. A second attempt to promote a different agent without `--confirm` is rejected. A third attempt to call `agenttower register-self --role master` is also rejected, regardless of `--confirm`, because master privilege MUST never be granted as a side effect of a self-registration handshake — only by an explicit, deliberate operator command on an existing agent record.

**Why this priority**: Master promotion is the single highest-stakes operation FEAT-006 introduces. Architecture §9 states "AgentTower must not silently grant master privileges"; PRD §9 states routing must be auditable and never escalate without explicit permission. This story isolates and tests the master-safety boundary. It is P2 (not P1) because basic registration is more foundational, but no later feature can ship until this safety property is proven.

**Independent Test**: Can be fully tested by seeding two registered agents (`slave`, `slave`) and exercising the closed set of role-change paths: (a) `set-role --role master --confirm`, (b) `set-role --role master` (no `--confirm`), (c) `register-self --role master` from a brand-new pane, (d) `register-self --role master --confirm` from a brand-new pane, (e) `set-role --role slave --confirm` (demotion). Paths (a) and (e) MUST succeed; paths (b), (c), and (d) MUST exit non-zero with a closed-set error code and the agent's role MUST remain unchanged. `list-agents` MUST reflect the only successful changes.

**Acceptance Scenarios**:

1. **Given** an agent exists with `role=slave`, **When** the user runs `agenttower set-role --target <agent-id> --role master --confirm`, **Then** the CLI exits `0`, the role is updated to `master`, `effective_permissions.can_send_to` becomes `["slave", "swarm"]`, and a JSONL audit record is appended.
2. **Given** an agent exists with `role=slave`, **When** the user runs `agenttower set-role --target <agent-id> --role master` (without `--confirm`), **Then** the CLI exits non-zero with `error: master role assignment requires --confirm` and the agent's role remains `slave`.
3. **Given** the caller's current pane is not yet registered, **When** the user runs `agenttower register-self --role master --confirm`, **Then** the CLI exits non-zero with `error: register-self cannot assign role=master; register first, then run \`agenttower set-role --role master --confirm\`` and no agent row is created.
4. **Given** an agent exists with `role=master`, **When** the user runs `agenttower set-label --target <agent-id> --label claude-main`, **Then** the CLI exits `0`, the label is updated, and the role and effective permissions are unchanged.
5. **Given** an agent exists with `role=slave` and `capability=codex`, **When** the user runs `agenttower set-capability --target <agent-id> --capability claude`, **Then** the CLI exits `0`, the capability is updated, and the role and effective permissions are unchanged.

### User Story 3 - Register a swarm child under an existing slave (Priority: P3)

A slave agent (Codex CLI in a tmux pane) spawns a child swarm worker (a Claude CLI sub-pane). The slave's master prompts the child to run `agenttower register-self --role swarm --parent <slave-agent-id> --capability claude --label claude-swarm-01`, the daemon validates the parent linkage (parent exists, parent is active, parent has `role=slave`), persists the swarm agent with its parent reference, and `agenttower list-agents` shows the swarm hierarchically under its parent slave with the swarm's effective permissions derived from `role=swarm`.

**Why this priority**: Architecture §11 makes swarm parentage a first-class concept of the MVP routing model: masters route work to slaves; slaves spawn swarm children; the registry must be able to represent that hierarchy explicitly. FEAT-006 only needs to support *explicit* swarm registration; *automatic swarm inference* is deferred to FEAT-010. This story is P3 because the only consumer of swarm parentage in MVP is FEAT-009 / FEAT-010, but the registration metadata MUST land here so those features find a stable shape.

**Independent Test**: Can be fully tested by registering a `slave` agent, then registering a second agent in a different tmux pane with `--role swarm --parent <slave-agent-id>`, then exercising the closed set of parent-validation failure paths: missing parent, parent inactive, parent role is `master`/`shell`/`unknown`/`test-runner`, parent is itself a `swarm`. The valid path MUST succeed and `list-agents` MUST surface the parent linkage; every failure path MUST exit non-zero with a closed-set error code and no agent row MUST be created.

**Acceptance Scenarios**:

1. **Given** a `slave` agent with `agent_id=agt_aaa` exists and is active, and the caller's current pane is a different active pane in the same bench container (or a different bench container), **When** the user runs `agenttower register-self --role swarm --parent agt_aaa --capability claude --label claude-swarm-01`, **Then** the CLI exits `0`, returns a new `agent_id`, the swarm agent is persisted with `parent_agent_id=agt_aaa`, and `list-agents` shows the swarm with its parent reference.
2. **Given** the caller passes `--parent <unknown-id>`, **When** they run `agenttower register-self --role swarm --parent <unknown-id> --capability claude`, **Then** the CLI exits non-zero with `error: parent agent <unknown-id> not found`, no agent row is created, and the closed-set error code `parent_not_found` is returned over the socket.
3. **Given** the caller passes `--parent <id>` where the parent's role is `master`, `test-runner`, `shell`, `unknown`, or `swarm`, **When** they run `agenttower register-self --role swarm --parent <id> --capability claude`, **Then** the CLI exits non-zero with `error: parent agent must have role=slave (got <role>)`, no agent row is created, and the closed-set error code `parent_role_invalid` is returned.
4. **Given** the caller passes `--parent <id>` where the parent agent is `active=false`, **When** they run `agenttower register-self --role swarm --parent <id> --capability claude`, **Then** the CLI exits non-zero with `error: parent agent <id> is inactive`, no agent row is created, and the closed-set error code `parent_inactive` is returned.
5. **Given** the caller runs `agenttower register-self --role swarm` without `--parent`, **When** the command runs, **Then** the CLI exits non-zero with `error: --role swarm requires --parent <agent-id>`, no agent row is created, and the closed-set error code `swarm_parent_required` is returned.
6. **Given** the caller runs `agenttower register-self --parent <id>` with a non-swarm role (`master`, `slave`, `test-runner`, `shell`, `unknown`), **When** the command runs, **Then** the CLI exits non-zero with `error: --parent only valid with --role swarm`, and the closed-set error code `parent_role_mismatch` is returned.
7. **Given** a swarm agent has been registered under a parent slave, **When** the user runs `agenttower list-agents --json`, **Then** the swarm row's `parent_agent_id` is the parent's id and the parent row carries no swarm-specific marker (parentage is encoded only on the child, not duplicated on the parent).

### Edge Cases

- The caller is on the host shell (not inside any container); FEAT-005 identity detection returns `host_context`. `register-self` MUST refuse with `error: register-self requires running inside a bench container; FEAT-006 MVP does not register host-only panes` and the closed-set error code `host_context_unsupported`.
- The caller is inside a container but FEAT-005 identity detection returns `no_match`, `no_candidate`, or `multi_match` for the container. `register-self` MUST refuse with the closed-set error code `container_unresolved` and surface the FEAT-005 candidate value, signal source, and actionable next step ("run `agenttower scan --containers` from the host").
- The caller's `$TMUX` is unset or `$TMUX_PANE` is unset/malformed. `register-self` MUST refuse with the closed-set error code `not_in_tmux` and instruct the caller to run inside an active tmux pane.
- The caller's resolved container is active in the FEAT-003 registry but the caller's tmux pane (composite key) is not present in the FEAT-004 `panes` table. `register-self` MUST trigger one focused FEAT-004 pane rescan scoped to the caller's container (re-using the FEAT-004 `scan_panes` codepath bounded to a single container) before declaring the pane unknown. If after the focused rescan the pane is still absent, `register-self` MUST refuse with the closed-set error code `pane_unknown_to_daemon` and instruct the caller to run `agenttower scan --panes` from the host.
- The caller's resolved tmux pane is in the FEAT-004 registry but `active=false`. `register-self` MUST treat this as the same `pane_unknown_to_daemon` reason path: trigger one focused rescan; if the pane reappears as active, proceed; otherwise refuse.
- The same composite pane key was previously bound to an agent that is now `active=false` (the pane disappeared and reappeared). Re-running `register-self` from that pane MUST re-activate the existing agent row (preserving `agent_id`, `created_at`, and `parent_agent_id`) and update mutable fields rather than creating a new agent.
- A second register-self call comes from a *different* composite pane key than the first registration of an agent. AgentTower MUST treat this as a *new* agent (bound to the new pane key); the old agent record MUST stay in history with whatever `active` flag the FEAT-004 reconciliation last assigned. FEAT-006 MUST NOT attempt to re-bind an existing agent to a different pane.
- Two `register-self` calls run concurrently from the same pane (e.g., a wrapper script and a manual invocation). Both MUST converge on the same `agent_id`; the second call's mutable-field writes MUST not race with the first call's insert. Concurrent registrations from *different* panes MUST proceed independently.
- The label, project path, or any other free-text field contains NUL bytes, terminal control bytes, or shell metacharacters. The fields MUST be sanitized using the FEAT-004 sanitization rules and bounded per-field (label ≤ 64 chars, project_path ≤ 4096 chars). Oversized values MUST be rejected with `error: field <name> exceeds maximum length <n>` and the closed-set error code `field_too_long` rather than silently truncated.
- A `--role`, `--capability`, or `--parent` value is outside the closed set. The CLI MUST reject with the closed-set error code `value_out_of_set` and list the valid values in the actionable message.
- `set-role --target <id> --role swarm` is requested for an agent that has no `parent_agent_id`. The change MUST be rejected with the closed-set error code `swarm_parent_required` (a swarm role can only be assigned during `register-self --parent`, never via `set-role` alone). To convert an existing non-swarm agent into a swarm, the operator MUST re-register from the original pane with `--role swarm --parent <id>`.
- `set-role --target <id> --role <r>` for an `<id>` that does not exist or is `active=false`. The CLI MUST reject with `error: agent <id> not found` (closed-set code `agent_not_found`) or `error: agent <id> is inactive` (`agent_inactive`) respectively.
- `set-role --target <id> --role master --confirm` for an agent in container that is itself `active=false` (FEAT-003 marked the container inactive between scans). The CLI MUST reject with `agent_inactive` and refuse the promotion; the operator must re-establish the pane in an active container before re-attempting promotion.
- The daemon receives a `register-self` call whose container_id and pane composite key resolve cleanly, but the bench user (FEAT-004 `container_user`) returned by FEAT-003/FEAT-004 differs from the socket peer's effective uid name. FEAT-006 MUST proceed with registration (the FEAT-002 socket-file authorization already gates host-user-only access; the in-container bench user is descriptive metadata, not an authorization boundary) but MUST surface the bench user verbatim in `list-agents` so the operator can audit it.
- The daemon is a *newer* schema than this CLI build supports. `register-self`, `list-agents`, `set-role`, `set-label`, and `set-capability` MUST inherit the FEAT-005 forward-compat policy: surface `schema_version_newer` and refuse the call without corrupting state.
- The daemon is unreachable. Every CLI in this feature MUST exit with the existing FEAT-002 daemon-unavailable exit code (`2`) and message; the daemon stays alive (it never saw the call).
- A `register-self` invocation specifies `--role unknown` explicitly. The call MUST be accepted (the `unknown` role is part of the closed set) but the agent's `effective_permissions` MUST yield `can_send=false, can_receive=false` (architecture §18: "unknown cannot send or receive input").
- A `--project` path is relative, contains `..`, or contains a NUL byte. The CLI MUST reject with the closed-set error code `project_path_invalid` and refuse to register; the project path MUST be a non-empty absolute path with no NUL bytes. The path MUST NOT be required to exist on the host filesystem (it is observable inside the container, not on the host) — only its shape is validated.
- The agent registry is queried with `list-agents --container <id-or-name>` where the filter matches no rows. The CLI MUST exit `0` with an empty result set (this is not an error — it is a valid query outcome).
- The agent registry contains an inactive agent whose pane has been reused by FEAT-004 for a different agent registration in a later scan. Both rows MUST coexist; the inactive agent's history is preserved, the new agent has its own `agent_id`, and `list-agents --active-only` shows only the active one.

## Requirements *(mandatory)*

### Functional Requirements

#### Identity & lifecycle

- **FR-001**: AgentTower MUST persist agents in a new SQLite `agents` table whose primary key is a stable, opaque `agent_id` of the form `agt_<12-character-lowercase-hex>` (96 bits of entropy, generated on first registration of a pane and never regenerated for that agent record). The format is strictly case-sensitive: only `[0-9a-f]` is accepted in the hex portion; mixed-case or uppercase inputs (e.g., `agt_ABC123def456`) MUST be rejected with `value_out_of_set` rather than normalized to lowercase. On `INSERT`, the daemon MUST handle PK-collision via a bounded retry loop (≤ 5 attempts) under the per-(container_id, pane_composite_key) registration mutex; an exhausted retry budget MUST surface as `internal_error` (FR-035) and the daemon MUST stay alive. Five attempts is overwhelmingly conservative at 96-bit entropy — expected retries at MVP scale are essentially zero.
- **FR-002**: Each agent record MUST carry, as immutable fields set at creation time: `agent_id`, `created_at`, the FEAT-004 pane composite key tuple `(container_id, tmux_socket_path, tmux_session_name, tmux_window_index, tmux_pane_index, tmux_pane_id)`, and `parent_agent_id` (nullable; only populated when the original `register-self` used `--role swarm --parent <id>`).
- **FR-003**: Each agent record MUST carry, as mutable fields that `register-self` and the explicit `set-*` commands may update: `role`, `capability`, `label`, `project_path`, `last_registered_at`, `last_seen_at`, `active`, and the derived `effective_permissions` JSON column (recomputed on every role write).
- **FR-004**: The set of valid `role` values is the closed set `{master, slave, swarm, test-runner, shell, unknown}`. Tokens are lowercase and case-sensitive; mixed-case inputs (`Slave`, `MASTER`, `Test-Runner`) MUST be rejected with `value_out_of_set` and MUST NOT be normalized. Any value outside the closed set MUST be rejected with the closed-set error code `value_out_of_set`. The actionable message MUST list the canonical lowercase tokens.
- **FR-005**: The set of valid `capability` values is the closed set `{claude, codex, gemini, opencode, shell, test-runner, unknown}`. Tokens are lowercase and case-sensitive; mixed-case inputs (`Claude`, `CODEX`) MUST be rejected with `value_out_of_set` and MUST NOT be normalized. Any value outside the closed set MUST be rejected with the closed-set error code `value_out_of_set`. The actionable message MUST list the canonical lowercase tokens.
- **FR-006**: An agent's binding to a FEAT-004 pane composite key is immutable for the life of the agent record. A `register-self` call from a *different* composite key MUST create a new agent record rather than re-binding an existing one. FEAT-006 MUST NOT provide any CLI surface for re-binding; that capability is deferred to a later feature.
- **FR-007**: When `register-self` is called from a composite pane key that already has an agent record, AgentTower MUST treat the call as idempotent: the existing `agent_id` is returned, mutable fields supplied on the call replace the corresponding stored fields, mutable fields not supplied are left unchanged, and `last_registered_at` is updated to the current daemon clock. The agent's `created_at`, `parent_agent_id`, and pane composite key MUST NOT change. "Supplied" means the user explicitly passed the flag on the command line; argparse-style CLI defaults (e.g., `--role unknown`, `--capability unknown`, `--label ""`, `--project ""`) MUST NOT be transmitted on the wire when the user did not pass the flag, so idempotent re-runs preserve the stored values rather than silently demoting role or clearing label/project. The CLI MUST distinguish "flag omitted" from "flag explicitly set to default value" and the daemon's `register_agent` request envelope MUST encode each mutable field as either present-with-value or absent.
- **FR-008**: When `register-self` is called from a composite pane key whose existing agent record is `active=false` (the pane was previously registered, then disappeared, then reappeared), AgentTower MUST re-activate the existing record (set `active=true`, update mutable fields per FR-007) rather than creating a new agent.
- **FR-009**: An agent's `active` flag MUST mirror the bound pane's FEAT-004 `active` flag at the most recent FEAT-004 pane scan that observed (or failed to observe) the pane. When FEAT-004 transitions a pane from active to inactive, the bound agent MUST also transition to inactive in the same SQLite transaction; when FEAT-004 transitions a pane from inactive to active (the pane reappeared), the bound agent's flag MUST NOT auto-flip — re-activation requires an explicit `register-self` call from inside that pane.
- **FR-009a**: Every FEAT-004 pane scan that observes an agent's bound pane as active MUST update the agent's `last_seen_at` column to the scan timestamp in the same SQLite transaction as the pane reconciliation. Scans that fail to observe the bound pane (or observe it as inactive) MUST NOT update `last_seen_at`. `last_seen_at` is independent of `last_registered_at`: the former records passive pane observation by the FEAT-004 scanner; the latter records explicit `register-self` intent. FEAT-006 MUST NOT update `last_seen_at` from any other codepath (CLI calls, list-agents, set-role, set-label, set-capability MUST NOT touch `last_seen_at`).

#### Master safety boundaries

- **FR-010**: `register-self` MUST reject any call that includes `--role master`, regardless of whether `--confirm` is also passed, with the closed-set error code `master_via_register_self_rejected` and the actionable message `register-self cannot assign role=master; register first, then run \`agenttower set-role --role master --confirm\``.
- **FR-011**: `set-role --role master` MUST require an explicit `--confirm` flag. Calls without `--confirm` MUST be rejected with the closed-set error code `master_confirm_required` and MUST NOT modify the agent record. Calls with `--confirm` MUST succeed if and only if the target agent exists, is `active=true`, and the target's bench container is `active=true`. The activeness check MUST be performed atomically with the role write inside one `BEGIN IMMEDIATE` SQLite transaction: (a) `BEGIN IMMEDIATE`; (b) re-SELECT `agents.active` for the target AND `containers.active` for the agent's bound `container_id`; (c) if either column is `0`, `ROLLBACK` and return `agent_inactive` (no role mutation, no `effective_permissions` recomputation, no JSONL audit row); (d) UPDATE the role and `effective_permissions`; (e) append the JSONL audit row; (f) `COMMIT`. SQLite's `BEGIN IMMEDIATE` serializes the master-promotion transaction against any concurrent FEAT-004 reconciliation transaction so the validate-then-write window is closed at the SQLite level.
- **FR-012**: `set-role --role <r>` for `<r>` ∈ `{slave, test-runner, shell, unknown}` MUST NOT require `--confirm`. `set-role --role swarm` MUST be rejected (closed-set code `swarm_role_via_set_role_rejected`) — swarm role assignment is only valid via `register-self --role swarm --parent <id>` because it requires the parent linkage in the same operation.
- **FR-013**: Demotion from `master` to any other valid role MUST NOT require `--confirm`. The current FEAT-006 master safety boundary is asymmetric: promotion requires explicit confirmation; demotion does not. (FEAT-009 / FEAT-010 may layer additional checks for in-flight prompts when those features ship; FEAT-006 MUST NOT pre-empt that design.)
- **FR-014**: Every successful role transition (any path that mutates `agents.role`, including the initial assignment when a brand-new agent row is created) MUST append exactly one JSONL audit record containing: `agent_id`, `prior_role`, `new_role`, `confirm_provided` (boolean), `socket_peer_uid`, and the daemon clock timestamp. The initial assignment on first `register-self` MUST be logged with `prior_role: null` and `confirm_provided: false`, and this MUST happen regardless of which role was assigned (including the default `--role unknown` case). `confirm_provided` MUST be the literal boolean the operator passed in the request's `confirm` parameter — `true` when `--confirm` was passed by the CLI caller, `false` when it was not — and MUST NOT be rewritten based on whether `--confirm` was *required* by the transition. Demotion from master with a redundant `--confirm` MUST log `confirm_provided: true`; `set-role` to any non-master role with redundant `--confirm` MUST also log `confirm_provided: true`; `register-self` creation MUST log `confirm_provided: false`. Idempotent re-registration that does not change `agents.role` MUST NOT append a new audit row (consistent with FR-027 — no-op writes are not logged); only writes that actually transition `agents.role` from one value to another are logged. Failed role transitions MUST NOT append a JSONL record (failures are surfaced through the socket response and CLI exit code only).

#### Swarm parent linkage

- **FR-015**: `register-self --role swarm` MUST require `--parent <agent-id>`. Missing `--parent` MUST be rejected with the closed-set error code `swarm_parent_required`.
- **FR-016**: `register-self --parent <agent-id>` MUST require `--role swarm`. Combining `--parent` with any other role MUST be rejected with the closed-set error code `parent_role_mismatch`.
- **FR-017**: When `--parent <agent-id>` is supplied, AgentTower MUST resolve the parent in the `agents` table and validate: (a) the parent exists (else `parent_not_found`); (b) the parent is `active=true` (else `parent_inactive`); (c) the parent has `role=slave` (else `parent_role_invalid`). Any failure MUST short-circuit the call before any agent row is written.
- **FR-018**: A swarm agent's `parent_agent_id` MUST be persisted as immutable metadata on creation (per FR-002). FEAT-006 MUST NOT provide any CLI surface for changing `parent_agent_id`; re-parenting is deferred. If the parent agent is later demoted, deleted, or marked inactive, the swarm child's `parent_agent_id` MUST remain pointing at the historical parent; downstream features (FEAT-009, FEAT-010) decide what to do with stale parent links.
- **FR-018a**: When `register-self` is re-run on an existing agent and explicitly passes `--parent <id>`: (a) if `<id>` equals the stored `parent_agent_id`, the call MUST proceed as an idempotent no-op success per FR-007 (returns the same `agent_id`, updates `last_registered_at`, applies any other supplied mutable fields); (b) if `<id>` differs from the stored `parent_agent_id` (including the case where the stored value is NULL and `<id>` is non-null, or the stored value is non-null and `<id>` is null/different), the call MUST be rejected with the closed-set error code `parent_immutable`, no mutable fields (role, capability, label, project_path) MUST be updated by this call even if other fields were also supplied, no JSONL audit record MUST be appended, and the SQLite transaction MUST be rolled back. The rejection is atomic and all-or-nothing.
- **FR-019**: `parent_agent_id` MUST always reference an agent with `role=slave` *at the time of swarm creation*. AgentTower MUST NOT enforce any continuing relationship constraint after creation (that is a routing concern for FEAT-009/FEAT-010).
- **FR-020**: Nested swarms (a swarm whose parent is itself a swarm) MUST be rejected at registration time per FR-017(c). Architecture §11 keeps the MVP swarm tree exactly two levels deep (slave → swarm); deeper nesting is deferred.

#### Effective permission derivation

- **FR-021**: AgentTower MUST materialize `effective_permissions` on every agent row as a JSON object with the closed-set fields `{can_send: bool, can_receive: bool, can_send_to_roles: [<role>...]}`, recomputed on every write that mutates `role`. The derivation rule for FEAT-006 is: `master` → `{can_send: true, can_receive: false, can_send_to_roles: ["slave", "swarm"]}`; `slave` → `{can_send: false, can_receive: true, can_send_to_roles: []}`; `swarm` → `{can_send: false, can_receive: true, can_send_to_roles: []}`; `test-runner` → `{can_send: false, can_receive: false, can_send_to_roles: []}`; `shell` → `{can_send: false, can_receive: false, can_send_to_roles: []}`; `unknown` → `{can_send: false, can_receive: false, can_send_to_roles: []}`.
- **FR-022**: `effective_permissions` MUST be exposed verbatim in `list-agents --json` output and summarized in the human-readable `list-agents` form. FEAT-006 MUST NOT consume `effective_permissions` for any decision; it materializes the derived state for FEAT-009 and FEAT-010 to consume later.

#### Daemon socket API

- **FR-023**: The daemon socket API MUST expose five new methods reusing the FEAT-002 newline-delimited JSON request/response framing and host-user authorization (`0600`, host user only): `register_agent`, `list_agents`, `set_role`, `set_label`, `set_capability`. Each method MUST return a closed-set response envelope consistent with FEAT-002/FEAT-003/FEAT-004.
- **FR-024**: `register_agent` MUST accept the resolved container id and the resolved pane composite key as inputs alongside the user-supplied role/capability/label/project/parent fields. The CLI is responsible for resolving the container id (via FEAT-005) and the pane composite key (via `$TMUX` / `$TMUX_PANE` and a focused FEAT-004 lookup); the daemon MUST NOT re-derive container identity from the socket peer.
- **FR-025**: `list_agents` MUST be read-only, MUST NOT call Docker or tmux, MUST NOT acquire any registration mutex, and MUST return the latest committed SQLite state with a deterministic order: `active DESC, container_id ASC, parent_agent_id NULLS FIRST, label ASC, agent_id ASC`.
- **FR-026**: `list_agents` MUST accept these filters in the request envelope: `role` (single role or list of roles), `container_id` (full id or 12-char short prefix), `active_only` (boolean), `parent_agent_id` (full id, used to fetch swarm children of a given slave). Filters MUST compose with AND semantics. Unknown filter keys MUST be rejected with the closed-set error code `unknown_filter`. All filter values are matched case-sensitively against the lowercase canonical form: `role` values not in the FR-004 closed set (incl. mixed-case) MUST be rejected with `value_out_of_set`; mixed-case `container_id` / `parent_agent_id` filter values MUST be rejected with `value_out_of_set` rather than normalized.
- **FR-027**: `set_role`, `set_label`, and `set_capability` MUST be idempotent — calling each with the value the agent already has MUST succeed without error and MUST NOT append a duplicate JSONL audit record (no-op writes are not logged; only actual transitions are logged per FR-014).

#### CLI surface

- **FR-028**: `agenttower register-self` MUST accept the flags `--role <r>` (default `unknown`, applied only on first registration of a pane), `--capability <c>` (default `unknown`, applied only on first registration), `--label <l>` (default empty string, applied only on first registration), `--project <path>` (default empty, applied only on first registration), `--parent <agent-id>` (only valid with `--role swarm`), and `--json` (emit one JSON object on stdout; suppress human-readable output). For idempotent re-registration of an existing pane, defaults MUST NOT be transmitted: only flags the user explicitly passed on the command line MUST be sent in the `register_agent` request envelope, so per FR-007 unsupplied fields leave the stored row unchanged. On success the JSON MUST contain `agent_id`, `role`, `capability`, `label`, `project_path`, `parent_agent_id`, the resolved pane composite key, and `effective_permissions`.
- **FR-029**: `agenttower list-agents` MUST print a stable, scriptable human-readable form by default and a JSON form under `--json`. The default form MUST emit tab-separated values (TSV) with a required header row as the first output line, and a locked column set in this exact order: `AGENT_ID`, `LABEL`, `ROLE`, `CAPABILITY`, `CONTAINER`, `PANE`, `PROJECT`, `PARENT`, `ACTIVE`. `AGENT_ID` MUST render as the full `agt_<12-character-lowercase-hex>` form (16 chars total). `CONTAINER` MUST render as the bare 12-character short container id (no prefix). `PARENT` MUST render as the full `agt_<12-character-lowercase-hex>` form (16 chars total) when `parent_agent_id` is non-null and as the literal `-` (single ASCII hyphen) when null. Only the `CONTAINER` column uses the bare-12-char short form; every agent-id-bearing column (`AGENT_ID`, `PARENT`) uses the full `agt_<12-hex>` form so operator pipelines (grep, awk by full id) compose uniformly across columns. `PANE` MUST render as `<session>:<window>.<pane>` using the FEAT-004 short pane form. `ACTIVE` MUST render as the literal `true` or `false`. Any future fields added to the agent record MUST NOT be added to the default form; they MUST be exposed via `--json` or a separately-introduced `--wide` flag, so this column set is a stable contract for operator scripts and snapshot tests. `--json` MUST emit every field listed in FR-003 plus the immutable fields from FR-002. The command MUST accept `--role`, `--container`, `--active-only`, and `--parent` filters mirroring the daemon API in FR-026.
- **FR-030**: `agenttower set-role --target <agent-id> --role <r>` MUST accept `--confirm` and `--json`. Without `--confirm`, the CLI MUST refuse `--role master` per FR-011. The CLI MUST refuse `--role swarm` per FR-012. Successful invocations MUST print the prior role and new role (or emit them in JSON).
- **FR-031**: `agenttower set-label --target <agent-id> --label <l>` and `agenttower set-capability --target <agent-id> --capability <c>` MUST validate inputs against FR-005 (capability) and the per-field bounds (FR-033) and emit a stable success summary or an actionable error.
- **FR-032**: All five CLI commands in this feature MUST inherit the FEAT-005 socket-resolution priority order (`AGENTTOWER_SOCKET` → in-container default → host default) and the FEAT-005 / FEAT-002 daemon-unreachable exit-code-`2` behavior. None of these commands MUST start the daemon implicitly; the operator runs `agenttower ensure-daemon` separately.

#### Sanitization, transactionality, schema

- **FR-033**: Free-text fields MUST be sanitized of NUL bytes and terminal control bytes and bounded per-field: `label` ≤ 64 chars, `project_path` ≤ 4096 chars. Oversized values MUST be rejected with the closed-set error code `field_too_long` (registration MUST NOT silently truncate).
- **FR-034**: `project_path` MUST be a non-empty absolute path when supplied (must start with `/`, must not contain NUL, must not contain a `..` segment after normalization). The path MUST NOT be required to resolve to an existing host filesystem entry — it is observed from inside the container and may name a path that exists only in the container's mount namespace.
- **FR-035**: A `register-self`, `set-role`, `set-label`, or `set-capability` write MUST commit in a single SQLite transaction together with its derived `effective_permissions` recomputation. If the transaction fails, AgentTower MUST roll it back, MUST NOT append a JSONL audit record, MUST keep the daemon alive, and MUST return `internal_error` over the socket.
- **FR-036**: The SQLite migration from the FEAT-005 schema version to the FEAT-006 schema version MUST run in one transaction, MUST be idempotent on re-open, MUST create the new `agents` table even for an otherwise unchanged FEAT-005 database, and MUST bump `schema_version` only after the new table exists. Migration failure MUST roll back and refuse to serve the daemon.
- **FR-037**: FEAT-006 MUST NOT modify the FEAT-001/FEAT-002/FEAT-003/FEAT-004 schemas beyond additive read access. The `agents` table MUST reference the FEAT-004 pane composite key by value (denormalized columns) rather than by foreign-key constraint, because FEAT-004 panes can transition active/inactive across scans and reuse pane ids — referential integrity is enforced at the application layer, not at the SQLite schema layer.

#### Concurrency

- **FR-038**: The daemon MUST serialize concurrent `register_agent` requests addressing the *same* pane composite key through a per-(container_id, pane_composite_key) advisory in-process mutex so the second concurrent caller observes the first's committed state and converges on the same `agent_id`. Concurrent `register_agent` requests addressing *different* composite keys MUST proceed in parallel. The per-key mutex covers `register_agent` against other `register_agent` calls only; it MUST NOT be acquired by FEAT-004 pane reconciliation. Cross-subsystem ordering between a `register_agent` transaction and a FEAT-004 reconciliation transaction touching the same `agents` row is provided exclusively by SQLite's `BEGIN IMMEDIATE` semantics — both transactions remain atomic, neither observes a partial write from the other, and the transaction that commits last wins for any overlapping mutable column. `SQLITE_BUSY` MUST surface as `internal_error` (FR-035) without daemon-side retry.
- **FR-039**: `set_role`, `set_label`, `set_capability` requests MUST be serialized per-(agent_id) through the same advisory mutex pattern. SQLite's single-writer semantics serialize commits internally; the in-process mutex is required only to keep read-modify-write of `effective_permissions` consistent.

#### Failure modes

- **FR-040**: Each CLI in this feature MUST surface a stable, closed-set error code on every failure path. The closed set comprises at least: `daemon_unavailable`, `not_in_tmux`, `tmux_pane_malformed`, `host_context_unsupported`, `container_unresolved`, `pane_unknown_to_daemon`, `agent_not_found`, `agent_inactive`, `parent_not_found`, `parent_inactive`, `parent_role_invalid`, `parent_role_mismatch`, `parent_immutable`, `swarm_parent_required`, `swarm_role_via_set_role_rejected`, `master_via_register_self_rejected`, `master_confirm_required`, `value_out_of_set`, `field_too_long`, `project_path_invalid`, `unknown_filter`, `internal_error`, `schema_version_newer`. New codes MAY be added later but the existing set MUST remain stable. Error codes MUST appear verbatim in `--json` output.
- **FR-041**: When `register-self` is invoked from a pane whose composite key is not yet known to the FEAT-004 registry, AgentTower MUST trigger one focused FEAT-004 pane rescan scoped to the caller's resolved container before declaring the pane unknown. The rescan MUST reuse the FEAT-004 `scan_panes` codepath, MUST be bounded by the FEAT-004 5-second per-call timeout, MUST be limited to the caller's container, and MUST NOT cascade to other containers. If after the rescan the pane is still absent, `register-self` MUST refuse with `pane_unknown_to_daemon`.
- **FR-042**: The CLIs in this feature MUST NOT attempt to send any input into any tmux pane, MUST NOT call `docker exec` for any purpose other than the FR-041 focused rescan, MUST NOT modify any pane log, and MUST NOT install any tmux hook. FEAT-006 is registry-only; every other capability is deferred to a later feature.

#### Out-of-scope guarantees

- **FR-043**: FEAT-006 MUST NOT add a network listener, an in-container daemon, prompt delivery, queue inspection, route configuration, log attachment, log offset tracking, event ingestion, automatic swarm inference, multi-master arbitration, TUI, or web UI. The new socket methods inherit the FEAT-002 socket-file authorization (`0600`, host user only) and add no roles or secondary auth checks.
- **FR-044**: The feature MUST be testable end-to-end without a real Docker daemon, real bench container, or real tmux server, by injecting fake adapter implementations or canned `docker exec` outputs (consistent with FEAT-003/FEAT-004/FEAT-005 testability requirements).

### Key Entities *(include if feature involves data)*

- **Agent Record**: A registered AgentTower agent, persisted as one row in a new SQLite `agents` table. Identified by an opaque `agent_id` of the form `agt_<12-hex>`. Bound to one FEAT-004 pane composite key for life. Stores immutable fields (`agent_id`, `created_at`, pane composite key tuple, `parent_agent_id`) and mutable fields (`role`, `capability`, `label`, `project_path`, `effective_permissions`, `last_registered_at`, `last_seen_at`, `active`).
- **Pane Composite Key**: The FEAT-004 immutable identity tuple `(container_id, tmux_socket_path, tmux_session_name, tmux_window_index, tmux_pane_index, tmux_pane_id)`. FEAT-006 stores this tuple denormalized on the agent record and uses it as the registration key. Across pane disappear/reappear cycles, the tuple is treated as the same pane only when every component matches.
- **Effective Permissions**: A derived JSON value `{can_send: bool, can_receive: bool, can_send_to_roles: [<role>...]}` materialized on every agent row from the role per FR-021. Recomputed atomically on every role write.
- **Role**: A closed-set classification of an agent's responsibility (`master`, `slave`, `swarm`, `test-runner`, `shell`, `unknown`). Drives effective permission derivation. Master assignment requires explicit `--confirm` (per FR-011).
- **Capability**: A closed-set descriptor of what software the agent runs (`claude`, `codex`, `gemini`, `opencode`, `shell`, `test-runner`, `unknown`). Independent of role.
- **Parent Linkage**: An immutable `parent_agent_id` reference set only when `register-self --role swarm --parent <id>` succeeds. Validates at registration time that the parent exists, is active, and has `role=slave`. After creation FEAT-006 enforces no continuing constraint on the parent.
- **Audit Record**: A JSONL row appended to the existing FEAT-001 events file on every successful role transition. Contains `agent_id`, prior role, new role, `confirm_provided`, `socket_peer_uid`, and timestamp.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: With one seeded active bench container hosting one seeded active tmux pane, a single `agenttower register-self --role slave --capability codex --label codex-01 --project /workspace/acme` invocation completes within 500 ms wall-clock (p95) and persists exactly one new agent row whose `agent_id` matches the documented `agt_<12-hex>` shape and whose pane composite key matches the seeded pane.
- **SC-002**: Re-running the same `register-self` command from the same pane within the same scenario as SC-001 returns the same `agent_id`, leaves the agent count at exactly 1, and updates `last_registered_at` to a strictly later timestamp than the first invocation.
- **SC-003**: `agenttower register-self --role master --confirm` from a brand-new pane MUST exit non-zero with the closed-set error code `master_via_register_self_rejected`, MUST NOT create an agent row, and MUST NOT append a JSONL audit record. After the rejection, a follow-up `register-self --role slave` succeeds and a subsequent `set-role --role master --confirm` succeeds; the `agents` table contains exactly one row with `role=master`.
- **SC-004**: `agenttower set-role --role master` (without `--confirm`) against an existing slave agent MUST exit non-zero with the closed-set error code `master_confirm_required` and MUST leave the agent's role unchanged. `list-agents` MUST report `role=slave`.
- **SC-005**: With one seeded active slave agent and one seeded active different pane, `agenttower register-self --role swarm --parent <slave-id> --capability claude --label claude-swarm-01` MUST exit `0` and persist a new agent row whose `parent_agent_id` matches the slave's id and whose `effective_permissions.can_receive=true`. `list-agents --parent <slave-id>` MUST return exactly that one swarm row.
- **SC-006**: Each of the five swarm parent failure paths (`parent_not_found`, `parent_inactive`, `parent_role_invalid`, `parent_role_mismatch`, `swarm_parent_required`) MUST exit non-zero with the matching closed-set error code and MUST NOT create an agent row. The five paths together cover every branch of FR-015 / FR-016 / FR-017.
- **SC-007**: `agenttower list-agents --json` returns valid JSON whose array length equals the registered agent count and whose every object contains every field listed in FR-003, FR-002, and FR-021. Field values MUST be sanitized of NUL bytes and terminal control bytes and bounded per FR-033.
- **SC-008**: When the caller's resolved pane composite key is not present in the FEAT-004 `panes` table, `register-self` triggers exactly one focused FEAT-004 pane rescan scoped to the caller's container (no cascade to other containers, no global scan), and the rescan completes within the FEAT-004 5-second per-call budget. If the pane appears in the rescan's results, registration succeeds; otherwise it fails with `pane_unknown_to_daemon` and the daemon stays alive.
- **SC-009**: When the daemon is unreachable, every CLI in this feature exits with code `2` and the existing FEAT-002 daemon-unavailable message; the daemon is never started implicitly. When the caller is on the host shell and FEAT-005 reports `host_context`, `register-self` exits non-zero with `host_context_unsupported`.
- **SC-010**: Existing FEAT-001/FEAT-002/FEAT-003/FEAT-004/FEAT-005 test suites continue to pass after FEAT-006 is implemented, and the SQLite migration runs cleanly against a database created by a FEAT-005-only build. No FEAT-005 CLI invocation, JSON shape, exit code, or socket method behavior is altered.
- **SC-011**: The feature has unit coverage for: `agent_id` generation and uniqueness, role/capability closed-set validation, label/project_path sanitization and bounds, idempotent re-registration, re-activation of previously inactive agents, swarm parent validation (all five failure paths plus the success path), `effective_permissions` derivation across all six roles, master-promotion safety (no `--confirm`, via `register-self`, via `set-role`), JSONL audit record shape, and per-(pane composite key) registration mutex serialization.
- **SC-012**: The feature has integration coverage for: `register-self` from a simulated in-container environment, idempotent re-registration with mutating mutable fields, `list-agents` JSON output, `list-agents` filters (`--role`, `--container`, `--active-only`, `--parent`), `set-role` master promotion path, `set-role` swarm-rejection path, swarm child registration end-to-end, focused FEAT-004 rescan trigger when the pane is unknown, daemon-unreachable CLI path, and host-shell `host_context_unsupported` path — none of which require a real Docker daemon, real container, or real tmux server.

## Assumptions

- The host daemon is the source of truth for the agent registry. There is no in-container daemon, no in-container relay, no network listener, and no secondary auth tier in FEAT-006. The new socket methods inherit the FEAT-002 socket-file authorization (`0600`, host user only) verbatim.
- FEAT-006 builds on FEAT-005's container identity detection. The CLI resolves the caller's container id from `AGENTTOWER_CONTAINER_ID`, `/proc/self/cgroup`, `/etc/hostname`, or `$HOSTNAME`, then cross-checks against the FEAT-003 `containers` table per FEAT-005 FR-006 / FR-007. Only the `unique_match` outcome makes registration possible; every other outcome (`multi_match`, `no_match`, `no_candidate`, `host_context`) MUST refuse registration.
- The current tmux pane is resolved from `$TMUX` and `$TMUX_PANE` per FEAT-005 FR-014 / FR-015 and cross-checked against the FEAT-004 `panes` table. Registration requires the resolved pane to appear as `active=true` in the FEAT-004 registry, with one focused per-container FEAT-004 rescan permitted on a miss before declaring the pane unknown (FR-041).
- The agent_id format is `agt_<12-character-lowercase-hex>` (96 bits of entropy, sufficient to make accidental collisions vanishingly unlikely at MVP scale). The format is locked for FEAT-006 to keep CLI/JSON contracts stable for FEAT-009 / FEAT-010; a change would require a coordinated migration in a later feature.
- The `agents` table is denormalized against the FEAT-004 pane composite key (no foreign-key constraint) because FEAT-004 panes transition active/inactive across scans and reuse pane ids. Application-layer reconciliation is enforced by the FEAT-004 reconciliation rules and FEAT-006's per-(container, composite-key) mutex (FR-038).
- Master promotion requires an explicit `--confirm` flag passed at the CLI layer. FEAT-006 does not require any out-of-band confirmation channel (no two-factor, no separate operator channel, no host-vs-container distinction beyond the existing socket-file authorization). The `--confirm` flag is the explicit human-controlled gate; this matches the architecture's "no silent promotion" rule while keeping MVP friction low.
- Swarm hierarchy is exactly two levels deep in MVP (slave → swarm). Nested swarms (swarm-of-swarm) are explicitly rejected at registration time per FR-017 / FR-020. Deeper nesting can be enabled in a later feature without breaking the FEAT-006 schema (only the parent-role validation rule changes).
- `effective_permissions` is materialized on the agent row but not consumed by FEAT-006. FEAT-006 provides the derived state for FEAT-009 (input delivery) and FEAT-010 (routing) to read; downstream features MAY layer additional dynamic checks (in-flight prompt arbitration, per-target queue ordering) but the role-derived static permissions remain the FEAT-006 contract.
- Re-binding an existing agent to a different pane composite key (for example, when an agent's tmux pane is destroyed and re-created with a different pane id) is out of scope for FEAT-006. The new pane registers as a new agent; the operator can reconcile manually via labels and project paths. A first-class re-bind command can be added later without breaking the FEAT-006 schema.
- Automatic swarm inference (detecting that a newly discovered pane is likely a child of an existing slave based on container, session, command, or timing) is out of scope for FEAT-006 per the architecture's "explicit swarm reporting" stance. FEAT-010 owns inferred swarm parentage.
- The FEAT-004 focused per-container rescan triggered by FR-041 is a read-only scoped invocation of FEAT-004's existing scan codepath. FEAT-006 MUST NOT introduce any new pane-discovery codepath. If FEAT-004 ever adds a request-scoped scan-one-container API, FEAT-006 SHOULD adopt it; otherwise FEAT-006 invokes the existing scan path with a single-element container set.
- The JSONL audit record for role transitions is appended to the existing FEAT-001 `events.jsonl` file. FEAT-006 MUST NOT introduce a new audit log file.
- Project paths are validated for shape (absolute, no NUL, no `..` segment after normalization) but not for existence. The path is observed inside the container's mount namespace and may not exist on the host filesystem. Existence checks belong to a later feature that can perform `docker exec` validation if it ships.
- Concurrent `register-self` calls from different panes are common in the master-driven swarm pattern (the master prompts a slave, the slave prompts each child to register itself in parallel). FEAT-006's per-(composite-key) mutex (FR-038) is sufficient because each child has a distinct pane composite key; no global registration mutex is required.
- All in-container subprocess work invoked by FEAT-006 (limited to the FR-041 focused FEAT-004 rescan) inherits FEAT-004's threat model and bounded sanitization. FEAT-006 introduces no new in-container subprocess shapes.
