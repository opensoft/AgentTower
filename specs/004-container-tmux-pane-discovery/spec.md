# Feature Specification: Container tmux Pane Discovery

**Feature Branch**: `004-container-tmux-pane-discovery`
**Created**: 2026-05-06
**Status**: Draft
**Input**: User description: "FEAT-004 Container tmux Pane Discovery: host daemon scans tmux inside running bench containers via `docker exec -u <bench-user>`, discovers the default tmux server and additional sockets under `/tmp/tmux-1000/`, persists pane records (container, tmux socket, session, window, pane, pid, tty, command, cwd, title, active status, last_scanned_at), exposes `agenttower scan --panes` and `agenttower list-panes`, marks missing panes inactive, and degrades gracefully when tmux is unavailable or scan commands fail. Out of scope: pane registration, log capture, input routing."

## User Scenarios & Testing *(mandatory)*

### User Story 1 - Discover tmux Panes Inside Active Bench Containers (Priority: P1)

A developer runs a pane scan and sees every tmux pane from every active bench container surfaced through the host daemon, with enough identity and process detail to tell panes apart at a glance.

**Why this priority**: Pane discovery is the first capability that makes AgentTower useful beyond container enumeration. Every later feature — registration, log attachment, event ingestion, prompt routing — depends on a stable pane registry keyed off this scan.

**Independent Test**: Can be fully tested with a fake `docker exec` adapter that returns canned `tmux list-panes` output for one or more known active bench containers. Invoking `agenttower scan --panes` followed by `agenttower list-panes` must surface the expected pane rows with all required fields populated.

**Acceptance Scenarios**:

1. **Given** the daemon is running and FEAT-003 has persisted one active bench container that hosts a default tmux server with two sessions and three panes, **When** the user runs `agenttower scan --panes`, **Then** AgentTower persists exactly three active pane records linked to that container.
2. **Given** at least one pane was persisted by an earlier scan, **When** the user runs `agenttower list-panes` (combining the default human-readable form with the `--json` form for the canonical full payload), **Then** each pane row exposes container id, container name, container user, tmux socket path, session name, window index, pane index, tmux pane id, pane pid, pane tty, pane current command, pane current working directory, pane title, active state, first seen timestamp, and last scanned timestamp. The default form MAY emit a compact subset for readability; the full set MUST appear in `--json`.
3. **Given** a previously persisted pane is no longer present in a successful pane scan, **When** the scan completes, **Then** AgentTower marks that pane row inactive and preserves its history rather than deleting it.

---

### User Story 2 - Discover Multiple tmux Sockets Per Container (Priority: P2)

A developer who runs more than one tmux server inside a bench container (for example, the default socket plus a named `tmux -L work` socket) sees panes from every socket in `agenttower list-panes`, not only from the default server.

**Why this priority**: Bench workflows commonly run several tmux servers per container to isolate agent groups, run scratch shells, or experiment with parallel slaves. A registry that only sees the default socket would silently hide most of the work AgentTower is meant to coordinate.

**Independent Test**: Can be tested with a fake adapter that reports two socket files under `/tmp/tmux-<uid>/` (where `<uid>` is the fixture's bound `id -u` value, e.g., `1000` for a typical bench user; for example sockets `default` and `work`) and returns distinct pane sets per socket. After `agenttower scan --panes`, `agenttower list-panes` must show panes from both sockets, each row carrying its originating socket path.

**Acceptance Scenarios**:

1. **Given** an active bench container whose `/tmp/tmux-<uid>/` directory contains two socket files (`default` and `work`), **When** the user runs `agenttower scan --panes`, **Then** AgentTower scans both sockets and persists the union of their panes with the correct `tmux_socket_path` on each row.
2. **Given** a container has no usable socket files under `/tmp/tmux-<uid>/` and no default tmux server is reachable, **When** the user runs `agenttower scan --panes`, **Then** AgentTower records a degraded per-container scan result for that container, keeps any prior pane history, and continues scanning the remaining containers.
3. **Given** one socket inside a container scans successfully and a second socket fails (for example, returns an error or stale-socket exit code), **When** the scan completes, **Then** AgentTower persists panes from the successful socket as active, leaves prior panes from the failed socket unchanged, and records the failed socket in the degraded scan result.

---

### User Story 3 - Handle tmux and `docker exec` Degraded States (Priority: P3)

A developer continues to receive useful CLI output when tmux is missing inside a container, when `docker exec` fails, when a container becomes inactive between FEAT-003 and FEAT-004 scans, or when the daemon itself is unreachable. The daemon never crashes because of a single bad container or socket.

**Why this priority**: Bench containers vary widely. Some are minimal images without tmux, some are paused, some are stopped between scans, and `docker exec` itself can fail for transient reasons. AgentTower must remain usable when discovery cannot complete in part of the fleet.

**Independent Test**: Can be tested with a fake adapter that injects `tmux: not found`, `no server running`, `docker exec` non-zero exit, `docker exec` timeout, malformed `tmux list-panes` rows, and a container that disappears between FEAT-003 and FEAT-004 scans. The scan must complete with a degraded result, persist a scan row, surface the failures in CLI output, and keep the daemon healthy.

**Acceptance Scenarios**:

1. **Given** a bench container has no `tmux` binary on its `PATH`, **When** the user runs `agenttower scan --panes`, **Then** AgentTower records a `tmux_unavailable` per-container result for that container, leaves any prior pane rows for that container unchanged, and the overall scan completes with degraded status.
2. **Given** `docker exec` against a bench container exceeds the per-call timeout, **When** the scan runs, **Then** AgentTower records a `docker_exec_timeout` per-container result, terminates the subprocess before returning, and does not crash the daemon.
3. **Given** a container that was active in FEAT-003 has been marked inactive (stopped or removed) before the FEAT-004 scan runs, **When** the user runs `agenttower scan --panes`, **Then** AgentTower does not invoke `docker exec` against the inactive container, marks all of its previously active panes inactive, and updates their `last_scanned_at`.
4. **Given** the daemon is not running, **When** the user runs `agenttower scan --panes` or `agenttower list-panes`, **Then** the CLI exits with a non-success status and prints an actionable daemon-unreachable message.

### Edge Cases

- A container has zero active tmux sessions (the default server is running but `list-panes -a` returns no rows).
- A container has one tmux server bound to a non-default socket name and no `default` socket file.
- The bench user has no `/tmp/tmux-<uid>/` directory at all (where `<uid>` is the per-scan `id -u` result; see FR-020 / R-006 — `1000` is illustrative, not hardcoded).
- `/tmp/tmux-<uid>/` contains files that look like sockets but are stale, broken, or owned by another user.
- A pane's title, command, or cwd contains shell metacharacters, ANSI escape bytes, embedded NULs, tabs, or newlines.
- A pane's title or cwd is unusually long (well past typical terminal widths).
- A pane id (`%N`) is reused after a tmux server restart, but the new pane is in a different session/window.
- Two pane scans run concurrently against the same daemon (the second pane scan MUST serialize behind the first via the pane-scan mutex; FR-017).
- A pane scan and a container scan run concurrently against the same daemon (the two scans MAY proceed in parallel because the pane-scan mutex is independent of the FEAT-003 container-scan mutex; SQLite's single-writer semantics serialize their commits internally; FR-017).
- The daemon restarts between scans; previously active panes retain history and are reconciled on the next successful scan.
- A container is renamed between scans; pane reconciliation continues to key on container id rather than name.
- The tmux format string is partially honored by an old tmux server (one or more fields missing); rows with the wrong number of fields must be treated as malformed and counted in the degraded result rather than persisted.
- A FEAT-003 container scan is in flight when a FEAT-004 pane scan starts (or vice versa); the two scans must not corrupt each other's transactions.
- A previously persisted pane is no longer present because tmux itself is unavailable inside its container — this MUST NOT mark the pane inactive, because the container's pane state is unknown rather than known-empty.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: The system MUST provide a tmux discovery adapter that executes `docker exec` plus `tmux` subcommands against running bench containers through a testable abstraction with no real Docker or tmux dependency in unit tests.
- **FR-002**: The adapter MUST scan only containers that FEAT-003 has persisted as `active` at the time the pane scan starts. The pane scan MUST NOT re-run FEAT-003's container scan implicitly.
- **FR-003**: For each active container, the adapter MUST attempt to discover the bench user's default tmux server plus every additional tmux socket file under `/tmp/tmux-<uid>/` belonging to that user, and scan each discovered socket independently.
- **FR-004**: Socket discovery MUST list `/tmp/tmux-<uid>/` contents through `docker exec`, treat the literal `default` socket name as the implicit default when present, and treat each additional regular file in that directory as a candidate socket. Subdirectories, symlinks pointing outside `/tmp/tmux-<uid>/`, and non-socket files MUST be skipped without failing the scan.
- **FR-005**: For each candidate socket, the adapter MUST run `tmux -S <socket-path> list-panes -a -F <format>` (with the format described in `docs/architecture.md` §7) inside the container and parse one row per pane.
- **FR-006**: Every persisted pane record MUST include: `container_id`, `container_name`, `container_user`, `tmux_socket_path`, `tmux_session_name`, `tmux_window_index`, `tmux_pane_index`, `tmux_pane_id`, `pane_pid`, `pane_tty`, `pane_current_command`, `pane_current_path` (cwd), `pane_title`, `active` flag, `first_seen_at`, and `last_scanned_at`.
- **FR-007**: The system MUST persist pane records in a new SQLite `panes` table whose primary identity for active/inactive reconciliation is the composite key `(container_id, tmux_socket_path, tmux_session_name, tmux_window_index, tmux_pane_index, tmux_pane_id)`. A pane is treated as the same pane across scans only when every component of that composite key matches.
- **FR-008**: When a previously active pane is no longer reported by a successful per-socket scan whose container is still active, AgentTower MUST mark that pane inactive and update its `last_scanned_at`. Pane rows MUST NOT be deleted during reconciliation.
- **FR-009**: When a container that previously had panes is marked inactive in the `containers` table at scan start, AgentTower MUST mark all of that container's previously active panes inactive in the same FEAT-004 scan transaction without invoking `docker exec` against it. Their `last_scanned_at` MUST be updated to the scan's start timestamp.
- **FR-010**: When `tmux` is unavailable inside a container (no binary, no server running, every socket scan fails), AgentTower MUST NOT mark that container's prior panes inactive. The pane state for that container is unknown for this scan; the prior `active` flag MUST be preserved and only `last_scanned_at` MUST be updated.
- **FR-011**: When at least one socket inside a container scans successfully, the panes from that successful socket are reconciled normally; prior panes belonging to a *different* socket on the same container that failed to scan MUST be left unchanged (active flag preserved, `last_scanned_at` updated). Reconciliation is per-(container, socket) tuple, not per-container.
- **FR-012**: The system MUST persist each FEAT-004 pane scan as one row in a new SQLite `pane_scans` table capturing scan id, started_at, completed_at, scan status (`ok` or `degraded`), counts (containers scanned, sockets scanned, panes seen, panes newly active, panes reconciled to inactive, containers skipped because inactive, containers with `tmux_unavailable`), and per-container/per-socket error details for the degraded path. Healthy scans MUST also persist a row with zero error details.
- **FR-013**: The daemon socket API MUST expose a `scan_panes` method (one-shot scan and persist) and a `list_panes` method (read persisted records). Both methods MUST return a closed-set response envelope consistent with the FEAT-002/FEAT-003 protocol.
- **FR-014**: `agenttower scan --panes` MUST call `scan_panes` over the existing Unix socket and print a concise scan summary including scan id, status, counts, and a bounded list of per-container/per-socket failures. `agenttower scan --panes --json` MUST emit one JSON object per invocation.
- **FR-015**: `agenttower list-panes` MUST call `list_panes` and print persisted pane records in a stable, scriptable format. By default the command MUST return all panes (active and inactive) with active rows ordered before inactive rows. The command MUST accept an `--active-only` flag, an optional `--container <id-or-name>` filter, and a `--json` flag that emits one JSON object per invocation.
- **FR-016**: `list_panes` MUST be read-only, MUST NOT call Docker or tmux, MUST NOT acquire the FEAT-004 scan mutex, and MUST return the latest committed SQLite state with a deterministic order: `active DESC, container_id ASC, tmux_socket_path ASC, tmux_session_name ASC, tmux_window_index ASC, tmux_pane_index ASC`.
- **FR-017**: The daemon MUST serialize concurrent `scan_panes` requests using a dedicated in-process mutex so that at most one pane scan runs at a time. Subsequent callers MUST block until the in-flight scan completes and then run, each receiving its own complete scan result. The pane-scan mutex MUST be independent of the FEAT-003 container-scan mutex; the two scans MAY proceed concurrently with respect to each other.
- **FR-018**: Every `docker exec` subprocess invocation issued by FEAT-004 (the `id -u` call, the socket-listing call, and each per-socket `tmux list-panes` call) MUST be bounded by a 5-second per-call timeout. A timeout MUST be normalized into a `docker_exec_timeout` per-container or per-socket error, the subprocess MUST be terminated and waited on before returning, and the daemon MUST keep running. If the termination itself fails or exceeds a secondary 1-second grace period (the kernel cannot reap the child, `kill()` raises `OSError`, or `wait()` re-times-out), AgentTower MUST stop attempting to recover that child, MUST escalate the per-`(container, socket?)` error to `internal_error` (still a degraded outcome — the daemon stays alive), MUST continue scanning the remaining containers, and MUST record the escalation in the `pane_scans.error_details_json` entry. The daemon MUST NOT block subsequent scans on an unrecovered child.
- **FR-019**: `docker exec` failures (command-not-found inside the container, non-zero exit, broken pipe, or unparseable output) MUST produce per-container or per-socket degraded results without crashing the daemon. Specific error codes used in the degraded result MUST include at least: `tmux_unavailable`, `tmux_no_server`, `socket_dir_missing`, `socket_unreadable`, `docker_exec_failed`, `docker_exec_timeout`, `output_malformed`, and `bench_user_unresolved`.
- **FR-020**: The bench user used by `docker exec -u <bench-user>` for a given container MUST be derived per scan from that container's persisted FEAT-003 inspect metadata when available (the `config user` field surfaced by FEAT-003), falling back to the host daemon's `$USER` env var when no container-side user is recorded, and falling back further to `pwd.getpwuid(os.getuid()).pw_name` when `$USER` is also unset or empty. If all three resolution paths return empty, AgentTower MUST emit a per-container `bench_user_unresolved` error (a closed-set error code added to FR-019), MUST skip the rest of that container's scan without invoking any `docker exec`, and MUST treat the container as `tmux_unavailable` for FR-010 preservation purposes. The numeric uid used to construct `/tmp/tmux-<uid>/` MUST be obtained inside the container at scan time via a bounded `id -u` call rather than assumed; the host-side bench user resolution path above is independent of the container-side uid resolution path. If the resolved bench user contains a `:uid` form (e.g., `user:1001`), AgentTower MUST split on the first `:` and use the left-hand component as the `-u <bench-user>` argv value (the in-container `id -u` then derives the authoritative numeric uid).
- **FR-021**: All `docker exec` invocations MUST use typed argv with `shell=False` (or the direct equivalent) and MUST NOT interpolate container ids, names, socket paths, the bench user, or any tmux output into shell strings. Shell metacharacters in container metadata, socket file names, pane titles, or pane cwds MUST be treated as ordinary data.
- **FR-022**: The daemon MUST resolve the Docker binary the same way FEAT-003 does (`shutil.which("docker")` against the daemon process environment `PATH`). FEAT-004 MUST NOT add a new PATH-pinning or PATH-scrubbing strategy. A missing or non-executable resolved binary MUST produce an `ok:false` socket envelope with `docker_unavailable`.
- **FR-023**: Pane field values stored in SQLite, JSONL events, socket responses, and CLI JSON output MUST be sanitized of NUL bytes and terminal control bytes and bounded to a per-field maximum (default: 2048 characters per field, with `pane_title` and `pane_current_command` allowed up to 2048 and `pane_current_path` allowed up to 4096). Oversized values MUST be truncated rather than rejecting the pane row, and the truncation MUST be recorded as a per-pane note in the scan result.
- **FR-024**: The `pane_scans` insert and all `panes` upsert/touch/inactivate writes for one scan MUST commit in one SQLite transaction. If that transaction fails, AgentTower MUST roll it back, MUST release the pane-scan mutex, MUST NOT append a JSONL event for that failed transaction, and MUST return `internal_error` while keeping the daemon alive.
- **FR-025**: Side effects of one pane scan MUST occur in this order: emit `pane_scan_started` after acquiring the mutex; load active containers and per-container bench user from SQLite; iterate containers, performing per-container `docker exec` calls; build the in-memory reconciliation set; commit the SQLite scan transaction; append a degraded JSONL event only when the committed scan is degraded; emit `pane_scan_completed`; return the socket response. Healthy scans MUST NOT append to `events.jsonl`. **Post-commit side-effect failure** (the SQLite transaction commits but the subsequent JSONL append, the `pane_scan_completed` lifecycle log emit, or any other post-commit step fails): the committed `pane_scans` row MUST NOT be rolled back (SQLite is the durable source of truth), the daemon MUST stay alive, the pane-scan mutex MUST be released, and the socket response MUST be `internal_error`. **Pre-commit lifecycle log failure** (the `pane_scan_started` emit fails after mutex acquisition but before any subprocess work): the daemon MUST release the mutex, MUST NOT call any `docker exec` for that scan, MUST NOT write a `pane_scans` row, and MUST return `internal_error`.
- **FR-026**: Degraded JSONL events and lifecycle log rows MUST contain only scan id, status, aggregate counts, the closed-set error code, a bounded/sanitized error message, and the affected `container_id`/`tmux_socket_path` where applicable. Raw `docker exec` stderr beyond the bounded message, raw tmux output, raw environment values, and raw pane field values MUST NOT be written to events or lifecycle logs.
- **FR-027**: A scan status is `ok` only when no per-container or per-socket errors occurred, every active container produced at least one parsed `tmux list-panes` row or an empty result from a reachable tmux server, and every persisted pane row passed sanitization without truncation. A "reachable tmux server" is defined as a `tmux -S <socket-path> list-panes -a -F <format>` invocation that exited `0` with empty stdout; a "no server running" exit pattern (typically exit `1` with `error connecting to <socket-path>` or `no server running on <socket-path>` on stderr) is the distinct `tmux_no_server` per-socket error and is NOT a reachable-empty result. A scan status is `degraded` when any per-container or per-socket error code applies, when at least one row required truncation, or when at least one `tmux list-panes` row was malformed.
- **FR-028**: Two scan requests against unchanged container/tmux state MUST produce distinct scan ids and distinct `pane_scans` rows while converging `panes` rows to the same final content. A degraded scan MUST append at most one JSONL event for its scan id.
- **FR-029**: A SQLite migration from the FEAT-003 schema version to a new schema version MUST run in one transaction, be idempotent on re-open, create the new `panes` and `pane_scans` tables even for an otherwise unchanged FEAT-003 database, and bump `schema_version` only after the new tables exist. If the migration fails, AgentTower MUST roll back and refuse to serve the daemon rather than serving partial schema. Schema versions newer than the build supports MUST cause daemon startup refusal.
- **FR-030**: Existing FEAT-001, FEAT-002, and FEAT-003 CLI behavior, daemon socket methods, and persisted state MUST remain backward compatible. FEAT-004 MUST NOT alter the FEAT-003 `containers` or `container_scans` schemas beyond additive read access.
- **FR-031**: FEAT-004 MUST NOT add a network listener, an in-container daemon, agent registration, role/capability metadata, log capture, log offset tracking, prompt queuing, or input delivery. The new socket methods inherit the FEAT-002 socket-file authorization (`0600`, host user only) and add no roles or secondary auth checks.
- **FR-032**: FEAT-004 MUST run all `docker exec` invocations as the existing daemon user (no `sudo`, no uid/gid change on the host), and MUST pass `-u <bench-user>` to scope the in-container execution to the bench user. FEAT-004 MUST NOT spawn shells inside the container beyond the enumerated tmux/`id`/`ls` invocations needed for socket and pane discovery.
- **FR-033**: The set of in-container subprocesses FEAT-004 may invoke is closed: (a) one `id -u` call per scanned container to resolve the bench uid, (b) one socket-listing call per container to enumerate `/tmp/tmux-<uid>/`, and (c) one `tmux -S <socket-path> list-panes -a -F <format>` call per discovered socket. Any other `docker exec` payload is out of scope for FEAT-004.
- **FR-034**: The feature MUST be testable end-to-end without a real Docker daemon, real bench container, or real tmux server, by injecting fake adapter implementations or canned `docker exec` outputs.

### Key Entities *(include if feature involves data)*

- **Pane Record**: A discovered tmux pane inside a bench container, persisted as one row in a new SQLite `panes` table. Identified across scans by the composite key `(container_id, tmux_socket_path, tmux_session_name, tmux_window_index, tmux_pane_index, tmux_pane_id)`. Stores container id/name/user, socket path, session, window/pane indices, tmux pane id, pane pid, tty, current command, cwd, title, active flag, first seen timestamp, and last scanned timestamp.
- **Pane Scan Result**: The outcome of one `scan_panes` request, persisted as one row in a new SQLite `pane_scans` table. Includes scan id, started_at, completed_at, status (`ok` or `degraded`), aggregate counts (containers scanned, sockets scanned, panes seen, panes newly active, panes reconciled to inactive, containers skipped because inactive, containers with `tmux_unavailable`), and per-container/per-socket error details for the degraded path.
- **Tmux Socket Reference**: A `(container_id, tmux_socket_path)` tuple representing one tmux server reachable inside a bench container. Reconciliation of pane active/inactive state is scoped per socket, not per container.
- **Tmux Discovery Error**: A normalized failure result for `tmux_unavailable`, `tmux_no_server`, `socket_dir_missing`, `socket_unreadable`, `docker_exec_failed`, `docker_exec_timeout`, or `output_malformed`. Each error is attached to either the affected container or the affected `(container, socket)` tuple.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: With a fake adapter reporting one active bench container that hosts two tmux sockets with three and two panes respectively, `agenttower scan --panes` persists exactly five active pane records distributed across the two sockets and returns a single `ok` scan result with matching counters.
- **SC-002**: After a successful scan where one previously active pane is removed from `tmux list-panes` output and another previously active pane remains, `agenttower list-panes` shows the removed pane as inactive and the surviving pane as active within the same scan invocation, with no pane rows deleted.
- **SC-003**: When FEAT-003 marks a previously active container inactive between scans, the next `agenttower scan --panes` invocation marks every prior pane belonging to that container inactive, does not invoke `docker exec` against it, and reports the container under the "containers skipped because inactive" counter.
- **SC-004**: When the bench container has no `tmux` binary (or no server running), `agenttower scan --panes` returns a `degraded` scan result with a `tmux_unavailable` per-container error, prior pane rows for that container retain their previous `active` flag, and `agenttower status` continues to succeed.
- **SC-005**: `agenttower list-panes --json` returns valid JSON containing every required pane field for each persisted record, with field values sanitized of NUL bytes and terminal control bytes and bounded to the per-field maximums.
- **SC-006**: A `docker exec` timeout against one container produces a `docker_exec_timeout` per-container error within the 5-second per-call budget, the underlying subprocess is terminated and waited on (no orphaned children), and the remaining containers in the same scan continue to be processed.
- **SC-007**: Existing FEAT-001, FEAT-002, and FEAT-003 test suites continue to pass after FEAT-004 is implemented, and the SQLite migration runs cleanly against a database created by a FEAT-003-only build.
- **SC-008**: The feature has unit coverage for socket enumeration, `tmux list-panes` row parsing, per-pane sanitization/truncation, per-(container, socket) reconciliation, and socket method response shapes.
- **SC-009**: The feature has integration coverage for `scan --panes`, `list-panes`, `--json` output, multi-socket scans, no-pane scans, inactive-container cascade, tmux-unavailable degraded path, `docker exec` timeout, and the daemon-unreachable CLI path — none of which require a real Docker daemon, real container, or real tmux server.

## Assumptions

- The host daemon performs pane discovery from the host environment by shelling out to `docker exec`. No in-container daemon, no in-container relay, and no Docker SDK runtime dependency is added in FEAT-004.
- FEAT-004 depends on FEAT-003 having already populated the `containers` table. `agenttower scan --panes` reads the active container set from SQLite at scan start; users who want fresh container data first run `agenttower scan --containers` themselves.
- The bench user inside each container is derived from FEAT-003's persisted inspect metadata when present (the container's configured `User`), and falls back to the host daemon's `$USER` when no container-side user is recorded. Per-container override via config is out of scope for FEAT-004 and can be added later if needed.
- The bench uid (used to construct `/tmp/tmux-<uid>/`) is resolved per scan by a bounded `id -u` call inside the container rather than assumed to be `1000`. Treating `1000` as a hardcoded uid is rejected because production bench users vary across machines.
- Pane reconciliation across scans uses the composite key `(container_id, tmux_socket_path, tmux_session_name, tmux_window_index, tmux_pane_index, tmux_pane_id)`. A pane is treated as the same pane only when every component matches; reused tmux pane ids in a different session/window are treated as new panes. Stable cross-restart pane identity is deferred to a future feature.
- Multiple tmux sockets per container are scanned independently and reconciled per-(container, socket). A failed socket inside an otherwise reachable container does not invalidate the panes already discovered on its sibling sockets.
- Pane scans are request-driven via the CLI in FEAT-004; recurring background pane scans, change watchers, and incremental rescans can be introduced later. Two consecutive request-driven scans against unchanged state are expected to converge `panes` rows to identical content while still producing distinct `pane_scans` rows.
- SQLite remains the durable source of truth for persisted pane records and scan history. The existing FEAT-001 JSONL events file is reused for degraded pane-scan events; a new events file is not introduced.
- The Unix socket API is extended with two additional closed-set methods (`scan_panes`, `list_panes`) reusing the FEAT-002 request/response framing and host-user authorization. No new transport, no new authentication tier, and no new permission roles are introduced.
- The pane-scan mutex is in-process only and is recreated on daemon restart. In-flight pane scans are aborted if the daemon process exits; partial work that has not yet committed to SQLite is discarded.
- FEAT-004 inherits the FEAT-003 threat model: the host user, daemon process environment, resolved Docker binary, and Docker daemon are trusted; container metadata, tmux output, pane titles, pane cwds, and `docker exec` stderr are untrusted data subject to bounded sanitization but not semantic secret redaction. Secret redaction of pane titles or cwds is deferred to FEAT-007.
- `agenttower scan --panes` and `agenttower list-panes` are usable from the host shell only in FEAT-004. Container-side CLI use of these commands is the goal of FEAT-005 and is not validated here.
- Pane field truncation thresholds (2048 chars for most fields, 4096 chars for cwd) are practical defaults for terminal-derived strings and may be revisited in a later feature without breaking persisted rows; older rows can be re-truncated on subsequent scans.
