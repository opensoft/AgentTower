# Quickstart: Package, Config, and State Foundation

**Branch**: `001-package-state-foundation` | **Date**: 2026-05-05

A five-minute, top-to-bottom walkthrough of the FEAT-001 surface area
once it ships. Use this as the manual smoke test that mirrors what the
automated `tests/integration/test_cli_version.py`,
`tests/integration/test_cli_paths.py`, and
`tests/integration/test_cli_init.py` cover.

Assumes Python 3.11+ and a checkout of the repo at
`/workspace/projects/AgentTower`.

---

## 1. Install in development mode

From the repo root:

```bash
python -m venv .venv
source .venv/bin/activate
pip install -e '.[test]'
```

This places the two console scripts on `$PATH`:

```bash
which agenttower agenttowerd
```

Both should resolve to scripts under `.venv/bin/`.

---

## 2. Confirm the package and version

```bash
agenttower --version
# → agenttower 0.1.0

agenttowerd --version
# → agenttowerd 0.1.0
```

Both commands MUST exit `0` and report the same version. In a fully
installed FEAT-001 package this is `0.1.0`; direct source-tree execution
without package metadata may report the fallback `0.0.0+local`. Neither
creates any files on disk yet.

---

## 3. Inspect the resolved paths (before init)

```bash
agenttower config paths
```

Expected stdout (six lines, fixed order, `KEY=value` shape):

```text
CONFIG_FILE=/home/<you>/.config/opensoft/agenttower/config.toml
STATE_DB=/home/<you>/.local/state/opensoft/agenttower/agenttower.sqlite3
EVENTS_FILE=/home/<you>/.local/state/opensoft/agenttower/events.jsonl
LOGS_DIR=/home/<you>/.local/state/opensoft/agenttower/logs
SOCKET=/home/<you>/.local/state/opensoft/agenttower/agenttowerd.sock
CACHE_DIR=/home/<you>/.cache/opensoft/agenttower
```

Expected stderr (because the database does not yet exist):

```text
note: agenttower has not been initialized; run `agenttower config init`
```

You can load every path into your shell environment in one step:

```bash
eval "$(agenttower config paths)"
echo "$STATE_DB"
```

---

## 4. Initialize the host layout

```bash
agenttower config init
```

Expected stdout (first run on a fresh host):

```text
created config: /home/<you>/.config/opensoft/agenttower/config.toml
created registry: /home/<you>/.local/state/opensoft/agenttower/agenttower.sqlite3
```

Expected stderr: empty. Exit code: `0`.

---

## 5. Verify the layout on disk

```bash
ls -la "$(dirname "$CONFIG_FILE")"
# Expected: directory mode drwx------ (0700), config.toml mode -rw------- (0600).

ls -la "$(dirname "$STATE_DB")"
# Expected: directory mode drwx------ (0700), agenttower.sqlite3 mode -rw------- (0600).
# Note: events.jsonl and agenttowerd.sock are NOT created yet — that is correct.

stat -c '%a %n' \
  "$CONFIG_FILE" "$STATE_DB" \
  "$LOGS_DIR" "$CACHE_DIR" \
  "$(dirname "$CONFIG_FILE")" "$(dirname "$STATE_DB")"
```

Expected output (modes only):

```text
600 …/config.toml
600 …/agenttower.sqlite3
700 …/logs
700 …/agenttower
700 …/agenttower
700 …/agenttower
```

---

## 6. Inspect the default configuration

```bash
cat "$CONFIG_FILE"
```

Expected content (verbatim):

```toml
# AgentTower configuration
# Generated by `agenttower config init`. Edit freely; subsequent
# `config init` runs will not overwrite this file.

[containers]
name_contains = ["bench"]
scan_interval_seconds = 5
```

---

## 7. Inspect the registry's schema-version row

```bash
sqlite3 "$STATE_DB" 'SELECT version FROM schema_version;'
# → 1

sqlite3 "$STATE_DB" 'SELECT COUNT(*) FROM schema_version;'
# → 1
```

The `schema_version` table contains exactly one row with `version=1`.

---

## 8. Confirm idempotence

```bash
for i in 1 2 3 4 5; do
  agenttower config init
done
```

Each invocation MUST exit `0`. After the first, all subsequent runs
print:

```text
already initialized: …/config.toml
already initialized: …/agenttower.sqlite3
```

`config.toml` byte content, `agenttower.sqlite3` row contents, and the
directory layout MUST be unchanged. Verify:

```bash
sha256sum "$CONFIG_FILE"        # same hash as after step 4
sqlite3 "$STATE_DB" 'SELECT version, COUNT(*) FROM schema_version;'
# → 1|1
```

---

## 9. Confirm `config paths` after init

```bash
agenttower config paths
```

Same six stdout lines as step 3. Stderr is now **empty** (the
informational note disappears once `STATE_DB` exists).

## 10. (Negative test) Run against an unwritable target

Simulate a read-only base directory:

```bash
mkdir -p /tmp/agenttower-ro/.local/state
chmod -w /tmp/agenttower-ro/.local/state
HOME=/tmp/agenttower-ro \
XDG_CONFIG_HOME= XDG_STATE_HOME= XDG_CACHE_HOME= \
  agenttower config init
echo "exit=$?"
```

Expected:

- Exit code `1`.
- Stderr like:
  ```text
  error: create directory: /tmp/agenttower-ro/.local/state/opensoft: Permission denied
  ```
- No partial `agenttower.sqlite3` file under `/tmp/agenttower-ro/`.

Cleanup:

```bash
chmod +w /tmp/agenttower-ro/.local/state
rm -rf /tmp/agenttower-ro
```

---

## 11. (Negative test) FEAT-001 emits no events

The event history file MUST NOT exist after any FEAT-001 command runs:

```bash
test ! -e "$EVENTS_FILE" && echo "OK: no events file created"
```

Per FR-016 / clarification Q4, no FEAT-001 CLI command writes to
`events.jsonl`. The event-writer ships ready for FEAT-002+ callers but
is exercised only by tests.

---

## 12. (Optional) Exercise the event-writer from a Python REPL

```python
from pathlib import Path
import os
from agenttower.paths import resolve_paths
from agenttower.events.writer import append_event

paths = resolve_paths()
append_event(paths.events_file, {"event_type": "manual_smoke_test", "n": 1})
print(Path(paths.events_file).read_text())
```

Expected: one line, valid JSON, containing both the writer-supplied
`ts` and the caller-supplied keys. The file mode MUST be `0o600`:

```bash
stat -c '%a' "$EVENTS_FILE"
# → 600
```

Delete it once you're done so subsequent runs of step 11 still pass:

```bash
rm "$EVENTS_FILE"
```

## What FEAT-001 deliberately does NOT do

Anything you might expect from a fuller agent-coordination tool is
**out of scope** for this feature. The following will all fail, hang,
or be unimplemented:

- starting a daemon (`agenttowerd` runs `--version` and exits)
- listening on the Unix socket (the path is reserved but not bound)
- discovering Docker containers or tmux panes
- registering an agent
- attaching pane logs
- emitting classified events
- routing prompts
- delivering input

These belong to FEAT-002 through FEAT-010. FEAT-001 only ships the
package, the path layout, the registry schema-version row, and the
event-writer utility.
